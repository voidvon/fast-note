## 2. 模块目标与现状

- 业务目标：在本地优先前提下实现多端最终一致，并保留可回滚的失败策略。
- 当前状态（已支持/部分支持/未支持）：已支持
- 关键证据：
  - `E1` [src/hooks/useSync.ts:244]：`sync(silent)` 未登录静默跳过/显式报错分支。
  - `E2` [src/hooks/useSync.ts:277]：双向同步主流程（本地增量 + 云端增量）。
  - `E3` [src/hooks/useSync.ts:307]：删除超过 30 天触发 `deleteLocal/delete`。
  - `E4` [src/hooks/useSync.ts:527]：每条成功记录推进 `localStorage.pocketbaseUpdated`。
  - `E5` [src/pocketbase/notes.ts:16]：按 `user_id` + `updated` 过滤拉取。
  - `E6` [src/adapters/pocketbase/realtime-adapter.ts:51]：Realtime 订阅按 `user_id` 过滤。
  - `E7` [src/components/UserProfile.vue:126]：手动同步入口和结果展示。
- 已知缺口：
  - `推断`：同步中途一条失败即整体中断，批处理容错能力偏弱。

## 3. 逐功能点方案

### 3.1 F3.1 双向增量同步与游标推进

#### 3.1.1 目标与边界

- 用户场景：登录用户在多设备编辑后，可通过自动或手动同步获得一致内容。
- 范围内：增量拉取、操作队列构建、游标推进。
- 范围外：附件映射细节（归属“附件与安全”模块）。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` [src/hooks/useSync.ts:281]：读取本地 `getNotesByUpdated(updated)`。
  - `E2` [src/hooks/useSync.ts:285]：读取云端 `notesService.getNotesByUpdated(updated)`。
  - `E3` [src/hooks/useSync.ts:416]：按 `updated` 排序后串行执行。
  - `E4` [src/hooks/useSync.ts:528]：单条完成即推进游标并写 `localStorage`。
- 现状结论：当前方案可满足“时间戳裁决 + 顺序执行”的基础一致性。
- 推断与待确认：
  - `待确认`：是否允许在失败后保留已成功结果并继续后续操作（部分成功语义）。

#### 3.1.3 技术方案

- 方案摘要：保持顺序执行和时间戳裁决，新增失败分级与重试队列。
- 架构落点（按层）：
  - UI（`views/components`）：同步结果细分“成功/失败条数”。
  - 编排（`hooks`）：`operations` 增加 `retryable` 标签与最大重试次数。
  - 领域（`stores`）：维持 `add/update/delete` 现有接口。
  - 持久化（`database`）：游标仍存 `localStorage.pocketbaseUpdated`。
  - 云端（`pocketbase`）：保持 `updated + user_id` 拉取条件。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：无新增业务字段，可新增内部同步结果类型。
  - 兼容与迁移：无。
- 同步与冲突策略：`updated` 新者覆盖旧者，删除状态保留 30 天窗口。
- 错误处理与回滚策略：失败操作进入重试队列；不回滚已成功条目。

#### 3.1.4 实施任务拆分

| 任务ID | 任务描述                         | 变更文件                         | 依赖         | 风险 |
| ------ | -------------------------------- | -------------------------------- | ------------ | ---- |
| T1     | 同步队列增加失败重试和结果聚合   | `src/hooks/useSync.ts`           | 无           | 中   |
| T2     | 用户面板展示失败条目数与重试按钮 | `src/components/UserProfile.vue` | T1           | 低   |
| T3     | 增量游标边界回归测试             | `tests/**`                       | 模拟云端数据 | 中   |

#### 3.1.5 验收标准

- AC1：同步完成后返回上传/下载/删除/失败统计。
- AC2：游标只推进到已成功处理的最大 `updated`。
- AC3：重试后可消化瞬时网络错误。

#### 3.1.6 测试与验证

- 单元测试：同步操作排序、游标推进逻辑。
- 集成/冒烟：A 端更新 + B 端更新 + 双向同步后数据一致。
- 回归关注点：silent 模式在未登录时必须返回 `null`。

#### 3.1.7 风险与未决问题

- 风险：重试机制可能导致重复更新请求，需要依赖 upsert 幂等。
- 未决问题：是否对不同错误类型（权限/网络/数据）使用不同重试策略。

### 3.2 F3.2 冲突裁决与删除保留策略

#### 3.2.1 目标与边界

- 用户场景：同一笔记在多端被修改或删除时，系统做可预测裁决。
- 范围内：时间戳冲突处理、30 天删除保留、本地/云端清理动作。
- 范围外：协作编辑冲突合并（OT/CRDT）。

#### 3.2.2 现状证据

- 代码事实：
  - `E1` [src/hooks/useSync.ts:353]：比较 `localTime` / `cloudTime` 决定 upload/download。
  - `E2` [src/hooks/useSync.ts:314]：本地已删除且超 30 天执行本地清理。
  - `E3` [src/hooks/useSync.ts:373]：云端已删除且超 30 天执行本地删除。
  - `E4` [src/stores/notes.ts:264]：回收站同样按 30 天过滤展示。
- 现状结论：删除保留窗口在展示层和同步层一致。
- 推断与待确认：
  - `推断`：时区/时间格式不一致时可能导致边界误差（`toISOString().replace('T',' ')`）。

#### 3.2.3 技术方案

- 方案摘要：统一时间格式生成来源，补充 30 天边界测试。
- 架构落点（按层）：
  - UI（`views/components`）：回收站提示文案保持“30 天”。
  - 编排（`hooks`）：同步与回收站共享同一时间工具。
  - 领域（`stores`）：删除过滤调用统一 helper。
  - 持久化（`database`）：无 schema 变更。
  - 云端（`pocketbase`）：无 API 变更。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：继续采用 `updated` 单因子裁决。
- 错误处理与回滚策略：边界异常可回退到“仅软删不清理”保护策略。

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述                              | 变更文件                                                           | 依赖     | 风险 |
| ------ | ------------------------------------- | ------------------------------------------------------------------ | -------- | ---- |
| T1     | 统一删除窗口时间计算工具              | `src/utils/date.ts`, `src/hooks/useSync.ts`, `src/stores/notes.ts` | 无       | 低   |
| T2     | 补 30 天边界自动化测试（29/30/31 天） | `tests/**`                                                         | 测试基座 | 中   |
| T3     | 删除策略文档化（开发/测试对齐）       | `doc/**`                                                           | 无       | 低   |

#### 3.2.5 验收标准

- AC1：29 天删除项仍可见，31 天删除项被清理。
- AC2：本地与云端对同一条删除记录行为一致。
- AC3：边界时间在不同时区环境结果一致。

#### 3.2.6 测试与验证

- 单元测试：删除窗口判断函数。
- 集成/冒烟：本地软删 -> 同步 -> 跨端校验回收站。
- 回归关注点：永久删除不会误删未过期记录。

#### 3.2.7 风险与未决问题

- 风险：依赖客户端时间，设备时钟漂移会影响裁决。
- 未决问题：是否需要服务端时间兜底。

### 3.3 F3.3 Realtime 与手动同步操作面板

#### 3.3.1 目标与边界

- 用户场景：用户登录后自动接收变更，必要时手动触发同步并查看结果。
- 范围内：Realtime 连接生命周期、手动同步入口、同步状态展示。
- 范围外：消息推送通知体系。

#### 3.3.2 现状证据

- 代码事实：
  - `E1` [src/App.vue:70]：登录态变化后统一触发 Realtime + Sync。
  - `E2` [src/adapters/pocketbase/realtime-adapter.ts:51]：订阅 `notes` 并过滤 `user_id`。
  - `E3` [src/adapters/pocketbase/realtime-adapter.ts:217]：重连使用指数退避。
  - `E4` [src/components/UserProfile.vue:145]：手动同步完成弹窗展示统计。
- 现状结论：已具备“自动 + 手动”双入口。
- 推断与待确认：
  - `推断`：当前状态展示粒度偏粗，缺少“同步进行到哪一步”可见性。

#### 3.3.3 技术方案

- 方案摘要：增强状态面板细粒度状态，统一自动与手动同步口径。
- 架构落点（按层）：
  - UI（`views/components`）：显示 `currentStep/progress`。
  - 编排（`hooks`）：在 `syncStatus` 中填充阶段进度。
  - 领域（`stores`）：无。
  - 持久化（`database`）：无。
  - 云端（`pocketbase`）：无。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：复用现有 `syncStatus` 字段。
  - 兼容与迁移：无。
- 同步与冲突策略：不变。
- 错误处理与回滚策略：Realtime 连接失败继续允许手动同步。

#### 3.3.4 实施任务拆分

| 任务ID | 任务描述                       | 变更文件                         | 依赖      | 风险 |
| ------ | ------------------------------ | -------------------------------- | --------- | ---- |
| T1     | 在 `useSync` 注入阶段进度更新  | `src/hooks/useSync.ts`           | 无        | 低   |
| T2     | UserProfile 展示进度与错误建议 | `src/components/UserProfile.vue` | T1        | 低   |
| T3     | Realtime 断线场景回归测试      | `tests/**`                       | 网络 mock | 中   |

#### 3.3.5 验收标准

- AC1：登录后可自动建立 Realtime 连接并完成一次静默同步。
- AC2：手动同步可展示上传/下载/删除结果。
- AC3：Realtime 失败时不阻塞手动同步。

#### 3.3.6 测试与验证

- 单元测试：Realtime 状态机切换。
- 集成/冒烟：登录 -> 自动同步 -> 手动同步 -> 退出登录断连。
- 回归关注点：退出登录必须断开 Realtime。

#### 3.3.7 风险与未决问题

- 风险：重连期间重复连接可能造成重复订阅。
- 未决问题：是否需要“前台/后台”区分连接策略。
