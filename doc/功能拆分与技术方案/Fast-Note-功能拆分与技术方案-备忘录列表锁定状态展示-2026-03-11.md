## 2. 模块目标与现状

- 业务目标：
  - 让用户在备忘录列表中直接识别已锁定 note，同时保持锁定和未锁定条目的标题对齐稳定，并兼容浅色/深色主题。
- 当前状态（已支持/部分支持/未支持）：部分支持。
- 关键证据：
  - `E1` `src/components/NoteListItem.vue:112`：文件夹行已经在标题左侧渲染 `folderOutline`，可直接作为锁图标的尺寸和间距参考。
  - `E2` `src/components/NoteListItem.vue:136`：note 行当前没有左侧状态槽位，标题从 `IonLabel` 起始，无法展示锁定状态。
  - `E3` `src/types/index.ts:20`：`Note` 已定义 `is_locked` 字段，不需要新增数据字段。
  - `E4` `src/types/index.ts:81`：缺失的锁字段会被标准化为 `is_locked = 0`，符合“缺失按未锁定处理”。
  - `E5` `src/stores/notes.ts:222`：列表数据进入 store 时统一执行 `normalizeNoteLockFields()`。
  - `E6` `src/database/dexie.ts:36`：本地数据库已包含 `is_locked` 字段，无需 schema 调整。
  - `E7` `src/hooks/useTheme.ts:24`：当前主题系统通过 `ion-palette-dark` 切换深色模式。
  - `E8` `src/css/var.scss:47`：现有样式 token 已提供图标色变量 `--c-icon`，可作为锁图标主题色候选。
- 已知缺口：
  - 列表页没有任何锁状态视觉标识。
  - note 行没有固定左侧槽位，若直接加图标会导致锁定/未锁定标题起始位置不一致。
  - 当前无针对该展示需求的自动化测试或明确冒烟清单。

## 3. 逐功能点方案

### 3.1 F1.1 列表项锁图标与占位布局

#### 3.1.1 目标与边界

- 用户场景：
  - 用户在首页、文件夹页、搜索结果等复用列表项的场景中，快速识别哪些 note 已锁定。
- 范围内：
  - note 行左侧增加固定宽度状态槽位。
  - `is_locked = 1` 时展示锁图标。
  - `is_locked = 0` 或字段缺失时不展示图标内容，但保留占位。
- 范围外：
  - 文件夹行增加锁状态。
  - 锁图标点击交互、提示文案或弹层。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` `src/components/NoteListItem.vue:112`：文件夹行当前结构为“左侧 `IonIcon` + 右侧 `IonLabel`”。
  - `E2` `src/components/NoteListItem.vue:136`：note 行当前直接渲染 `IonLabel`，没有图标容器。
  - `E3` `src/components/NoteListItem.vue:140`：标题文本和右侧日期区域共处同一 `h2` flex 行中。
  - `E4` `src/components/NoteList.vue:222`：所有列表项统一复用 `NoteListItem`，改造可覆盖首页和文件夹树场景。
  - `E5` `src/types/index.ts:73`：标准化后只有 `is_locked === 1` 才被视为锁定。
- 现状结论：
  - 现有组件具备清晰的“leading icon + label”模式，但仅用于文件夹；note 行缺失同类结构，因此需要在 `NoteListItem` 内部补统一状态槽位。
- 推断与待确认：
  - 推断：最佳改造点是仅对 note 行增加一个固定宽度的前导容器，而不是把图标拼进标题文本内部。
  - 待确认：多行标题时锁图标以视觉居中还是首行基线对齐为准；若没有设计稿，建议先按文件夹图标的视觉居中对齐实现。

#### 3.1.3 技术方案

- 方案摘要：
  - 在 `NoteListItem` 的 note 分支新增前导状态槽位，宽度与现有文件夹图标区域保持同量级；槽位内部按 `is_locked` 决定显示锁图标还是空白占位，从而保证所有标题左边界一致。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/components/NoteListItem.vue`
      - note 分支新增 `IonIcon` 或等价容器；
      - 对 `noteData.is_locked === 1` 渲染锁图标；
      - 对未锁定 note 保留同宽空白槽位；
      - 调整 `IonItem` 和 `IonLabel` 样式，避免图标挤压摘要与时间。
    - `src/components/NoteList.vue`
      - 无需改数据流，仅确认“全部备忘录 / 备忘录 / 最近删除”等伪文件夹项继续走文件夹分支。
  - 编排（`hooks`）：
    - 无改动；列表展示直接消费标准化后的 `noteData.is_locked`。
  - 领域（`stores`）：
    - 无改动；继续由 `useNote` 提供标准化后的 note 数据。
  - 持久化（`database`）：
    - 无改动。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动。
- 数据契约变更：
  - 字段/类型：
    - 不新增字段，继续使用 `Note.is_locked`。
  - 兼容与迁移：
    - 缺失 `is_locked` 的旧数据已由 `normalizeNoteLockFields()` 归一为 `0`，无需迁移。
- 同步与冲突策略：
  - 本次仅消费现有锁状态字段，不新增写操作，不影响 offline-first 语义和同步冲突裁决。
- 错误处理与回滚策略：
  - 若图标资源或样式异常，回退到“仅保留空白槽位”也不会影响业务数据。
  - 若出现列表对齐回归，可仅回退 `NoteListItem` 的状态槽位样式，不需要回退锁逻辑。

#### 3.2.5 验收标准

- AC1：`is_locked = 1` 的 note 在列表左侧显示锁图标。
- AC2：`is_locked = 0` 或字段缺失的 note 不显示锁图标内容，但保留与锁图标同宽的空白占位。
- AC3：锁定和未锁定 note 在同一列表中展示时，标题起始位置一致。
- AC4：长标题换行时，锁图标位置稳定，不遮挡标题、摘要或时间。

#### 3.2.6 测试与验证

- 单元测试：
  - 规划 `tests/unit/components/note-list-item-lock-indicator.spec.ts`，验证锁定、未锁定、字段缺失三种渲染结果。
- 集成/冒烟：
  - 验证首页列表、文件夹页列表、搜索结果列表是否都正确展示锁图标和占位。
  - 验证桌面端与移动端列表项宽度变化时，标题起始位置仍一致。
- 回归关注点：
  - note 点击跳转不受影响。
  - 文件夹行图标和展开箭头不受影响。
  - `showParentFolder` 场景下底部父文件夹信息仍正常显示。

#### 3.2.7 风险与未决问题

- 风险：
  - R1：若直接复用文件夹图标的 `mr-3` 间距但没有统一槽位宽度，未锁定项仍可能出现像素级错位。
  - R2：若状态槽位宽度设置过大，长标题会更早换行。
- 未决问题：
  - Q1：锁图标是完全复用文件夹图标区域宽度，还是只复用视觉尺寸、间距单独微调。

### 3.2 F1.2 深色模式与视觉复用

#### 3.2.1 目标与边界

- 用户场景：
  - 用户切换浅色/深色主题后，锁图标应保持清晰可见，且与现有文件夹图标尺寸风格一致。
- 范围内：
  - 锁图标颜色 token 选择。
  - 锁图标尺寸、线宽、间距与文件夹图标视觉对齐。
- 范围外：
  - 全局主题系统重构。
  - 自定义图标集替换。

#### 3.2.2 现状证据

- 代码事实：
  - `E6` `src/hooks/useTheme.ts:24`：应用通过在根节点切换 `ion-palette-dark` 进入深色模式。
  - `E7` `src/css/var.scss:47`：浅色模式定义了 `--c-icon` 图标色 token。
  - `E8` `src/css/var.scss:53`：深色模式下同一批 token 会替换为 dark 版本。
  - `E9` `src/components/NoteListItem.vue:112`：文件夹图标当前使用 `folder-icon mr-3 primary` 样式，可作为尺寸和间距参考。
- 现状结论：
  - 主题切换与色值 token 已有基础设施，本需求不需要新增 dark-mode 机制，只需要把锁图标接到现有 token。
- 推断与待确认：
  - 推断：锁图标不宜使用过强强调色，优先复用 `--c-icon` 或与文件夹图标同层级图标色，避免列表噪音过大。
  - 待确认：若产品希望锁定 note 更显眼，可退一步改为 `primary` 或更高对比度 token。

#### 3.2.3 技术方案

- 方案摘要：
  - 锁图标尺寸以文件夹图标为视觉基准，颜色优先复用现有图标 token，并依赖 `ion-palette-dark` 现有切换机制完成深色模式适配。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/components/NoteListItem.vue`
      - 为锁图标增加独立 class，例如 `note-lock-icon`；
      - 在 scoped/unscoped 样式中设置与文件夹图标接近的 `font-size`、占位宽度和间距。
  - 编排（`hooks`）：
    - 无改动。
  - 领域（`stores`）：
    - 无改动。
  - 持久化（`database`）：
    - 无改动。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动。
- 数据契约变更：
  - 字段/类型：
    - 无。
  - 兼容与迁移：
    - 无。
- 同步与冲突策略：
  - 纯样式改造，不涉及同步。
- 错误处理与回滚策略：
  - 若锁图标在深色模式下对比度不足，优先切换到更高可读性的图标 token；不修改业务逻辑。

| T-FN-LOCKLIST-04 | 将锁图标颜色接入现有主题 token，验证浅色/深色模式对比度 | `src/components/NoteListItem.vue`, `src/css/var.scss` | T-FN-LOCKLIST-03 | 中 |

#### 3.1.6 测试与验证

- 单元测试：
  - 可选快照测试，校验锁图标 class 在锁定态下存在，在未锁定态下仅保留占位 class。

  - 手动切换浅色/深色模式，验证锁图标对比度、尺寸和占位。

- 未决问题：
  - Q2：最终颜色是复用 `--c-icon`、`--primary`，还是定义新的列表状态图标色 token。

### 3.3 F1.3 列表展示回归验证

#### 3.3.1 目标与边界

- 用户场景：
  - 该需求虽然只改展示层，但涉及列表通用组件，需要用最小验证覆盖多个入口和边界数据，避免回归。
- 范围内：
  - 锁定态、未锁定态、字段缺失、多行标题、主题切换的验证。
- 范围外：
  - 锁管理流程和解锁流程自动化覆盖。

    - 无改动。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动。
- 数据契约变更：
  - 字段/类型：
