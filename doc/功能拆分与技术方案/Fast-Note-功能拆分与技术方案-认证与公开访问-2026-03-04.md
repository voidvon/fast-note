  - `E4` [src/shared/api/pocketbase/auth.ts:94]：`authRefresh` 失败后清理 authStore。
  - `E5` [src/shared/api/pocketbase/users.ts:30]：公开用户通过 `public_users` 查询。
  - `E6` [src/pages/user-public-notes/ui/user-public-notes-page.vue:61]：公开主页加载流程。
  - `E7` [src/entities/public-note/model/state/public-note-store.ts:11]：每个公开用户独立 Dexie 库。
- 已知缺口：
  - `代码事实`：公开页桌面详情组件传参为 `note-uuid`，而详情组件定义为 `noteId`（参数错配风险）。

## 3. 逐功能点方案

### 3.1 F4.1 登录注册与会话生命周期

#### 3.1.1 目标与边界

- 用户场景：用户注册后自动登录，后续可免重复登录并安全退出。
- 范围内：登录、注册、会话恢复、登出。
- 范围外：多因子认证、第三方 OAuth。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` [src/pages/login/ui/login-page.vue:63]：调用 `login(...)` 完成登录。
  - `E2` [src/pages/register/ui/register-page.vue:87]：调用 `register(...)` 并自动回退登录态。
  - `E3` [src/processes/session/model/auth-manager.ts:100]：先读取本地认证用户，再刷新验证。
  - `E4` [src/shared/api/pocketbase/auth.ts:94]：`authRefresh` 校验 token 有效性。
- 现状结论：会话恢复机制可用，认证失败可清理本地脏 token。
- 推断与待确认：
  - `待确认`：长时间离线后恢复时的用户提示策略（静默登出 vs 弹窗提示）。

#### 3.1.3 技术方案

- 方案摘要：保持 `authManager` 统一入口，补登录/注册失败分类提示与会话过期引导。
- 架构落点（按层）：
  - 页面/UI（`src/pages/login/*`, `src/pages/register/*`）：明确区分输入错误、网络错误、会话过期提示。
  - 编排（`src/processes/session/*`）：保持 `authManager` 作为统一认证入口。
  - 领域（`src/entities/auth/*`）：继续暴露登录态查询能力，不额外引入新 store。
  - 持久化（`src/shared/lib/storage/*`）：无。
  - 云端（`src/shared/api/pocketbase/*`）：继续使用 `authWithPassword/authRefresh`。
  - 路由（`src/app/router/*`）：会话过期后可引导到 `/login`。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：登录态变化触发同步模块，不在本功能点重复实现。
- 错误处理与回滚策略：认证异常时 `authStore.clear()`，回退到未登录态。

| ------ | -------------------------- | ----------------------------------------------------------------------------------- | -------- | ---- |

#### 3.1.5 验收标准

- AC1：注册成功后可自动登录。
- AC2：刷新页面后可恢复有效会话。
- AC3：token 失效时清理登录态并给出可理解提示。

#### 3.1.6 测试与验证

- 单元测试：`authManager.initialize/login/register/logout`。
- 集成/冒烟：注册 -> 退出 -> 登录 -> 刷新恢复。
- 回归关注点：退出必须同时断开 Realtime（跨模块）。

#### 3.1.7 风险与未决问题

- 风险：并发登录请求可能导致 loading 状态抖动。
- 未决问题：是否需要“记住我”与会话时长配置。

### 3.2 F4.2 公开用户内容读取与本地隔离缓存

#### 3.2.1 目标与边界

- 用户场景：访客访问 `/:username` 查看该用户公开文件夹与笔记。
- 范围内：用户信息读取、公开笔记拉取、用户级本地库隔离。
- 范围外：公开内容编辑、公开评论互动。

#### 3.2.2 现状证据

- 代码事实：
  - `E1` [src/shared/api/pocketbase/users.ts:30]：用户名映射 `public_users`。
  - `E2` [src/shared/api/pocketbase/notes.ts:196]：按 `is_public=true && user_id` 拉取公开笔记。
  - `E3` [src/entities/public-note/model/state/public-note-store.ts:11]：库名 `UserPublicNotes_{username}`。
  - `E4` [src/processes/public-notes/model/use-user-public-notes-sync.ts:9]：`username -> userInfo -> publicNotes`。
- 现状结论：公开数据链路完整，具备按用户隔离本地缓存能力。
- 推断与待确认：
  - `推断`：公开数据刷新触发路径分散在路由守卫与页面初始化，维护成本较高。

#### 3.2.3 技术方案

- 方案摘要：统一公开数据初始化入口，减少守卫和页面双处重复触发。
- 架构落点（按层）：
  - 页面/UI（`src/pages/user-public-notes/*`）：公开页保留错误态/空态。
  - 编排（`src/processes/public-notes/*`）：统一公开数据初始化入口。
  - 领域（`src/entities/public-note/*`）：继续使用按用户名分库。
  - 持久化（`src/shared/lib/storage/*`）：无 schema 变化。
  - 云端（`src/shared/api/pocketbase/*`）：维持 `public_users + notes` 查询。
  - 路由（`src/app/router/*`）：守卫仅调用统一入口。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：公开链路单向拉取，无冲突写回。
- 错误处理与回滚策略：请求失败保留本地缓存并提示“数据可能非最新”。

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述                     | 变更文件                                                                                          | 依赖         | 风险 |
| ------ | ---------------------------- | ------------------------------------------------------------------------------------------------- | ------------ | ---- |
| T1     | 抽取统一公开上下文初始化函数 | `src/processes/public-notes/model/use-user-public-notes-sync.ts`, `src/app/router/index.ts`, `src/pages/user-public-notes/ui/user-public-notes-page.vue` | 无           | 中   |
| T2     | 公开页显示缓存时间与刷新状态 | `src/pages/user-public-notes/ui/user-public-notes-page.vue`                                                               | T1           | 低   |
| T3     | 公开路径冒烟回归（含错误态） | `tests/**`                                                                                        | 测试用户数据 | 中   |

#### 3.2.5 验收标准

- AC1：访问 `/:username` 可展示公开文件夹树。
- AC2：用户不存在时显示错误态而非空白页。
- AC3：公开数据支持手动刷新并更新本地缓存。

#### 3.2.6 测试与验证

- 单元测试：`useUserPublicNotesSync` 在有/无用户信息场景。
- 集成/冒烟：`/:username` -> `/:username/f/:id` -> `/:username/n/:id`。
- 回归关注点：私有用户数据与公开缓存不混淆。

#### 3.2.7 风险与未决问题

- 风险：多公开用户频繁切换时，本地库数量增长。
- 未决问题：是否需要公开缓存清理策略（LRU/TTL）。

### 3.3 F4.3 公开详情桌面态参数对齐（稳定性修复）

#### 3.3.1 目标与边界

- 用户场景：桌面端在公开主页选择公开笔记，应正常显示详情。
- 范围内：组件传参名称、公开详情加载路径。
- 范围外：详情编辑能力（公开详情应只读）。

#### 3.3.2 现状证据

- 代码事实：
  - `E1` [src/pages/user-public-notes/ui/user-public-notes-page.vue:170]：`<NoteDetail :note-uuid="state.noteUuid" />`。
  - `E2` [src/pages/note-detail/ui/note-detail-page.vue:27]：组件 props 为 `noteId`。
  - `E3` [src/pages/note-detail/ui/note-detail-page.vue:62]：详情 ID 来源于 `route.params` 或 `props.noteId`。
- 现状结论：存在显式参数命名错配，桌面公开详情有加载失败风险。
- 推断与待确认：
  - `推断`：该问题在移动端不明显，因为移动端主要走路由参数。

#### 3.3.3 技术方案

- 方案摘要：将公开页桌面模式改为 `:note-id="state.noteUuid"`，并补充回归测试。
- 架构落点（按层）：
  - 页面/UI（`src/pages/user-public-notes/*`, `src/pages/note-detail/*`）：仅改 props 绑定。
  - 编排（`src/processes/public-notes/*`）：无。
  - 领域（`src/entities/public-note/*`）：无。
  - 持久化（`src/shared/lib/storage/*`）：无。
  - 云端（`src/shared/api/pocketbase/*`）：无。
  - 路由（`src/app/router/*`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：无。
- 错误处理与回滚策略：变更单点可快速回滚。

#### 3.3.4 实施任务拆分

| 任务ID | 任务描述                      | 变更文件                            | 依赖         | 风险 |

#### 3.3.5 验收标准

- AC1：桌面端公开主页点击笔记后，详情可正常加载。
- AC2：公开详情保持只读。
- AC3：移动端行为不受影响。

#### 3.3.6 测试与验证

- 单元测试：无（以集成测试为主）。
- 集成/冒烟：桌面态 `/:username` 选择笔记并验证标题/内容渲染。
- 回归关注点：私有详情 props 绑定不受影响。

#### 3.3.7 风险与未决问题

- 风险：命名修复后若存在旧调用方，可能暴露其他隐藏错配。
- 未决问题：是否统一约定组件 props 只使用 `kebab-case note-id`。
