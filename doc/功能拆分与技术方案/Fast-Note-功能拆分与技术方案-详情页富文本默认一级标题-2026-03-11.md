## 2. 模块目标与现状

- 业务目标：
  - 让用户在新建详情页时直接从一级标题开始输入，减少一次格式切换，并保持已有内容、离线保存和同步链路不回退。
- 当前状态（已支持/部分支持/未支持）：部分支持。
- 关键证据：
  - `E1` `src/views/NoteDetail.vue:121`：新建详情页分支会立即清空编辑器并延迟聚焦，是默认标题接入的主入口。
  - `E2` `src/hooks/useEditor.ts:99`：编辑器统一在 `useEditor` 中创建，当前默认 `content` 为空字符串。
  - `E3` `src/components/TextFormatModal.vue:45`：编辑器已具备一级标题命令能力，说明不需要新增富文本扩展。
  - `E4` `src/hooks/useEditor.ts:221`：标题抽取取首个块文本，已兼容 `heading` 与 `paragraph`。
  - `E5` `src/views/NoteDetail.vue:342`：当前空内容判断基于 HTML 字符串是否为空，无法区分“默认空标题骨架”和“用户真实输入”。
  - `E6` `src/types/index.ts:36`：`Note.content` 仍是字符串富文本内容，无独立标题字段。
- 已知缺口：
  - 新建详情页没有默认一级标题能力。
  - 如果直接注入空 `h1`，现有保存逻辑会误把它当成真实内容。
  - 当前自动化测试尚未覆盖“新建默认标题 + 不误保存空白草稿”的组合场景。

## 3. 逐功能点方案

### 3.1 F1.1 新建详情页首块默认一级标题

#### 3.1.1 目标与边界

- 用户场景：
  - 用户进入空白详情页后，不需要打开格式面板，也不需要先输入正文再改样式，首个输入位置就是一级标题。
- 范围内：
  - 私有详情页新建场景（`id === '0'`）的编辑器初始化。
  - 新建空白编辑器首块的默认块类型设定。
- 范围外：
  - 已有笔记、公开只读详情页、锁定态详情页内容重写。
  - 文本格式面板交互改造。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` `src/views/NoteDetail.vue:121`：新建笔记时会执行 `editorRef.value.setContent('')` 并延迟 `focus()`。
  - `E2` `src/hooks/useEditor.ts:128`：当前编辑器初始 `content` 为 `''`，没有默认块类型注入。
  - `E3` `src/components/TextFormatModal.vue:45`：手动切换一级标题通过 `toggleHeading({ level: 1 })` 完成，说明 `StarterKit` 已支持标题节点。
  - `E4` `src/hooks/useEditor.ts:203`：标题提取逻辑已将 `heading` 与 `paragraph` 同等纳入首块文本抽取。
- 现状结论：
  - 编辑器能力层已具备一级标题节点和现成命令，但详情页新建链路仍停留在“空白 paragraph 起步”。
- 推断与待确认：
  - 推断：最佳改造点不是在 `TextFormatModal` 做被动切换，而是在 `NoteDetail` 新建分支和 `YYEditor/useEditor` 暴露一个“仅对空编辑器生效”的默认块能力。
  - 待确认：默认一级标题是否只在首次进入新建详情页生效，还是在空草稿再次恢复时也应继续保持。

#### 3.1.3 技术方案

- 方案摘要：
  - 为编辑器增加“空文档默认块类型”能力，仅在新建详情页、编辑器当前没有真实内容时，将首块置为 `heading level=1`；已有内容加载与只读场景完全绕过。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/views/NoteDetail.vue`：
      - 在 `id === '0'` 的新建分支，不再只做 `setContent('') + focus()`；
      - 改为调用 `editorRef.value.applyDefaultNewNoteHeading()` 或等价能力，在空文档上插入一级标题起始块。
    - `src/components/YYEditor.vue`：
      - 对父组件暴露默认块初始化方法，避免页面层直接拼 Tiptap 命令细节。
  - 编排（`hooks`）：
    - `src/hooks/useEditor.ts`：
      - 新增 `applyDefaultHeadingIfEmpty()` 之类的 helper；
      - helper 内部先判断当前文档是否为空，再执行 `setHeading({ level: 1 })` 或等价初始化。
  - 领域（`stores`）：
    - 无改动，笔记新增/更新仍通过 `useNote`。
  - 持久化（`database`）：
    - 无改动。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动，仍以 `id === '0'` 区分新建场景。
- 数据契约变更：
  - 字段/类型：
    - `Note`、Dexie schema、PocketBase notes 字段均不变。
    - 持久化内容仍为 `content: string`，只是首块 HTML 可能以 `h1` 起步。
  - 兼容与迁移：
    - 无历史数据迁移。
- 同步与冲突策略：
  - 继续沿用当前 `updated` 驱动的同步策略；
  - 仅当用户产生真实内容后才进入新增/更新链路，不改变云端冲突处理规则。
- 错误处理与回滚策略：
  - 若默认标题初始化失败，回退为当前空白编辑器 + 手动切换格式的现状。
  - 方法实现需保证幂等，多次调用不会重复插入多个空标题块。

| T-FN-030 | 在 `NoteDetail` 新建分支接入默认一级标题初始化，仅作用于 `id === '0'` 空白详情页 | `src/views/NoteDetail.vue` | T-FN-029 | 中 |

#### 3.1.5 验收标准

- AC1：用户进入新建详情页后，未手动切换格式直接输入，首行内容以一级标题样式展示。
- AC2：用户打开已有笔记时，系统按现有内容渲染，不自动将首块改为一级标题。
- AC3：用户在新建详情页手动切换为正文或其他格式后，系统不再强制改回一级标题。

#### 3.1.6 测试与验证

- 单元测试：
  - 规划 `tests/unit/views/note-detail-default-heading.spec.ts`，验证新建分支会触发默认一级标题能力，而已有笔记不会触发。
- 集成/冒烟：
  - 规划 `tests/integration/editor/new-note-default-heading.spec.ts`，覆盖新建详情页首行默认一级标题、切换格式后继续编辑。
  - 可扩展 `tests/integration/home/desktop-empty-detail-create.spec.ts`，在“点击空白详情开始新建”后继续验证首块样式。
- 回归关注点：
  - 桌面端空白详情点击进入新建。
  - 移动端 `id === '0'` 新建后首次聚焦。
  - 锁定笔记、公开笔记、缺失笔记等非可编辑场景不应误触发默认标题。

#### 3.1.7 风险与未决问题

- 风险：
  - R1：若默认标题能力直接写入空 `h1` HTML，而不做空内容语义兜底，可能触发误保存空白笔记。
  - R2：若默认标题初始化时机晚于 `focus()`，用户可能短暂看到正文态再切成标题态，造成闪动。
- 未决问题：
  - Q1：空白草稿重新进入详情页时，是否也应恢复一级标题默认态。
  - Q2：回车后的下一行是否继续采用编辑器默认行为，还是产品上强制切为正文。

### 3.2 F1.2 默认标题骨架的空内容判定与保存兜底

#### 3.2.1 目标与边界

- 用户场景：
  - 用户进入新建详情页后未输入任何有效内容就离开，不应因为系统预置了一个空一级标题而自动生成一条空白笔记。
- 范围内：
  - 详情页保存前的内容有效性判断。
  - 默认空 `h1`、空 `p` 等骨架节点的语义空内容识别。
- 范围外：
  - 附件上传、表格编辑、同步重试策略重构。

#### 3.2.2 现状证据

- 代码事实：
  - `E5` `src/views/NoteDetail.vue:342`：新建空笔记仅在 `!content` 时才跳过保存。
  - `E6` `src/views/NoteDetail.vue:387`：只要 `content` 为真值，就会进入新增或更新逻辑。
  - `E7` `src/views/NoteDetail.vue:469`：当前“空内容删除”也依赖 HTML 字符串真假，而非语义层判断。
  - `E8` `src/hooks/useEditor.ts:245`：页面层当前拿到的是原始 HTML 字符串。
- 现状结论：
  - 当前保存链路默认假设“HTML 非空 = 用户有真实内容”，这个假设在引入默认空标题骨架后不再成立。
- 推断与待确认：
  - 推断：应把“内容是否有效”封装到编辑器层，而不是让 `NoteDetail` 继续用字符串真假做判断。
  - 待确认：附件-only、表格-only、任务列表-only 笔记是否都应视为“有真实内容”。从现有行为看，建议视为有内容并保留。

#### 3.2.3 技术方案

- 方案摘要：
  - 在 `useEditor` 增加语义空内容判断方法，例如 `isMeaningfulContent()`；`NoteDetail` 保存时改为基于语义判断而非原始 HTML 字符串真假。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/components/YYEditor.vue`：
      - 暴露 `isMeaningfulContent()` 给 `NoteDetail` 使用。
  - 编排（`hooks`）：
    - `src/hooks/useEditor.ts`：
      - 基于 `editor.getJSON()` 遍历节点；
      - 将“只有空 heading/paragraph/空列表容器”的情况视为无有效内容；
      - 将非空文本、附件节点、表格、任务项等视为有效内容。
  - 领域（`stores`）：
    - 无改动。
  - 持久化（`database`）：
    - 无改动。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动。
- 数据契约变更：
  - 字段/类型：
    - 无新增字段。
  - 兼容与迁移：
    - 无数据库迁移；仅调整保存前判定规则。
- 同步与冲突策略：
  - 只有被判定为“有真实内容”的编辑结果才会进入本地写入和后续同步；
  - 不改变 `updated`、版本号和云端裁决逻辑。
- 错误处理与回滚策略：
  - 若语义判定出错，保守回退到当前字符串判断，避免阻断正常保存。
  - 判定逻辑必须覆盖附件-only 场景，避免误删已有真实内容。

| T-FN-031 | 在编辑器层新增语义空内容判定，区分空标题骨架与真实内容 | `src/hooks/useEditor.ts`, `src/components/YYEditor.vue` | T-FN-029 | 高 |
| T-FN-032 | 将 `NoteDetail` 保存逻辑改为使用语义判定，覆盖新建不保存、空内容删除、已有内容更新 | `src/views/NoteDetail.vue` | T-FN-031 | 高 |

#### 3.2.5 验收标准

- AC4：用户进入新建详情页但未输入任何有效内容后离开，不会新增空白笔记。
- AC5：用户只输入一级标题文本后保存，会正常创建笔记，且 `title` 取首个一级标题文本。
- AC6：仅含附件或其他非文本有效节点的笔记，仍会被视为有效内容并正常保存。

#### 3.2.6 测试与验证

- 单元测试：
  - 规划 `tests/unit/hooks/useEditor-meaningful-content.spec.ts`，覆盖空 `h1`、空 `p`、非空 `h1`、附件-only 内容。
  - 规划 `tests/unit/views/note-detail-default-heading-save.spec.ts`，验证默认空标题不会触发新增。
- 集成/冒烟：
  - 规划 `tests/integration/editor/new-note-default-heading.spec.ts`，串联新建默认标题与离开不保存。
  - 可扩展 `tests/unit/views/note-detail-save-status.spec.ts` 的 helper，覆盖默认空标题场景下 `addNote` 不应被调用。
- 回归关注点：
  - 新建进入后直接返回。
  - 新建后输入标题再保存。
  - 新建后仅插入附件或表格。
  - 已有笔记清空内容后删除行为不回退。

#### 3.2.7 风险与未决问题

- 风险：
  - R3：语义空内容判定若遗漏特殊节点，可能导致有内容被误判为空。
  - R4：若把“空标题骨架”判断写死在页面层，后续编辑器默认块扩展会再次产生分叉逻辑。
- 未决问题：
  - Q3：是否要把“内容有效性判定”沉淀为通用编辑器能力，供其他富文本入口复用。
  - Q4：是否需要把“默认块类型”做成可配置项，为后续标题/正文策略切换预留能力。
