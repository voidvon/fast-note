## 2. 迁移结论

- 当前状态：部分支持。
- 架构结论：游客态数据处理已经从“仅有 scope 切换基础设施”推进到 `processes/session` 编排 + `shared/lib/storage/guest-data` 跨库处理，但目前只支持“合并到当前账号”与“保留游客态共存”两条路径，尚未支持“覆盖登录态”。
- 代码证据：
  - `E1` `src/processes/session/model/use-session-bootstrap.ts:44`：登录后若检测到游客数据，会弹出明确决策弹窗，而不是静默忽略。
  - `E2` `src/processes/session/model/use-session-bootstrap.ts:84`：当前已接入 `hasGuestData()` 检测。
  - `E3` `src/processes/session/model/use-session-bootstrap.ts:91`：用户选择“上传到云端”时，已执行 `mergeGuestDataIntoCurrent()` 并重新准备登录态上下文。
  - `E4` `src/shared/lib/storage/guest-data.ts:72`：游客态数据统计已覆盖 `notes`、`note_files`、`metadata`、`user_info` 四类表。
  - `E5` `src/shared/lib/storage/guest-data.ts:93`：合并流程已经能够跨库读取游客态数据并写入当前登录态数据库。
  - `E6` `src/shared/lib/storage/guest-data.ts:116`：`notes` 按 `id` + `updated` 做保留较新记录的 upsert。
  - `E7` `src/shared/lib/storage/guest-data.ts:128`：`note_files` 按 `hash` + `updated` 做同样的 upsert。
  - `E8` `src/shared/lib/storage/guest-data.ts:181`：合并完成后会清空游客态数据库。
  - `E9` `src/shared/lib/user-scope/index.ts:3`：游客态仍使用独立 `guest` scope 和独立数据库名。
  - `E10` `src/processes/session/model/prepare-session-context.ts:4`：登录态与游客态切换最终都通过 `initializeDatabase(userId)` + `initializeNotes()` 完成上下文切换。
- 现状结论：当前已经具备“登录后检测游客数据并选择是否合并”的最小闭环，真实完成度应定义为“部分支持”。

## 3. 功能点状态

### 3.1 F1.1 登录后游客态数据检测与弹窗决策

- 当前状态：部分支持。
- 已落地能力：
  - 仅在“上一时刻未登录、当前时刻已登录”时触发游客态数据决策。
  - 弹窗已提供两个分支：`coexist` 和 `merge`。
  - 游客态无数据时不会弹窗。
- 当前缺口：
  - 还没有“覆盖登录态”选项。
  - 也没有覆盖场景需要的二次确认。
  - 弹窗当前使用 `alertController` 直接实现，未抽成独立业务组件。

### 3.2 F1.2 合并策略（按 `id` + `updated`）

- 当前状态：部分支持。
- 已落地能力：
  - `notes` 已按 `id` + `updated` 合并，游客态更新更晚时会覆盖当前登录态记录。
  - `note_files` 已按 `hash` + `updated` 合并。
  - `metadata` 与 `user_info` 已会把游客库中“当前登录态不存在”的记录补写到当前库。
  - 合并完成后会重新加载登录态数据库上下文。
- 当前缺口：
  - `metadata` 与 `user_info` 当前只补入 guest-only 记录；若登录态已存在同 key / id，会保留登录态版本。
  - 时间相同的记录会保留当前登录态版本，未提供更细粒度冲突策略。
  - 合并完成后只是写入当前库，后续仍依赖既有同步链路再上传到云端。

### 3.3 F1.3 覆盖策略（全量）

- 当前状态：未支持。
- 真实情况：
  - 代码中没有“清空登录态后导入游客态”的实现。
  - 弹窗也没有覆盖分支入口。
- 后续若要补：
  - 需要单独设计不可逆操作确认、逐表清空顺序和失败回滚策略。

### 3.4 F1.4 共存与登出恢复

- 当前状态：部分支持。
- 已落地能力：
  - 用户可以选择 `coexist`，此时不会迁移游客数据。
  - 游客态与登录态本身就分属不同作用域数据库，登出后再次切回游客态上下文是可行的。
- 当前缺口：
  - “共存”本质上是保持两套数据库并在登录态不触碰游客库，没有额外说明或恢复提示。
  - 没有“下次登录是否还提示”的显式控制。

## 4. 按层落点

### 4.1 `processes`

- `src/processes/session/model/use-session-bootstrap.ts`
  - 承担登录态变化时的游客数据检测、弹窗决策和后续会话重建编排。
  - 决策发生在登录成功后的会话初始化阶段，而不是在页面层临时判断。

### 4.2 `shared`

- `src/shared/lib/storage/guest-data.ts`
  - 承担游客库统计、跨库读取、合并写入、清空游客库等基础设施逻辑。
- `src/shared/lib/user-scope/index.ts`
  - 承担游客态与登录态数据库作用域切换。
- `src/shared/lib/storage/dexie.ts`
  - 提供按 scope 打开的隔离数据库能力。

### 4.3 `entities`

- `src/entities/note/model/state/note-store.ts`
  - 继续负责在当前数据库上下文下重建本地笔记状态。
  - 不直接参与跨库合并规则定义。

## 5. 真实边界与风险

- 当前弹窗文案只有“仅在未登录时显示”和“上传到云端”两项，没有覆盖选项。
- 合并后会清空游客库，因此当前 `merge` 不是“复制”而是“迁移”。
- `metadata` 与 `user_info` 已纳入检测与合并范围，但冲突时当前仍优先保留登录态版本，缺少更明确的业务策略。
- `hasGuestData()` 已覆盖 `metadata` / `user_info`；如果后续再新增游客态资产表，需要同步扩展统计口径。
- 当前没有看到该流程的自动化测试结果，后续若继续调整登录初始化时序，需重点回归游客态决策弹窗与合并结果。

## 6. 迁移状态说明

- 迁移结果：已建立正式实现，但整体完成度仍为“部分支持”。

  - 跨库统计、合并和清理已经正式收口到 `src/shared/lib/storage/guest-data.ts`。
  - 当前真实状态为“已支持检测 + 合并 + 共存，未支持覆盖”。
  - 后续仍可继续迁移的重点：
