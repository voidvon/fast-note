- 关键证据：
  - E1 `src/App.vue:43` 登录态变更后初始化本地数据库与笔记状态
  - E2 `src/utils/userScope.ts:3` 访客与登录态使用不同 scope（`guest`）
  - E3 `src/database/dexie.ts:35` 数据库按 userId 作用域切换并重新打开
  - E4 `src/stores/notes.ts:156` 笔记状态按当前数据库名重建
  - E5 `src/database/sync.ts:107` 同步时按 `updated` 判断更新
  - E6 `src/types/index.ts:14` `Note` 包含 `id` 与 `updated` 字段
- 已知缺口：
  - 登录后未检测游客态数据，亦无合并/覆盖/共存流程

## 3. 逐功能点方案

### 3.1 F1.1 登录后游客态数据检测与弹窗决策

#### 3.1.1 目标与边界

- 用户场景：用户登录后仍希望保留未登录时的备忘录
- 范围内：检测游客态数据、弹窗选择合并/覆盖/共存、覆盖二次确认
- 范围外：复杂冲突编辑器与跨设备迁移

#### 3.1.2 现状证据

- 代码事实：
  - E1 `src/App.vue:43` 登录态变更时初始化数据库与笔记状态
  - E2 `src/utils/userScope.ts:3` `guest` scope 与数据库名
  - E3 `src/database/dexie.ts:35` 数据库按 userId 切换
- 现状结论：具备按账号切换数据库的能力，但没有游客态检测与决策弹窗
- 推断与待确认：游客态“有数据”判定需明确是否包含 `note_files` 与 `metadata`

#### 3.1.3 技术方案

- 方案摘要：在登录成功后的会话初始化前，读取游客态数据库统计，若存在数据则弹窗决策
- 架构落点（按层）：
  - UI（`views/components`）：新增 `GuestDataDecisionModal` 或使用 `alertController` 实现三选一弹窗
  - 编排（`hooks`）：新增 `useGuestDataDecision`，负责流程时序与状态
  - 领域（`stores`）：复用 `initializeNotes` 以重新加载登录态数据
  - 持久化（`database`）：新增“只读打开指定 scope 数据库”的辅助函数
  - 云端（`pocketbase`）：无
  - 路由（`router`）：无
- 数据契约变更：无
- 同步与冲突策略：弹窗流程需在 `sync(true)` 前完成，避免同步早于合并/覆盖
- 错误处理与回滚策略：读取游客态失败时记录错误并继续登录态流程

#### 3.1.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 增加游客态数据统计读取接口 | `src/database/dexie.ts` | 无 | 读取失败需兜底 |
| T2 | 登录流程增加游客态检测与弹窗 | `src/App.vue` | T1 | 时序影响同步 |
| T3 | 弹窗 UI 与二次确认 | `src/components/GuestDataDecisionModal.vue` 或 `src/components/UserProfile.vue` 风格 | 无 | UX 误解 |

#### 3.1.5 验收标准

- AC1：游客态有数据时，每次登录必须弹窗
- AC2：覆盖操作需二次确认

#### 3.1.6 测试与验证

- 单元测试：游客态统计读取与判定逻辑
- 集成/冒烟：登录后弹窗显示与三选项可操作
- 回归关注点：登录初始化与同步流程不被阻断

#### 3.1.7 风险与未决问题

- 风险：弹窗阻塞登录态恢复与同步初始化
- 未决问题：游客态数据范围是否包含附件与元数据

### 3.2 F1.2 合并策略（按 `id` + `updated`）

#### 3.2.1 目标与边界

- 用户场景：用户选择“合并”，希望保留游客内容并与登录态融合
- 范围内：按 `id` 匹配，`updated` 最新覆盖旧记录
- 范围外：正文级冲突编辑

#### 3.2.2 现状证据

- 代码事实：
  - E5 `src/database/sync.ts:107` 已存在基于 `updated` 的更新策略
  - E6 `src/types/index.ts:14` `Note` 具备 `id` 与 `updated`
- 现状结论：字段与同步逻辑支持 `updated` 比较，但未实现跨库合并
- 推断与待确认：关联表（`note_files`、`metadata`）是否需同样按 `updated` 合并

#### 3.2.3 技术方案

- 方案摘要：读取游客态与登录态数据集合，按 `id` 比对并写入登录态
- 架构落点（按层）：
  - UI：复用 F1.1 弹窗
  - 编排：新增 `mergeGuestData()` 作为独立流程
  - 领域：合并后调用 `initializeNotes` 重新加载
  - 持久化：读游客态 DB + 登录态 DB，执行 `bulkPut`
  - 云端：无
  - 路由：无
- 数据契约变更：无
- 同步与冲突策略：以 `updated` 为准，时间相同保留登录态记录
- 错误处理与回滚策略：写入失败不清理登录态数据，保持可重试

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T4 | 实现按 `id` + `updated` 合并算法 | `src/database/guestData.ts`（新） | T1 | 数据一致性 |
| T5 | 合并完成后刷新本地 store | `src/stores/notes.ts` 或调用点 | T4 | UI 刷新滞后 |

#### 3.2.5 验收标准

- AC3：同 `id` 记录按 `updated` 最新保留，时间相同保留登录态
- AC4：游客态新增 `id` 进入登录态

#### 3.2.6 测试与验证

- 单元测试：合并算法（同 `id` 新旧、时间相等）
- 集成/冒烟：登录后合并完成可见数据
- 回归关注点：合并后同步不产生重复

#### 3.2.7 风险与未决问题

- 风险：`updated` 时间不可信导致覆盖错误
- 未决问题：关联表的合并策略

### 3.3 F1.3 覆盖策略（全量）

#### 3.3.1 目标与边界

- 用户场景：用户选择“覆盖”以游客态数据替换登录态
- 范围内：登录态全量清空后导入游客态
- 范围外：服务端回滚

#### 3.3.2 现状证据

- 代码事实：
  - E3 `src/database/dexie.ts:19` 表结构明确
- 现状结论：具备本地表结构，但未实现跨库覆盖
- 推断与待确认：覆盖是否包含 `user_info` 与 `metadata`

#### 3.3.3 技术方案

- 方案摘要：读取游客态全量数据，清空登录态所有表后写入
- 架构落点（按层）：
  - UI：覆盖二次确认
  - 编排：新增 `overwriteWithGuestData()`
  - 领域：覆盖后调用 `initializeNotes`
  - 持久化：逐表 `clear()` + `bulkPut()`
  - 云端：无
  - 路由：无
- 数据契约变更：无
- 同步与冲突策略：覆盖后由登录态同步流程统一推进
- 错误处理与回滚策略：若写入失败需停止清理或回滚（建议先读游客态再清空）

#### 3.3.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T6 | 实现覆盖流程（清空+写入） | `src/database/guestData.ts`（新） | T1 | 覆盖不可逆 |
| T7 | 覆盖确认与失败提示 | `src/components/GuestDataDecisionModal.vue` | T6 | UX 误操作 |

#### 3.3.5 验收标准

- AC5：覆盖后登录态仅保留游客态数据

#### 3.3.6 测试与验证

- 单元测试：覆盖后数据与游客态一致
- 集成/冒烟：覆盖后刷新 UI 与同步
- 回归关注点：覆盖前数据不可见且不残留

#### 3.3.7 风险与未决问题

- 风险：覆盖导致用户误丢失登录态数据
- 未决问题：是否需要更强的确认与备份策略

### 3.4 F1.4 共存与登出恢复

#### 3.4.1 目标与边界

- 用户场景：用户希望保留两套数据并可切换
- 范围内：共存不迁移数据，登出后恢复游客态
- 范围外：跨设备共存

#### 3.4.2 现状证据

- 代码事实：
  - E1 `src/App.vue:102` 登录态变更触发 `prepareSessionContext`
  - E2 `src/utils/userScope.ts:3` `guest` scope 可用于登出恢复
- 现状结论：登出时会切回游客态数据库，但无共存明确流程与提示
- 推断与待确认：登出后是否需提示“已切回游客态”

#### 3.4.3 技术方案

- 方案摘要：共存选项只记录选择，不迁移数据；登出时恢复游客态
- 架构落点（按层）：
  - UI：共存选择说明
  - 编排：在登录成功流程中处理共存分支
  - 领域：登出后 `initializeNotes` 重新加载游客态
  - 持久化：无额外变更
  - 云端：无
  - 路由：无
- 数据契约变更：无
- 同步与冲突策略：无
- 错误处理与回滚策略：登出切回失败时提示并保持登录态

#### 3.4.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T8 | 共存分支流程编排 | `src/App.vue` | T2 | 状态切换时序 |
| T9 | 登出后游客态恢复提示（可选） | `src/components/GuestDataDecisionModal.vue` | 无 | 信息噪音 |

#### 3.4.5 验收标准

- AC6：选择共存后，登出可恢复游客态数据展示

#### 3.4.6 测试与验证

- 单元测试：无
- 集成/冒烟：共存后登出切换正确
- 回归关注点：登出流程与路由恢复不冲突

#### 3.4.7 风险与未决问题

- 风险：用户不理解共存导致期望偏差
- 未决问题：是否需要“下次登录仍提示”开关（当前需求为必须提示）
