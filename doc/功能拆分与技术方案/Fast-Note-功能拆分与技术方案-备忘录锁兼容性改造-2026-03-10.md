## 2. 背景与问题

当前中间态实现已经引入 PIN，但模型是“每篇 note 独立 PIN”。这会带来三个问题：

1. 用户心智重，锁 3 篇备忘录可能要记 3 个 PIN。
2. 数据模型不合理，PIN hash 被放在 note 上，导致同步粒度错误。
3. “修改 PIN”语义模糊，实际会变成只修改单篇备忘录密码。

本轮改造的目标是把锁模型收口为：

- note 只表达“这篇是否受保护”
- PIN 只表达“这个账号当前使用哪个全局解锁码”
- 生物识别只表达“这个设备是否允许快捷解锁”

## 3. 目标与非目标

### 3.1 目标

- 用户全局只维护一个 6 位 PIN
- 首次锁定任意备忘录时创建 PIN
- 已存在全局 PIN 时，后续锁定其他备忘录不再创建新 PIN
- 已锁备忘录通过全局 PIN 或当前设备生物识别解锁
- 全局 PIN 同步到 PocketBase `users` 表
- 保持 offline-first，本地可先用，本地与云端后续对齐

### 3.2 非目标

- 不做 legacy WebAuthn 迁移
- 不做忘记 PIN 找回
- 不做内容级加密
- 不做每篇备忘录独立密码

## 4. 业务规则

### 4.1 锁与 PIN 的关系

- `is_locked=1` 表示该 note 需要解锁后才能查看/编辑
- 全局 PIN 不再存入 note
- 关闭某篇备忘录锁，只影响该 note 的 `is_locked`
- 修改 PIN，影响当前账号下所有已锁备忘录

### 4.2 首次锁定规则

- 用户第一次锁定任何 note 时，若系统未发现有效全局 PIN：
  - 必须先创建 6 位 PIN
  - 创建成功后再锁定当前 note
- 如果用户已存在全局 PIN：
  - 锁定当前 note 时不再出现“创建 PIN”流程
  - 直接为当前 note 开锁开关即可

### 4.3 解锁规则

- note 打开时，若 `is_locked=0`，直接进入编辑器
- 若 `is_locked=1`：
  - 本地会话有效，则直接进入
  - 本地会话无效，则显示解锁面板
- 生物识别失败、取消、不支持时，必须始终允许输入 PIN
- PIN 连续输错达到阈值后进入冷却

### 4.4 会话规则

- 会话仍按 note 维度存本地
- 修改 PIN 不自动清除所有 note 会话，但新会话一律按新 PIN 校验生成
- “立即重新锁定”只清除当前 note 的会话

## 5. 技术设计

### 5.1 分层职责

- `src/components/*`
  - 承载首次创建 PIN、解锁、管理弹层 UI
- `src/views/NoteDetail.vue`
  - 根据 `is_locked` 和会话状态决定显示编辑器还是解锁面板
- `src/hooks/useNoteLock.ts`
  - 承载全局 PIN 的本地读写、校验、锁 note、解锁会话、生物识别组合逻辑
- `src/hooks/useSync.ts` 或独立同步编排
  - 负责把全局 PIN 本地镜像与 PocketBase `users` 表对齐
- `src/database/dexie.ts`
  - 提供 `security_settings`、`device_security_state`、`note_unlock_sessions`
- `src/stores/notes.ts`
  - 继续只维护 note 本身的 `is_locked`

### 5.2 数据模型

#### 5.2.1 Note

- 保留：
  - `is_locked`
- 移除业务语义：
  - `lock_type`
  - `lock_secret_salt`
  - `lock_secret_hash`
  - `lock_version`

#### 5.2.2 Dexie

- `notes`
  - 继续保留 `is_locked`
- `security_settings`
  - `scope_key`
  - `pin_secret_salt`
  - `pin_secret_hash`
  - `pin_version`
  - `updated`
- `device_security_state`
  - `scope_key`
  - `webauthn_credential_id`
  - `biometric_enabled`
  - `updated`
- `note_unlock_sessions`
  - `note_id`
  - `verified_at`
  - `expires_at`
  - `failed_attempts`
  - `cooldown_until`
  - `updated`

#### 5.2.3 PocketBase `users`

- 新增字段：
  - `note_lock_pin_salt`
  - `note_lock_pin_hash`
  - `note_lock_pin_version`

说明：

- 字段放在 `users` 表，是因为它们属于账号级安全配置，不属于单篇 note。
- 不新增独立 collection，避免把账号安全设置拆散到新的云端实体。

### 5.3 同步策略

#### 5.3.1 note 同步

- `notes` 仍按现有增量同步
- 锁相关只同步 `is_locked`

#### 5.3.2 全局 PIN 同步

- 登录态下：
  - 首次创建 PIN 或修改 PIN 时，写本地 `security_settings`
  - 然后写 PocketBase `users`
- 拉取用户资料时：
  - 若服务端存在 PIN 配置，则回写本地 `security_settings`
- 冲突策略：
  - 游客态本地 PIN 与登录后账号 PIN 冲突时，以云端 `users` 表为准
  - 登录成功并拿到服务端 PIN 配置后，直接覆盖本地 `security_settings`
  - 仅当服务端不存在 PIN 配置时，才允许本地首次创建后上推

#### 5.3.3 生物识别同步

- 不同步
- 只存在本地设备态

## 6. 关键流程

### 6.1 首次锁定

```text
点击“锁定”
  -> 读取本地 security_settings
  -> 若无全局 PIN
    -> 打开创建 PIN 弹层
    -> 输入 PIN / 确认 PIN
    -> 写本地 security_settings
    -> 登录态同步 users 表
  -> 将当前 note.is_locked 置 1
  -> 清除当前 note 会话
```

### 6.2 解锁

```text
打开 note
  -> note.is_locked = 0 -> 直接进入
  -> note.is_locked = 1
    -> 检查 note session
    -> 有效 -> 直接进入
    -> 无效 -> 显示解锁面板
      -> 尝试生物识别
        -> 成功 -> 写当前 note session
        -> 失败 -> 停留面板
      -> 输入全局 PIN
        -> 正确 -> 写当前 note session
        -> 错误过多 -> 进入冷却
```

### 6.3 修改 PIN

```text
打开锁管理
  -> 进入“修改全局 PIN”
  -> 输入新 PIN / 确认新 PIN
  -> 写本地 security_settings
  -> 同步 users 表
  -> 返回成功提示“全局 PIN 已更新”
```

## 8. 验收标准

- AC1：用户首次锁定任意备忘录时，若账号尚无 PIN，必须先创建全局 PIN
- AC2：用户锁定第二篇、第三篇备忘录时，不再被要求创建新 PIN
- AC3：任意已锁备忘录都使用同一个 PIN 解锁
- AC4：修改 PIN 后，其余已锁备忘录后续均按新 PIN 解锁
- AC5：关闭某篇备忘录锁时，不删除全局 PIN
- AC6：PIN 配置在多设备登录同一账号后可通过 `users` 表保持一致
- AC7：生物识别失败或不支持时，PIN 入口始终存在
