## 3. 当前业务规则

### 3.1 锁模型

- `note.is_locked` 只表达“这篇笔记是否需要解锁”。
- 全局 PIN 不再存放在 note 记录上，而是进入当前账号作用域的 `security_settings`。
- 生物识别开关和凭据只保存在当前设备的 `device_security_state`。
- note 解锁成功后，会生成该 note 的本地临时会话，而不是生成全局“已解锁”状态。

### 3.2 首次锁定

- 如果当前账号或当前作用域还没有全局 PIN，锁定任意 note 时必须先完成 PIN 创建。
- PIN 创建成功后立即锁定当前 note，并清除该 note 的旧解锁会话。
- 若已存在全局 PIN，则后续锁定其他 note 不再重复要求创建 PIN。

### 3.3 解锁

- 详情页打开已锁 note 时，优先检查 note 级本地会话。
- 会话有效则直接进入编辑器；会话失效则展示 `NoteUnlockPanel`。
- 用户可以通过 PIN 解锁，也可以在设备支持且已开启时走生物识别快捷解锁。
- PIN 连续输错达到阈值后会进入冷却，当前默认阈值和冷却时长分别为 3 次、30 秒。

### 3.4 同步

- 登录态下，全局 PIN 会同步到 `users` 表。
- 云端存在 PIN 配置时，会优先回写本地 `security_settings`。
- 生物识别能力与凭据不做跨设备同步，只在本机有效。

## 5. 真实边界与剩余限制

- 全局 PIN 仅支持 6 位数字，不支持找回流程，也不做内容级加密。
- 生物识别是设备本地快捷解锁能力，不会随账号跨设备漫游。
- 云端同步依赖 PocketBase `users` 表已开放 `note_lock_pin_salt/hash/version` 字段；缺字段时会返回明确错误。
- 登录态下若云端同步失败，本地仍会先保存 PIN，并返回“本地已生效、云端失败”的告警反馈。
- 解锁会话仍按 note 维度保存，修改全局 PIN 不会主动清除所有既有 note 会话。
- 当前已经有 `useNoteLock`、modal flow、view flow 和 setup/manage 交互的单测与集成测试，但仍缺更完整的跨设备和真实云端异常链路回归。
