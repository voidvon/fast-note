## 4. 里程碑与发布建议

- 里程碑划分：
  - M1：把列表图标判断从“静态 `is_locked`”升级为“`is_locked` + 本地解锁会话”。
  - M2：补列表状态刷新机制，覆盖解锁成功、手动重新锁定、会话自然过期三种变化源。
  - M3：补单测、集成测试与桌面/移动主路径冒烟。
- 上线策略：
  - 作为现有列表锁图标需求的增量小版本上线，不涉及 schema 或云端协议变更。
- 灰度/回滚：
  - 若出现状态错乱或列表频繁重渲染，可仅回退“解锁图标和会话刷新逻辑”，保留现有锁图标占位方案。

## 5. 跨模块依赖与风险

- 依赖关系：
  - 依赖 `useNoteLock` 已有的本地会话判定能力。
  - 依赖 `NoteList` 作为首页、文件夹页等列表入口的统一渲染组件。
  - 依赖详情页与锁管理弹层在解锁、重新锁定时正确写入或清理本地会话。
- 关键风险：
  - 当前代码中的 `cooldown` 表示“输错 PIN 后禁止重试”，而本需求描述中的“冷却时间”实际更接近“解锁有效期”，术语混用容易实现偏差。
  - 如果列表只在进入页面时计算一次状态，桌面分栏场景下详情页解锁后列表不会即时更新。
  - 若对每个列表项单独直连 Dexie，容易把展示组件做重，破坏组件边界。
- 需要业务决策的问题：
  - 文档与交互文案中，是否统一使用“解锁有效期”指代成功解锁后的时间窗口。
  - 解锁图标是否仅更换为 `lockOpen`，颜色继续复用现有锁图标 token。
