## 2. 模块目标与现状

- 业务目标：
  - 用户在列表中能识别“该备忘录启用了锁”之外，还能识别“当前设备上这篇备忘录是否仍处于临时已解锁状态”。
- 当前状态（已支持/部分支持/未支持）：部分支持。
- 关键证据：
  - `E1` `src/components/NoteListItem.vue:46`：当前 `showLockIcon` 只判断 `item_type === NOTE && is_locked === 1`。
  - `E2` `src/components/NoteListItem.vue:146`：列表项目前只会渲染 `lockClosed`，没有 `lockOpen` 或解锁态枚举。
  - `E3` `src/hooks/useNoteLock.ts:155`：解锁成功后会创建带 `expires_at` 的本地会话，现有默认有效期为 5 分钟。
  - `E4` `src/hooks/useNoteLock.ts:975`：`getLockViewState()` 已能产出 `unlocked / locked / cooldown` 视图快照。
  - `E5` `src/views/NoteDetail.vue:573`：详情页 PIN 解锁成功后，当前详情页会切到 `unlocked`。
  - `E6` `src/components/NoteLockManageModal.vue:130`：立即重新锁定会清掉当前 note 的本地会话。
  - `E7` `tests/unit/components/note-list-item-lock-indicator.spec.ts:73`：现有测试仅覆盖锁住态与空占位。
- 已知缺口：
  - 列表展示没有消费本地解锁会话，因此解锁后仍显示锁住图标。
  - 列表层没有自动感知会话过期或“立即重新锁定”的刷新机制。
  - 现有自动化测试未覆盖解锁图标、会话过期回锁和桌面分栏场景。

## 3. 逐功能点方案

### 3.1 F1.1 列表项会话感知图标状态

#### 3.1.1 目标与边界

- 用户场景：
  - 用户在详情页输入正确 PIN 或生物识别成功后回到列表，期望立刻看到该 note 变为解锁图标。
  - 用户在桌面分栏中直接在右侧详情区解锁，左侧列表也应同步切换图标。
- 范围内：
  - 对 note 行增加 `locked / unlocked / placeholder` 三态展示语义。
  - `is_locked=1` 且当前设备存在有效会话时显示解锁图标。
  - `is_locked=1` 且没有有效会话或处于输错冷却时显示锁住图标。
- 范围外：
  - 未锁定 note 增加新交互。
  - 锁图标点击行为调整。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` `src/components/NoteListItem.vue:46`：当前只以 `is_locked` 控制图标显示，不读取会话。
  - `E2` `src/components/NoteListItem.vue:142`：状态槽位当前只有 `locked / placeholder` 两种 `data-lock-state`。
  - `E3` `src/components/NoteList.vue:268`：真实 note 列表统一经过 `NoteListItem`，适合作为唯一渲染出口。
  - `E4` `src/hooks/useNoteLock.ts:995`：会话有效时 `getLockViewState()` 会返回 `viewState: 'unlocked'`。
  - `E5` `src/hooks/useNoteLock.ts:1006`：输错 PIN 冷却时会返回 `viewState: 'cooldown'`，该状态不应映射为解锁图标。
- 现状结论：
  - 锁定能力的业务语义已经完整存在于 `useNoteLock`，缺的是列表层把它消费为展示状态。
- 推断与待确认：
  - 推断：`NoteListItem` 应继续保持展示组件角色，不直接访问 Dexie。
  - 待确认：解锁图标优先使用 `lockOpen`，颜色先复用现有 `--c-icon`，仅通过形状表达“当前已解锁”。

#### 3.1.3 技术方案

- 方案摘要：
  - 在列表层引入“锁图标展示态”派生状态，统一由 `NoteList` 或专用 hook 计算后传给 `NoteListItem`，避免展示组件直接读会话。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/components/NoteListItem.vue`
      - 接收新的展示态输入，例如 `lockIndicatorState?: 'locked' | 'unlocked' | 'placeholder'`。
      - `locked` 渲染 `lockClosed`，`unlocked` 渲染 `lockOpen`，`placeholder` 继续保留空白占位。
      - 将 `data-lock-state` 扩展为三态，保留现有宽度、间距和主题 token。
    - `src/components/NoteList.vue`
      - 维护 `noteId -> lockIndicatorState` 的派生映射。
      - 对文件夹项和伪文件夹项继续强制走 `placeholder` 或不传状态。
  - 编排（`hooks`）：
    - 新增一个轻量 hook，例如 `useNoteLockIndicatorState`，负责批量读取可见 note 的 `getLockViewState()` 结果并输出展示态 map。
    - 展示态映射规则：
      - `note.is_locked !== 1` -> `placeholder`
      - `snapshot.viewState === 'unlocked'` -> `unlocked`
      - `snapshot.viewState === 'locked' | 'cooldown'` -> `locked`
  - 领域（`stores`）：
    - 无需修改 `Note` 契约；状态 map 保持为 UI 派生态，不写回 store。
  - 持久化（`database`）：
    - 无变更；继续读取现有 `note_unlock_sessions`。
  - 云端（`pocketbase`）：
    - 无变更。
  - 路由（`router`）：
    - 无变更。
- 数据契约变更：
  - 字段/类型：
    - 不修改 `Note`。
    - 允许 `NoteListItem` 新增仅 UI 使用的 prop。
  - 兼容与迁移：
    - 无迁移；未传新 prop 时可回退到现有 `is_locked` 逻辑，便于分阶段落地。
- 同步与冲突策略：
  - 纯本地展示态，不参与云端同步，也不影响 offline-first 主数据真相源。
- 错误处理与回滚策略：
  - 若会话读取失败，默认回退为 `locked`，避免把未确认状态误展示成解锁态。
  - 若展示态 map 计算异常，可只回退 `NoteListItem` 到现有 `is_locked` 逻辑。

#### 3.1.5 验收标准

- AC1：`is_locked=1` 且无有效会话的 note 显示锁住图标。
- AC2：`is_locked=1` 且存在有效会话的 note 显示解锁图标。
- AC3：`is_locked=1` 且处于输错 PIN 冷却中的 note 仍显示锁住图标。
- AC4：`is_locked=0` 或缺失字段的 note 仍不显示图标内容，但保留占位。
- AC5：锁住态、解锁态、未锁定态混排时，标题左边界保持一致。

#### 3.1.6 测试与验证

- 单元测试：
  - 为 `NoteListItem` 补解锁图标渲染断言，覆盖三态 `data-lock-state`。
  - 为列表锁图标展示态 hook 补映射测试，覆盖 `unlocked / locked / cooldown / placeholder`。
- 集成/冒烟：
  - 在首页混合列表中验证锁住态、解锁态、旧数据未锁定态并存。
  - 在深色模式下验证 `lockOpen` 与 `lockClosed` 都可读。
- 回归关注点：
  - 文件夹行、最近删除、全部备忘录等伪文件夹项不误显示解锁图标。
  - 长标题和多行摘要布局不回退。

#### 3.1.7 风险与未决问题

- 风险：
  - 若 `NoteListItem` 自己读取会话，会使纯展示组件承担数据访问职责，增加复用成本。
  - 若把 `cooldown` 错映射成 `unlocked`，会造成明显安全语义错误。
- 未决问题：
  - 是否允许在交互上通过颜色区分 `lockOpen`，还是仅用形状区分。

### 3.2 F1.2 解锁有效期到期与重新锁定联动刷新

#### 3.2.1 目标与边界

- 用户场景：
  - 用户在桌面端右侧详情区刚解锁某篇 note，左侧列表立刻变为解锁图标。
  - 用户停留在列表页不操作，等到会话过期后，图标自动恢复锁住态。
  - 用户点击“立即重新锁定”后，列表立刻恢复锁住图标。
- 范围内：
  - 定义列表图标的刷新触发源和最小刷新机制。
  - 保证桌面分栏、移动返回列表、列表停留超时三类场景都能生效。
- 范围外：
  - 改造详情页整体锁流程。
  - 新增复杂全局状态管理框架。

#### 3.2.2 现状证据

- 代码事实：
  - `E6` `src/views/NoteDetail.vue:562`：PIN 解锁成功后只更新详情页本地状态，不会直接通知列表。
  - `E7` `src/views/NoteDetail.vue:594`：生物识别解锁成功同样只更新详情页状态。
  - `E8` `src/components/NoteLockManageModal.vue:130`：手动重新锁定通过 `noteLock.relock(noteId)` 清除会话。
  - `E9` `src/hooks/useNoteLock.ts:867`：`relock()` 当前只删除会话，未提供 UI 侧事件通知。
  - `E10` `src/hooks/useNoteLock.ts:167`：会话结构中已有 `expires_at`，可用于推导下一次自动回锁时机。
- 现状结论：
  - 业务层已有“会话何时有效、何时失效”的完整依据，但 UI 没有统一刷新机制。
- 推断与待确认：
  - 推断：最稳妥的方案是“会话变更事件 + 最近到期定时器”双通道。
  - 待确认：是否接受列表页在后台时不持续计时，而在重新可见时统一重算一次。

#### 3.2.3 技术方案

- 方案摘要：
  - 用“会话变更主动通知 + 基于最早过期时间的单一计时器”驱动列表刷新，避免高频轮询。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/components/NoteList.vue`
      - 订阅锁会话变更事件，收到后只重算当前可见列表中的相关 note。
      - 记录当前可见解锁态 note 的最近 `expires_at`，设置单个 `setTimeout`，到点后重算 map。
  - 编排（`hooks`）：
    - `src/hooks/useNoteLock.ts`
      - 在 `verifyPin()` 成功写入会话、`tryBiometricUnlock()` 成功写入会话、`relock()` / `clearSession()` 清理会话后，统一发出会话变更事件。
    - `src/hooks/useNoteLockIndicatorState.ts`
      - 封装订阅、重算、销毁和最近过期时间调度逻辑。
  - 领域（`stores`）：
    - 无改动；列表刷新不依赖修改 note store。
  - 持久化（`database`）：
    - 无改动；继续由 Dexie 持久化会话。
  - 云端（`pocketbase`）：
    - 无改动。
  - 路由（`router`）：
    - 无改动；移动端返回列表可额外在页面 `ionViewWillEnter` 上做一次兜底重算，但不作为唯一刷新来源。
- 数据契约变更：
  - 字段/类型：
    - 建议新增仅前端内部使用的事件 payload 类型，例如 `{ noteId: string, reason: 'verified' | 'biometric' | 'relock' | 'session_cleared' }`。
  - 兼容与迁移：
    - 无迁移；事件机制为纯前端运行时能力。
- 同步与冲突策略：
  - 状态源仍以 Dexie 中的 `note_unlock_sessions` 为准，事件仅负责触发重算，不承载最终真相。
- 错误处理与回滚策略：
  - 事件丢失时，页面重新可见或重新进入列表时仍执行全量重算作为兜底。
  - 若定时器逻辑引入异常重渲染，可先回退为“页面重新可见时重算”，保留正确性优先。

- AC6：用户 PIN 解锁成功后，当前可见列表中的对应 note 图标立即变为解锁图标。
- AC7：用户生物识别解锁成功后，当前可见列表中的对应 note 图标立即变为解锁图标。

#### 3.2.6 测试与验证

- 单元测试：
  - 为 `useNoteLock` 增加会话变更事件派发测试。
  - 为计时器调度逻辑补过期回锁测试，验证不会持续轮询。
- 集成/冒烟：
  - 桌面分栏中解锁当前详情 note，观察左侧列表图标即时切换。

#### 3.2.7 风险与未决问题

- 风险：
  - 若采用固定间隔轮询，列表页会引入不必要的性能消耗。
  - 若仅依赖页面重新进入触发刷新，桌面分栏即时性不足。
- 未决问题：
  - 是否需要为“当前可见 note 数量很多”的情况增加批处理节流。

### 3.3 F1.3 自动化验证与回归覆盖

  - 组件单测、hook 单测、列表集成测试和人工冒烟清单。
