- 模块名称：编辑反馈与输入聚焦
- 输出日期：2026-03-08
- 输出文件：`doc/功能拆分与技术方案/Fast-Note-功能拆分与技术方案-编辑反馈与输入聚焦-2026-03-08.md`

## 2. 模块目标与现状

- 业务目标：让“新建文件夹”和“详情自动保存”两条高频编辑链路具备更稳定的输入体验和更轻量的状态反馈，降低键盘闪断、保存成功提示打扰和失败不可感知的问题。
- 当前状态（已支持/部分支持/未支持）：部分支持。
- 关键证据：
  - `E1` `src/views/HomePage.vue:403`：根目录新建文件夹弹窗已改为 `keyboardClose=false + didPresent focus`，说明首页入口已具备输入聚焦稳定化能力。
  - `E2` `src/views/FolderPage.vue:348`：文件夹页新建文件夹弹窗也已接入相同聚焦策略，说明二级入口已保持一致。
  - `E3` `src/components/LongPressMenu.vue:36`：重命名仍使用 `alertController.create()` 的默认输入配置，尚未统一 `didPresent` 聚焦策略。
  - `E4` `src/views/NoteDetail.vue:53`：详情页已新增 `state.isSaving`，表明保存状态被提升为页面编排态。
  - `E5` `src/views/NoteDetail.vue:154`：保存失败统一通过 `toastController` 以顶部 danger toast 展示。
  - `E6` `src/views/NoteDetail.vue:168`：`handleNoteSaving()` 在非静默保存中显式管理 loading 生命周期，并区分本地保存失败与同步失败。
  - `E7` `src/views/NoteDetail.vue:416`：详情页导航栏右侧已新增 `IonSpinner`，位于更多按钮左侧，替代原底部成功横条。
- 已知缺口：
  - 重命名弹窗尚未接入同一套输入聚焦兜底。
  - 本次体验改造尚未补自动化测试，回归仍以手工验证为主。
  - 保存失败已统一到顶部提示，但尚未形成全局通用提示封装。

## 3. 逐功能点方案

### 3.1 F2.5 新建文件夹弹窗输入聚焦稳定化

#### 3.1.1 目标与边界

- 用户场景：用户在首页或文件夹页点击“新建文件夹”后，弹窗出现时输入框应自动获得焦点，并尽量避免键盘先弹起后被弹窗焦点管理抢走。
- 范围内：`HomePage`、`FolderPage` 两个“新建文件夹”入口的 `IonAlert` 焦点治理。
- 范围外：`LongPressMenu` 中的重命名弹窗、`IonModal` 方案替换、原生键盘插件层面的系统兼容问题。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` `src/views/HomePage.vue:403`：首页创建文件夹使用模板式 `IonAlert`，并已通过 `@didPresent="focusFolderAlertInput"` 进行二次聚焦。
  - `E2` `src/views/FolderPage.vue:348`：文件夹页的 `IonAlert` 同样在展示完成后手动聚焦输入框。
  - `E3` `src/components/LongPressMenu.vue:36`：重命名逻辑仍使用 `alertController.create()` 动态创建 alert，未接入同样的键盘与焦点治理。
- 现状结论：根目录和文件夹内新建的主路径已实现较稳定的自动聚焦，但相似弹窗未完全统一，说明该方案目前属于“局部稳定化”而非“全局 overlay 输入聚焦规范”。
- 推断与待确认：
  - 推断：当前选择 `didPresent + focus()` 而非单纯 `autofocus`，是为了规避 Ionic `Alert` 在 `present()` 结束时把焦点切回 wrapper 的问题。
  - 待确认：是否需要把重命名、移动等所有文本输入型弹窗收敛到同一套 focus helper。

#### 3.1.3 技术方案

- 方案摘要：保留 `IonAlert`，通过 `keyboardClose=false` 禁止弹窗呈现时主动收键盘，并在 `didPresent` 后对 `.alert-input` 进行一次延迟 `focus()`，作为当前架构下的最小侵入方案。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/views/HomePage.vue`：为首页“新建文件夹”弹窗增加 `focusFolderAlertInput()` 与 `@didPresent`。
    - `src/views/FolderPage.vue`：为文件夹页“新建文件夹”弹窗增加同样的聚焦兜底。
  - 编排（`hooks`）：无新增 Hook，本期先将焦点逻辑留在页面层，避免过早抽象。
  - 领域（`stores`）：无改动，文件夹创建仍通过现有 `addNote()` 进入本地 store。
  - 持久化（`database`）：无改动。
  - 云端（`pocketbase`）：无改动。
  - 路由（`router`）：无改动。
- 数据契约变更：
  - 字段/类型：无新增业务字段。
  - 兼容与迁移：无数据库或类型迁移。
- 同步与冲突策略：不涉及同步链路，仍由创建成功后的现有保存/同步逻辑处理。
- 错误处理与回滚策略：
  - 若某些设备上 `IonAlert` 聚焦仍不稳定，优先回退到“用户手动点击输入框”这一默认可用路径。
  - 若后续需要更高稳定性，可升级为 `IonModal + IonInput.setFocus()` 方案，但本期不做结构替换。

#### 3.1.5 验收标准

- AC1：首页点击“新建文件夹”后，弹窗输入框自动获得焦点，用户可直接输入。
- AC2：文件夹页点击“新建文件夹”后，弹窗输入框自动获得焦点，重复打开仍稳定。
- AC3：主路径中不再出现“先聚焦一瞬间再失焦”的明显闪断体验。
- AC4：不改变现有 `addNote()` 创建文件夹的数据写入和同步行为。

#### 3.1.6 测试与验证

- 单元测试：

- 集成/冒烟：
  - 手工验证首页新建文件夹首次打开、关闭后二次打开、文件夹页打开三条路径。
  - 规划 `tests/integration/folder/folder-alert-focus.spec.ts` 覆盖主路径。
- 回归关注点：
  - iOS/Android 真机上的软键盘弹起时序。
  - `IonAlert` 关闭后二次打开是否仍能重新聚焦。

#### 3.1.7 风险与未决问题

- 风险：
  - R1：Ionic overlay 焦点策略升级后，`didPresent + focus()` 可能需要重新校准延迟时间。
  - R2：不同 WebView 对自动弹键盘限制不同，极端设备上仍可能退化为“只有聚焦，无自动弹键盘”。
- 未决问题：
  - Q1：是否要把 `LongPressMenu` 的重命名弹窗也纳入同一期统一治理。
  - Q2：是否接受后续升级为 `IonModal + IonInput` 以追求更强稳定性。

### 3.2 F2.6 详情保存状态顶部可视化与错误提示统一

#### 3.2.1 目标与边界

- 用户场景：用户编辑笔记并触发自动保存时，希望在导航栏看到轻量的“正在保存”反馈；保存完成后无需额外打扰；保存失败时应通过统一顶部错误提示获知异常。
- 范围内：`NoteDetail` 的非静默保存状态可视化、保存失败/同步失败错误提示统一、底部成功横条移除。
- 范围外：保存重试按钮、全局通知中心、静默保存可视化、同步进度多阶段拆分（上传/下载/冲突）。

#### 3.2.2 现状证据

- 代码事实：
  - `E4` `src/views/NoteDetail.vue:53`：页面状态已新增 `isSaving`，用于表示详情保存中的即时状态。
  - `E5` `src/views/NoteDetail.vue:154`：错误提示统一收敛到 `presentTopError()`，使用顶部 `danger` toast。
  - `E6` `src/views/NoteDetail.vue:168`：`handleNoteSaving()` 在真实保存前开启 loading，在 `finally` 中关闭，并区分本地保存异常与同步异常。
  - `E7` `src/views/NoteDetail.vue:416`：导航栏右侧更多按钮左边已新增 `IonSpinner` 作为保存中状态提示。
- 现状结论：详情页已从“底部 toast 成功/失败反馈”切换到“头部 saving spinner + 顶部错误提示”的组合，但该能力目前仅在 `NoteDetail` 页面局部生效，且默认不覆盖 `silent=true` 的静默保存分支。
- 推断与待确认：
  - 推断：`silent=true` 不显示 loading，是为了避免桌面端切换笔记时右上角频繁闪动。
  - 待确认：是否需要在未来为桌面端静默保存提供更轻量的无感知状态（如点状指示器）。

#### 3.2.3 技术方案

- 方案摘要：将保存状态定义为页面编排态，在 `handleNoteSaving()` 里统一驱动 `isSaving` 生命周期；成功态不再弹 toast；异常态统一通过顶部 `toastController` 展示错误。
- 架构落点（按层）：
  - UI（`views/components`）：
    - `src/views/NoteDetail.vue`：
      - 导航栏右侧增加 `IonSpinner`；
      - 删除底部成功 `IonToast`；
      - 新增顶部错误 toast 调用入口。
  - 编排（`hooks`）：
    - 继续复用 `useSync().sync(true)`；不额外新增保存状态 Hook。
  - 领域（`stores`）：
    - 继续通过 `useNote` 的 `addNote/updateNote/deleteNote` 执行本地保存；不新增 `Note` 状态字段。
  - 持久化（`database`）：无改动。
  - 云端（`pocketbase`）：无改动。
  - 路由（`router`）：无改动。
- 数据契约变更：
  - 字段/类型：无 `Note` 契约变更；新增的是 `NoteDetail` 页面本地状态 `isSaving:boolean`。
  - 兼容与迁移：无 Dexie / PocketBase / 类型迁移。
- 同步与冲突策略：
  - 继续沿用 `sync(true)` 的现有语义：未登录静默跳过；真正同步异常抛错并由页面层转为顶部错误提示。
  - 不改变 `updated` 驱动的本地优先与远端同步裁决规则。
- 错误处理与回滚策略：
  - 本地保存异常：顶部提示 `保存失败，请重试`。
  - 同步异常：顶部提示 `同步失败，请检查网络连接`。
  - 若新交互导致用户不适，可回退到原底部 toast 方案，但会恢复更高打扰度。

| T-FN-027 | 将详情页保存失败与同步失败统一收敛为顶部错误提示，并移除底部成功横条 | `src/views/NoteDetail.vue` | T-FN-026 | 低 |
| T-FN-028 | 补充保存状态与错误提示自动化覆盖，验证 loading 显示/隐藏和错误 toast 展示 | `tests/unit/views/note-detail-save-status.spec.ts`, `tests/integration/editor/note-detail-save-feedback.spec.ts`（规划） | T-FN-026,T-FN-027 | 中 |

#### 3.2.5 验收标准

- AC1：用户编辑笔记并触发非静默保存时，导航栏右侧更多按钮左侧显示旋转 loading 图标。
- AC2：保存和同步结束后 loading 图标自动消失，不再展示底部“保存完成/同步成功”横条。
- AC3：本地保存失败时显示顶部错误提示，文案为通用错误提示，不影响后续继续编辑。
- AC4：同步失败时显示顶部错误提示，仍保留本地已保存内容。
- AC5：桌面端切换笔记触发的 `silent=true` 静默保存不显示导航栏 loading，避免界面频繁闪动。

#### 3.2.6 测试与验证

- 单元测试：
  - 规划 `tests/unit/views/note-detail-save-status.spec.ts`，覆盖 `isSaving` 在成功/失败/静默保存中的状态切换。
- 集成/冒烟：
  - 规划 `tests/integration/editor/note-detail-save-feedback.spec.ts`，覆盖 blur 触发保存、loading 显示/隐藏、同步失败顶部提示。
  - 手工验证新建笔记、编辑已有笔记、断网同步失败、桌面端切换笔记四个场景。
- 回归关注点：
  - `handleNoteSaving()` 异常捕获后，是否仍能正确关闭 loading。
  - `silent=true` 的桌面切换保存是否仍保持无感。
  - 保存成功时不再出现底部 toast，避免与顶部错误提示混用。

#### 3.2.7 风险与未决问题

- 风险：
  - R3：若保存链路中存在未预期的长耗时异步分支，header spinner 可能停留过久，需要后续拆出更细粒度状态。
  - R4：目前错误提示仍由页面自行创建 toast，多个页面并发错误时可能出现风格不完全统一。
- 未决问题：
  - Q3：是否需要为“保存成功但同步跳过（未登录）”提供完全无提示还是轻量静默状态。
  - Q4：是否要把更多页面的错误提示都统一抽到全局 toast helper。
