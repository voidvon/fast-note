## 1. 文档信息

- 模块名称：同步逻辑
- 输出日期：2026-03-09
- 输出文件：`doc/功能拆分与技术方案/Fast-Note-功能拆分与技术方案-同步逻辑-2026-03-09.md`

- 不在范围：PocketBase 服务端表结构设计、服务端权限规则重写、协作编辑 OT/CRDT

## 2. 模块目标与现状

- 业务目标：保持 Fast-Note 的 offline-first 语义，在本地 Dexie 为即时真相源的前提下，实现登录后的云端增量同步与实时回写。
- 当前状态（已支持/部分支持/未支持）：部分支持
- 关键证据：
  - `E1` [src/processes/navigation/model/use-last-visited-route.ts:37]：最后访问路由恢复通过 `restoreLastVisitedRoute()` 执行。
  - `E2` [src/processes/session/model/use-session-bootstrap.ts:134]：登录会话初始化阶段会执行一次静默同步 `sync(true)`。
  - `E3` [src/processes/sync-notes/model/use-sync-notes.ts:26]：同步游标 key 通过 `createScopedStorageKey()` 按用户作用域生成。
  - `E4` [src/entities/note/model/state/note-store.ts:308]：本地增量基于 `getNotesByUpdated(updated)` 获取。
  - `E5` [src/shared/api/pocketbase/notes.ts:117]：云端增量基于 `updated > lastUpdated && user_id = auth.id` 查询。
  - `E6` [src/processes/sync-notes/model/use-sync-notes.ts:506]：本地已同步记录若本轮云端增量中不存在，仍会按 `update` 处理。
  - `E7` [src/shared/api/pocketbase/notes.ts:180]：云端 upsert 在 `auto` 模式下先尝试 `update`，404 时 fallback 到 `create`。
  - `E8` [src/shared/api/pocketbase/realtime-service.ts:40]：Realtime 订阅按 `user_id` 过滤。
- 已知缺口：
  - `代码事实`：当前同步不是“全量取全库”，而是“按游标做双向增量”。
  - `推断`：当本地数据缺失但游标已经前进时，旧云端记录不会出现在本轮增量结果中。
  - `代码事实`：当前保存链路已在“非新建态且本地未命中”时设置 `missingPrivateNote` 并阻止误创建。[src/features/note-save/model/use-note-save.ts:157]
  - `推断`：当前主要风险已从“误创建撞主键”转为“旧路由恢复后先看到缺失态，随后等待补齐式增量同步修复”。

## 3. 逐功能点方案

### 3.1 F3.1 同步触发入口与会话启动

#### 3.1.1 目标与边界

- 用户场景：用户打开应用、登录账号、编辑笔记或手动点击同步时，系统触发同步。
- 范围内：应用启动恢复、登录态会话初始化、保存后静默同步、手动同步入口。
- 范围外：首次安装引导、推送通知。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` [src/processes/session/model/use-session-bootstrap.ts:35]：应用启动时先恢复立即可访问的最后路由。
  - `E2` [src/processes/session/model/use-session-bootstrap.ts:134]：`bootstrapLoggedInSession()` 在登录后建立 Realtime 并执行 `sync(true)`。
  - `E3` [src/features/note-save/model/use-note-save.ts:199]：详情页保存成功后会触发静默同步。
  - `E4` [src/widgets/user-profile/ui/user-profile.vue:125]：用户面板提供显式手动同步入口。
- 现状结论：同步具备自动与手动双入口，启动顺序已调整为“先恢复立即路由，再完成登录同步，最后恢复需延后的私有路由”。
- 推断与待确认：
  - `推断`：若恢复到 `/n/:id` 且本地尚未存在该笔记，页面可能先进入缺数据状态。
  - `待确认`：是否允许未完成首轮同步前恢复私有详情路由。

#### 3.1.3 技术方案

- 方案摘要：将同步入口按“启动入口、保存入口、手动入口”统一梳理，保留自动同步能力，但应在首次会话恢复时增加“同步完成前的私有详情保护”。
- 架构落点（按层）：
  - UI（`pages/widgets`）：私有详情页在“路由存在但数据未到位”时展示加载态或缺失态，而不是直接可编辑。
  - 编排（`processes/features`）：为同步增加会话初始化状态，区分“路由恢复前/同步中/同步后”。
  - 领域（`entities`）：继续以 `useNote()` 提供本地查询与更新能力。
  - 持久化（`shared/lib/storage`）：不调整 Dexie 读写语义。
  - 云端（`shared/api/pocketbase`）：不调整接口，仅改变调用时机。
  - 路由（`processes/navigation` / `app/router`）：建议在首轮同步完成后再恢复私有详情路由，或对 `/n/:id` 增加数据守卫。
- 数据契约变更：
  - 字段/类型：无强制新增业务字段。
  - 兼容与迁移：无。
- 同步与冲突策略：保持本地优先、登录后增量补齐。
- 错误处理与回滚策略：首轮同步失败时保留本地可用态，但不应把“未拉到远端旧笔记”误判为新建。

#### 3.1.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 增加“会话初始化完成”状态 | `src/processes/session/model/use-session-bootstrap.ts`, `src/processes/sync-notes/model/use-sync-notes.ts` | 无 | 中 |
| T2 | 私有详情页缺数据时禁止进入新建保存链路 | `src/pages/note-detail/ui/note-detail-page.vue`, `src/features/note-save/model/use-note-save.ts` | T1 | 高 |
| T3 | 启动路由恢复时机调整或加守卫 | `src/processes/session/model/use-session-bootstrap.ts`, `src/processes/navigation/model/use-last-visited-route.ts` | T1 | 中 |

#### 3.1.5 验收标准

- AC1：登录后恢复到旧私有详情路由时，不会因为本地暂时缺数据而自动走新建链路。
- AC2：保存后静默同步仍然可用，未登录时继续静默跳过。
- AC3：手动同步入口仍可显示上传、下载、删除统计。

#### 3.1.6 测试与验证

- 单元测试：同步入口在不同登录态下的行为分支。
- 集成/冒烟：刷新应用 -> 恢复到 `/n/:id` -> 登录 -> 完成首轮同步 -> 页面显示正确笔记。
- 回归关注点：离线打开已有本地笔记时不能被首轮同步状态阻塞。

#### 3.1.7 风险与未决问题

- 风险：延后路由恢复可能影响桌面端“秒开上次页面”的体验。
- 未决问题：若远端笔记确实已被删除，详情页应显示“已删除”还是“无权限/不存在”。

### 3.2 F3.2 双向增量同步与游标推进

#### 3.2.1 目标与边界

- 用户场景：多设备编辑后，系统按增量拉取和推送记录，逐步达成最终一致。
- 范围内：本地增量、云端增量、操作队列构建、游标推进。
- 范围外：全量重建策略的产品交互。

#### 3.2.2 现状证据

- 代码事实：
  - `E1` [src/processes/sync-notes/model/use-sync-notes.ts:14]：默认游标是 `2010/01/01 00:00:00`。
  - `E2` [src/processes/sync-notes/model/use-sync-notes.ts:26]：真实游标通过按用户作用域的 key 读取。
  - `E3` [src/entities/note/model/state/note-store.ts:308]：本地增量只读取 `updated > cursor` 的记录。
  - `E4` [src/shared/api/pocketbase/notes.ts:117]：云端增量只读取 `updated > cursor` 的当前用户记录。
  - `E5` [src/processes/sync-notes/model/use-sync-notes.ts:702]：每成功一条记录即推进游标。
- 现状结论：当前同步实现是“按游标的双向增量同步”，不是“每次全量拉取所有 id”。
- 推断与待确认：
  - `推断`：只要游标已经前进，而某条远端记录的 `updated` 不大于游标，该记录就不会再次出现在增量结果里。
  - `待确认`：是否允许在本地数据被清空、但游标仍存在时继续按增量执行。

#### 3.2.3 技术方案

- 方案摘要：保留增量同步策略，但需要补齐“游标与本地数据一致性”的保护机制。
- 架构落点（按层）：
  - UI（`widgets`）：手动同步失败时提示“可尝试重建本地索引/重置同步状态”。
  - 编排（`processes/sync-notes`）：为游标增加“按用户隔离”与“本地空库保护”。
  - 领域（`entities/note`）：补充“本地是否为空 / 是否缺关键记录”的检测方法。
  - 持久化（`shared/lib/storage`）：Dexie 仍作为本地即时真相源，但应与游标联合校验。
  - 云端（`shared/api/pocketbase`）：继续使用 `updated + user_id` 过滤，不改变服务端契约。
  - 路由（`processes/navigation`）：无直接变更。
- 数据契约变更：
  - 字段/类型：建议新增内部同步元信息类型，如 `SyncBootstrapState`，不改业务 `Note` 字段。
  - 兼容与迁移：当前实现已使用按用户 key 的同步游标；若仍需兼容早期全局 key，可在首次登录时补一次迁移读取。
- 同步与冲突策略：仍按 `updated` 比较新旧；但在“本地空库 + 非初始游标”时，优先触发补拉或重置游标。
- 错误处理与回滚策略：不回滚已成功同步的条目；若发现“游标与本地断裂”，阻止继续推进游标并转为修复流程。

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 游标改为按用户隔离 | `src/processes/sync-notes/model/use-sync-notes.ts` | 无 | 中 |
| T2 | 增加“本地空库/缺关键记录”检测 | `src/processes/sync-notes/model/use-sync-notes.ts`, `src/entities/note/model/state/note-store.ts` | T1 | 中 |
| T3 | 提供重置游标或补拉入口 | `src/processes/sync-notes/model/use-sync-notes.ts`, `src/widgets/user-profile/ui/user-profile.vue` | T1 | 中 |

#### 3.2.5 验收标准

- AC1：切换账号后不会共用上一账号的同步游标。
- AC2：本地清空但游标存在时，不会误以为“云端没有旧数据”。
- AC3：同步游标只在本地与云端状态一致时继续推进。

#### 3.2.6 测试与验证

- 单元测试：游标初始化、游标推进、按用户隔离 key 的读写。
- 集成/冒烟：A 账号同步后退出 -> B 账号登录 -> 游标独立；清空本地后再次同步能补回历史数据。
- 回归关注点：首次登录仍能通过旧默认时间完成历史拉取。

#### 3.2.7 风险与未决问题

- 风险：简单重置游标可能带来较大的全量下载量。
- 未决问题：本地缺失检测应以“空库”为准，还是按“路由正在访问的关键 note id 未命中”为准。

### 3.3 F3.3 同步操作队列与冲突裁决

#### 3.3.1 目标与边界

- 用户场景：本地和云端都有变更时，系统按统一规则决定上传、下载、删除或本地清理。
- 范围内：操作队列构建、`updated` 冲突裁决、30 天删除窗口。
- 范围外：多人协作冲突合并。

#### 3.3.2 现状证据

- 代码事实：
  - `E1` [src/processes/sync-notes/model/use-sync-notes.ts:447]：同步前会构建 `localNotesMap` 与 `cloudNotesMap`。
  - `E2` [src/processes/sync-notes/model/use-sync-notes.ts:462]：本地记录优先参与一轮操作判断。
  - `E3` [src/processes/sync-notes/model/use-sync-notes.ts:526]：云端记录再补一轮下载判断。
  - `E4` [src/processes/sync-notes/model/use-sync-notes.ts:575]：操作按 `updated` 升序串行执行。
  - `E5` [src/processes/sync-notes/model/use-sync-notes.ts:470]：删除 30 天后触发 `deleteLocal` 或服务端删除状态同步。
- 现状结论：当前冲突策略是基于时间戳的新者覆盖旧者，删除记录保留 30 天。
- 推断与待确认：
  - `推断`：串行执行有利于游标顺序推进，但单条失败会中断后续操作。
  - `待确认`：是否允许“部分成功 + 失败聚合上报”。

#### 3.3.3 技术方案

- 方案摘要：保持当前时间戳裁决模型，但需要在结果聚合与失败恢复上增强可观测性。
- 架构落点（按层）：
  - UI（`widgets`）：同步结果展示增加失败条数与建议动作。
  - 编排（`processes/sync-notes`）：操作结果改为 `uploaded/downloaded/deleted/failed` 聚合。
  - 领域（`entities/note`）：保持 `addNote/updateNote/deleteNote` 不变。
  - 持久化（`shared/lib/storage`）：无需 schema 变更。
  - 云端（`shared/api/pocketbase`）：保持现有查询/写入契约。
  - 路由（`processes/navigation`）：无。
- 数据契约变更：
  - 字段/类型：建议补充内部 `SyncOperationResult` 类型。
  - 兼容与迁移：无。
- 同步与冲突策略：继续采用 `updated` 单因子裁决，删除窗口继续保持 30 天。
- 错误处理与回滚策略：单条失败时保留已成功结果，失败记录进入重试或提示人工修复。

| ------ | -------- | -------- | ---- | ---- |
| T1 | 同步结果增加失败统计 | `src/processes/sync-notes/model/use-sync-notes.ts` | 无 | 中 |

#### 3.3.5 验收标准

- AC1：同步返回值可区分上传、下载、删除和失败数量。
- AC2：删除 29/30/31 天边界行为一致且可测试。
- AC3：单条失败不会导致已成功条目被错误回滚。

#### 3.3.6 测试与验证

- 单元测试：操作排序、冲突裁决、删除窗口判断。
- 集成/冒烟：双端同时编辑同一笔记后，更新较新的内容覆盖较旧内容。
- 回归关注点：软删除在 30 天窗口内仍需参与同步。

#### 3.3.7 风险与未决问题

- 风险：完全依赖客户端时间可能受设备时钟漂移影响。
- 未决问题：是否需要服务端时间或版本号作为第二裁决因子。

### 3.4 F3.4 云端 upsert 与附件同步

#### 3.4.1 目标与边界

- 用户场景：笔记内容和附件在本地编辑后，能够可靠地同步到 PocketBase。
- 范围内：云端查找记录、create/update 分支、附件上传与富文本回填。
- 范围外：文件压缩、图片转码、对象存储迁移。

#### 3.4.2 现状证据

- 代码事实：
  - `E1` [src/shared/api/pocketbase/notes.ts:156]：`create` 模式下若命中“已存在”错误，会 fallback 到 `update`。
  - `E2` [src/shared/api/pocketbase/notes.ts:168]：`update` 模式下若命中 404，会 fallback 到 `create`。
  - `E3` [src/shared/api/pocketbase/notes.ts:180]：`auto` 模式默认先尝试 `update`，404 时再 `create`。
  - `E4` [src/processes/sync-notes/model/use-sync-notes.ts:164]：同步前会把富文本中的本地 hash 替换成临时文件标识符。
  - `E5` [src/processes/sync-notes/model/use-sync-notes.ts:605]：上传成功后再将临时标识符回填为 PocketBase 文件名。
- 现状结论：当前云端写入已改为基于错误分支的客户端 upsert；附件同步仍采用“两阶段替换”策略。
- 推断与待确认：
  - `推断`：如果现有记录查询 miss，但服务端已有相同 `id` 的记录，`create` 会触发主键冲突。
  - `待确认`：服务端是否允许客户端自定义 `id` 的同时仍按 `user_id` 查询隔离所有历史数据。

#### 3.4.3 技术方案

- 方案摘要：保留客户端 upsert，但需对“查不到不等于不存在”做保护，尤其是在已知本地缺失或跨账号历史数据场景下。
- 架构落点（按层）：
  - UI（`pages/features`）：保存失败时给出“同步失败”与“本地已保存”区分提示。
  - 编排（`processes/sync-notes`）：在 `upload` 前增加存在性保护或降级策略。
  - 领域（`entities/note`）：不改本地 `id` 生成与存储契约。
  - 持久化（`shared/lib/storage`）：继续保存本地 hash、正文与 files 元数据。
  - 云端（`shared/api/pocketbase`）：保留 `create/update/auto` 三种写入模式，并评估是否需要更强的存在性确认。
  - 路由（`processes/navigation`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：附件仍采用“内容替换后再次回写”的两阶段策略。
- 错误处理与回滚策略：若附件上传成功但正文回填失败，应保留本地记录并提供后续重试能力。

#### 3.4.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 强化远端存在性判断，避免误 create | `src/shared/api/pocketbase/notes.ts` | 无 | 高 |
| T2 | 区分“本地保存成功”和“云端同步失败”提示 | `src/pages/note-detail/ui/note-detail-page.vue`, `src/features/note-save/model/use-note-save.ts` | T1 | 中 |
| T3 | 补附件两阶段同步回归测试 | `src/processes/sync-notes/model/use-sync-notes.ts`, `tests/**` | T1 | 中 |

#### 3.4.5 验收标准

- AC1：远端已有同 id 记录时，不会因为查询 miss 而重复 create。
- AC2：附件上传后富文本 URL 能正确从本地 hash 回填为 PocketBase 文件名。
- AC3：本地保存成功但云端失败时，用户收到明确区分提示。

#### 3.4.6 测试与验证

- 单元测试：无附件、有附件、附件回填失败三种分支。
- 集成/冒烟：新建笔记插入附件 -> 保存 -> 同步 -> 刷新后内容和附件可读取。
- 回归关注点：重复引用同一文件时不得重复上传。

#### 3.4.7 风险与未决问题

- 风险：附件同步是两阶段写入，过程失败可能产生“正文与 files 元数据暂时不一致”。
- 未决问题：是否需要为附件同步增加事务态或补偿队列。

### 3.5 F3.5 Realtime 回写与本地索引一致性

#### 3.5.1 目标与边界

- 用户场景：另一端完成创建、更新或删除后，当前设备能通过 Realtime 快速收到变化并写回本地。
- 范围内：建立连接、订阅过滤、消息处理、重连。
- 范围外：前后台连接策略细化。

#### 3.5.2 现状证据

- 代码事实：
  - `E1` [src/shared/api/pocketbase/realtime-service.ts:31]：Realtime 连接前校验登录态。
  - `E2` [src/shared/api/pocketbase/realtime-service.ts:40]：订阅 `notes` 集合并按 `user_id` 过滤。
  - `E3` [src/shared/api/pocketbase/realtime-service.ts:123]：create/update/delete 分别映射为本地 `addNote/updateNote/deleteNote`。
  - `E4` [src/shared/api/pocketbase/realtime-service.ts:165]：断线后采用指数退避重连。
  - `E5` [src/entities/note/model/state/note-store.ts:214]：`addNote` 会更新本地响应式数组与索引。
- 现状结论：Realtime 是增量同步之外的“在线补丁流”，用于降低跨端最终一致的延迟。
- 推断与待确认：
  - `推断`：Realtime 能补充“同步之后的新变化”，但不能替代首轮历史数据对齐。
  - `待确认`：是否需要对重复推送或乱序消息增加显式去重日志。

#### 3.5.3 技术方案

- 方案摘要：保持 Realtime 作为在线增量回写通道，但明确它不是历史全量恢复手段。
- 架构落点（按层）：
  - UI（`widgets`）：用户面板可展示 Realtime 连接状态。
  - 编排（`processes/session` / `processes/sync-notes`）：同步完成后再标记 Realtime ready，避免误解为“已全量对齐”。
  - 领域（`entities/note`）：继续依赖 `add/update/delete` 维护索引。
  - 持久化（`shared/lib/storage`）：借助 store 到 Dexie 的同步保持本地持久化一致。
  - 云端（`shared/api/pocketbase`）：保留订阅过滤与重连逻辑。
  - 路由（`processes/navigation`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：Realtime update 仍按 `remoteTime > localTime` 决定是否覆盖。
- 错误处理与回滚策略：连接失败时可降级为手动同步，不阻塞本地编辑。

| T3 | 补重连和乱序消息回归测试 | `tests/**` | T2 | 中 |

#### 3.5.5 验收标准

- AC1：登录后能建立 Realtime 连接并接收 create/update/delete。
- AC2：Realtime 失败时仍可手动同步，不影响本地编辑。
- AC3：用户不会将“Realtime 已连接”误认为“历史数据已全量对齐”。

#### 3.5.6 测试与验证

- 单元测试：消息分发与时间戳覆盖判断。
- 集成/冒烟：设备 A 修改 -> 设备 B 在线收到更新并写回本地。
- 回归关注点：登出时必须断开订阅，避免串账号数据。

#### 3.5.7 风险与未决问题

- 风险：重复连接可能导致重复订阅与重复写回。
- 未决问题：是否需要后台休眠后重新进入前台时执行一次补同步。
