## 3. 逐功能点方案

### 3.1 F2.1 文件夹树与计数一致性

#### 3.1.1 目标与边界

- 用户场景：用户在多层文件夹下组织笔记并查看准确计数。
- 范围内：文件夹树构建、父级计数更新、移动后计数修正。
- 范围外：公开用户文件夹树（归属“认证与公开访问”模块）。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` [src/stores/notes.ts:305]：过滤文件夹并构建树。
  - `E2` [src/stores/notes.ts:397]：`getNoteCountByParentId` + while 递归更新父级。
  - `E3` [src/components/NoteMove.vue:63]：移动后分别更新旧/新父级 `note_count`。
- 现状结论：常规创建和移动流程下计数可更新。
- 推断与待确认：
  - `推断`：跨多层移动和批量删除并发时可能出现短时计数抖动。

#### 3.1.3 技术方案

- 方案摘要：保留现有索引结构（`notesMap` + `parentIdMap`），补充批量操作后的一次性计数校准。
- 架构落点（按层）：
  - UI（`views/components`）：无结构调整，仅增加失败提示。
  - 编排（`hooks`）：无。
  - 领域（`stores`）：新增 `recalculateFolderCounts(rootId?)`，在高风险操作后触发。
  - 持久化（`database`）：不改 schema。
  - 云端（`pocketbase`）：无。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：无新增字段。
  - 兼容与迁移：无。
- 同步与冲突策略：计数字段由本地重算后再参与同步，避免脏计数上云。
- 错误处理与回滚策略：计数失败时记录日志并回退到全量重算。

#### 3.1.4 实施任务拆分

| 任务ID | 任务描述                           | 变更文件                                             | 依赖         | 风险 |
| ------ | ---------------------------------- | ---------------------------------------------------- | ------------ | ---- |
| T1     | 新增计数全量校准方法并在移动后调用 | `src/stores/notes.ts`, `src/components/NoteMove.vue` | 无           | 中   |
| T2     | 为计数重算失败补充可观测日志       | `src/stores/notes.ts`                                | logger       | 低   |
| T3     | 补充多层文件夹移动回归用例         | `tests/**`                                           | 测试数据构造 | 中   |

#### 3.1.5 验收标准

- AC1：多层文件夹下移动笔记后，旧父级与新父级计数均正确。
- AC2：重启应用后计数与列表展示一致。
- AC3：计数异常时可触发一次性重算恢复。

#### 3.1.6 测试与验证

- 单元测试：`getNoteCountByParentId`、`updateParentFolderSubcount`。
- 集成/冒烟：连续创建文件夹 -> 新建笔记 -> 跨层移动。
- 回归关注点：`allnotes`、`unfilednotes` 统计不受影响。

#### 3.1.7 风险与未决问题

- 风险：大树重算可能影响低端设备响应。
- 未决问题：是否将 `note_count` 改为只读派生字段（渲染时计算）。

### 3.2 F2.2 编辑自动保存与新建笔记流程

#### 3.2.1 目标与边界

- 用户场景：用户编辑笔记时无需手动保存，切换笔记不丢内容。
- 范围内：新建 ID 分配、标题/摘要提取、防抖保存、空内容删除。
- 范围外：附件上传与回填（归属“附件与安全”模块）。

#### 3.2.2 现状证据

- 代码事实：
  - `E1` [src/views/NoteDetail.vue:75]：新建笔记自动生成 `nanoid(12)`。
  - `E2` [src/views/NoteDetail.vue:173]：内容未变化直接跳过保存。
  - `E3` [src/views/NoteDetail.vue:281]：保存防抖 800ms。
  - `E4` [src/views/NoteDetail.vue:274]：空内容删除笔记。
- 现状结论：编辑主链路完整，桌面端笔记切换前执行静默保存。
- 推断与待确认：
  - `待确认`：长文档编辑时 800ms 固定防抖是否需要自适应。

#### 3.2.3 技术方案

- 方案摘要：保持防抖保存机制，增加“保存状态”可视反馈与失败重试按钮。
- 架构落点（按层）：
  - UI（`views/components`）：标题栏增加“保存中/已保存/失败”。
  - 编排（`hooks`）：封装 `saveWithRetry(maxRetries)`。
  - 领域（`stores`）：维持 `addNote/updateNote/deleteNote` 接口。
  - 持久化（`database`）：沿用 `useRefDBSync` 防抖落库。
  - 云端（`pocketbase`）：仍走 `sync(true)` 静默策略。
  - 路由（`router`）：新建移动端继续 `router.replace('/n/:id')`。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：本地保存成功后再触发静默同步，失败不阻断本地编辑。
- 错误处理与回滚策略：同步失败仅 toast，不回滚本地内容。

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述                             | 变更文件                   | 依赖         | 风险 |
| ------ | ------------------------------------ | -------------------------- | ------------ | ---- |
| T1     | 增加保存状态 UI 与失败重试操作       | `src/views/NoteDetail.vue` | 无           | 低   |
| T2     | 抽离保存重试逻辑，统一 blur/离页保存 | `src/views/NoteDetail.vue` | 无           | 中   |
| T3     | 增加桌面切换笔记保存回归用例         | `tests/**`                 | 桌面模式测试 | 中   |

#### 3.2.5 验收标准

- AC1：编辑失焦后 800ms 内完成本地保存。
- AC2：切换笔记前当前内容可被静默保存。
- AC3：空内容笔记自动删除，不残留脏记录。

#### 3.2.6 测试与验证

- 单元测试：内容变更判断、防抖函数行为。
- 集成/冒烟：新建 -> 编辑 -> 切换 -> 返回 -> 内容存在。
- 回归关注点：公开上下文必须保持只读。

#### 3.2.7 风险与未决问题

- 风险：定时器保存与手动离页保存可能发生竞态。
- 未决问题：是否启用“强制保存”按钮用于异常场景。

### 3.3 F2.3 删除、回收站、恢复与永久删除

#### 3.3.1 目标与边界

- 用户场景：用户可安全删除并在 30 天窗口恢复，必要时永久删除。
- 范围内：软删除、恢复、永久删除标记、回收站展示。
- 范围外：超过 30 天的云端清理实现细节（归属同步模块）。

#### 3.3.2 现状证据

- 代码事实：
  - `E1` [src/components/LongPressMenu.vue:68]：删除时置 `is_deleted=1`。
  - `E2` [src/components/LongPressMenu.vue:99]：永久删除将 `updated` 设为 epoch。
  - `E3` [src/stores/notes.ts:264]：最近删除按 30 天过滤。
  - `E4` [src/views/DeletedPage.vue:80]：回收站提供 restore / deleteNow 菜单。
- 现状结论：软删除和恢复链路可用，永久删除通过时间戳驱动后续清理。
- 推断与待确认：
  - `代码事实`：文件夹删除仅处理一层 children，深层后代未显式递归（`src/components/LongPressMenu.vue:70`）。

#### 3.3.3 技术方案

- 方案摘要：补齐“文件夹深层递归软删”并统一恢复策略。
- 架构落点（按层）：
  - UI（`views/components`）：删除弹窗补充“影响范围数量”。
  - 编排（`hooks`）：无。
  - 领域（`stores`）：新增 `markSubtreeDeleted(rootId)` 与 `restoreSubtree(rootId)`。
  - 持久化（`database`）：不改 schema。
  - 云端（`pocketbase`）：依旧由同步模块识别 `is_deleted/updated`。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：删除状态优先级由 `updated` 决定，和现有同步规则兼容。
- 错误处理与回滚策略：子树删除失败时中止并保留已处理记录日志。

#### 3.3.4 实施任务拆分

| 任务ID | 任务描述                         | 变更文件                                                  | 依赖     | 风险 |
| ------ | -------------------------------- | --------------------------------------------------------- | -------- | ---- |
| T1     | 实现文件夹深层递归软删           | `src/stores/notes.ts`, `src/components/LongPressMenu.vue` | 无       | 中   |
| T2     | 恢复流程支持子树恢复（可选策略） | `src/stores/notes.ts`, `src/components/LongPressMenu.vue` | 产品确认 | 中   |
| T3     | 删除影响范围提示（节点计数）     | `src/components/LongPressMenu.vue`                        | 无       | 低   |

#### 3.3.5 验收标准

- AC1：删除多层文件夹后，所有后代均进入回收站。
- AC2：回收站仅展示 30 天内删除项。
- AC3：永久删除记录可被同步模块清理。

#### 3.3.6 测试与验证

- 单元测试：子树删除/恢复递归函数。
- 集成/冒烟：创建三层目录并删除中间层，验证后代状态。
- 回归关注点：`note_count` 随删除/恢复正确更新。

#### 3.3.7 风险与未决问题

- 风险：恢复子树可能与父节点已删除状态冲突。
- 未决问题：恢复是否必须恢复到原父级，还是允许回到根目录。
