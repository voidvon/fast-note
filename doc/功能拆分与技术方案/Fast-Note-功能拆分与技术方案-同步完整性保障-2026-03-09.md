  - `E5` [src/hooks/useSync.ts:281]：本地增量读取只依赖 `updated > cursor`。
  - `E6` [src/pocketbase/notes.ts:16]：云端增量读取只依赖 `updated > cursor && user_id = auth.id`。
  - `E7` [src/views/NoteDetail.vue:217]：详情页保存按本地 `getNote(id)` 是否命中决定更新或创建。
  - `E8` [src/stores/notes.ts:162]：本地 Dexie 落盘通过 `useRefDBSync` 以 `300ms` 防抖写入。
- 已知缺口：
  - `代码事实`：同步上下文（游标、数据库、最后访问路由）没有按用户隔离。
  - `代码事实`：私有详情路由在首轮同步前可能已恢复到页面层。
  - `代码事实`：非新建路由在本地 miss 时仍可能走创建分支。
  - `推断`：本地数据集不完整时，当前增量同步模型无法自行证明“全量完整”，需要健康检查与补齐机制协助。

## 3. 逐功能点方案

### 3.1 F3.4 同步上下文按用户隔离

#### 3.1.1 目标与边界

- 用户场景：用户切换账号、重新登录或多账号轮换使用时，本地缓存、同步游标和最后访问路由互不污染。
- 范围内：Dexie 库名、同步游标、最后访问路由 key 的按用户隔离。
- 范围外：服务端多租户模型、公开用户缓存库隔离重构。

#### 3.1.2 现状证据

- 代码事实：
  - `E1` [src/hooks/useSync.ts:13]：同步游标当前是全局单 key。
  - `E2` [src/database/dexie.ts:25]：Dexie 库名当前固定为 `note`。
  - `E3` [src/hooks/useLastVisitedRoute.ts:4]：最后访问路由 key 当前固定为 `flashnote_last_visited_route`。
- 现状结论：当前“当前用户的数据上下文”没有被持久化层和同步层完整表达。
- 推断与待确认：
  - `推断`：切账号后即使 UI 登录态变化正确，本地缓存视角仍可能携带上一账号残留上下文。
  - `待确认`：guest 模式是否继续保留独立本地库 `note:guest`。

#### 3.1.3 技术方案

- 方案摘要：以 `userId` 作为同步上下文根键，对本地数据库、同步游标和最后访问路由全部做 key 隔离。
- 架构落点（按层）：
  - UI（`views/components`）：无直接结构改动，仅消费隔离后的上下文恢复结果。
  - 编排（`hooks`）：`useSync`、`useLastVisitedRoute` 增加按用户 key 的读写 helper。
  - 领域（`stores`）：保持现有 `useNote` API，不引入账号字段到 `Note` 内存索引。
  - 持久化（`database`）：Dexie 库名改为 `note:<userId>`，未登录使用 `note:guest`。
  - 云端（`pocketbase`）：继续按 `user_id` 过滤，不改服务端接口。
  - 路由（`router`）：恢复逻辑按当前用户读取路由快照。
- 数据契约变更：
  - 字段/类型：无 `Note` 字段新增；新增内部 helper：`getSyncCursorKey(userId)`、`getLastRouteKey(userId)`、`getDatabaseName(userId)`。
  - 兼容与迁移：当前项目按开发版处理，直接切换到按用户隔离方案，不做旧 key/旧库兼容读取与迁移。
- 同步与冲突策略：保持 `updated` 单因子裁决，但同步状态必须在“当前用户上下文”中读取与推进。
- 错误处理与回滚策略：若用户上下文缺失，回退到 guest 上下文；不保留旧结构兼容分支，避免开发期双轨逻辑复杂化。

#### 3.1.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 为 Dexie 库名增加按用户隔离能力 | `src/database/dexie.ts` | 无 | 中 |
| T2 | 为同步游标增加按用户隔离 key | `src/hooks/useSync.ts` | T1 | 中 |
| T3 | 为最后访问路由增加按用户隔离 key | `src/hooks/useLastVisitedRoute.ts`, `src/App.vue` | T2 | 低 |

#### 3.1.5 验收标准

- AC1：切换账号后不会共用上一账号的同步游标。
- AC2：切换账号后不会恢复到上一账号的私有详情路由。
- AC3：未登录模式与登录模式可分别保留自己的本地缓存。

#### 3.1.6 测试与验证

- 单元测试：用户 key 生成、guest/user 切换读写。
- 集成/冒烟：A 账号登录并同步 -> 退出 -> B 账号登录 -> 验证游标、路由、本地数据相互独立。
- 回归关注点：开发环境切换账号后，guest 与登录用户上下文保持隔离。

#### 3.1.7 风险与未决问题

- 风险：多库切换会增加初始化与上下文切换复杂度。
- 未决问题：是否需要提供“切账号后清理旧账号本地缓存”的显式入口。

### 3.2 F3.5 私有路由恢复时序治理

#### 3.2.1 目标与边界

- 用户场景：用户刷新页面或重新打开应用时，私有详情路由的恢复必须建立在“当前用户同步上下文已就绪”的前提下。
- 范围内：私有 `/n/:id`、`/f/*` 路由恢复时机治理。
- 范围外：公开用户路由恢复时机调整。

#### 3.2.2 现状证据

- 代码事实：
  - `E1` [src/App.vue:24]：当前会在 `onMounted` 前立即恢复最后访问路由。
  - `E2` [src/App.vue:54]：当前静默同步发生在登录会话初始化阶段。
  - `E3` [src/hooks/useLastVisitedRoute.ts:38]：`restoreLastVisitedRoute()` 当前不区分公开/私有路由。
- 现状结论：当前路由恢复早于会话同步，私有详情页可能在缓存未补齐前先进入页面。
- 推断与待确认：
  - `推断`：公开路由可立即恢复，私有路由应延后到同步上下文 ready 后。
  - `待确认`：桌面端 `/home` 是否继续允许立即恢复（建议允许）。

#### 3.2.3 技术方案

- 方案摘要：把最后访问路由分为“可立即恢复”和“需会话准备后恢复”两类；私有详情路由只在同步上下文 ready 后恢复。
- 架构落点（按层）：
  - UI（`views/components`）：详情页可展示短暂加载态，避免闪跳。
  - 编排（`hooks`）：`useLastVisitedRoute` 增加 `isPrivateRestorableRoute()` 判断。
  - 领域（`stores`）：无。
  - 持久化（`database`）：无。
  - 云端（`pocketbase`）：无。
  - 路由（`router`）：必要时补私有详情守卫，禁止上下文未 ready 时直接进入编辑态。
- 数据契约变更：
  - 字段/类型：可新增内部枚举 `RouteRestoreMode = immediate | deferred`。
  - 兼容与迁移：无。
- 同步与冲突策略：不改变同步裁决，只改变进入页面的时机。
- 错误处理与回滚策略：若同步初始化失败，回退到 `/home` 或保留当前安全路由，不强制恢复私有详情。

#### 3.2.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 路由恢复分离“立即恢复/延后恢复”策略 | `src/hooks/useLastVisitedRoute.ts`, `src/App.vue` | 无 | 中 |
| T2 | 私有详情上下文未 ready 时回退到安全态 | `src/App.vue`, `src/router/index.ts` | T1 | 中 |

#### 3.2.5 验收标准

- AC1：刷新后若最后访问的是私有 `/n/:id`，应用会在会话同步完成后再恢复。
- AC2：刷新后若最后访问的是公开路由或 `/home`，仍可立即恢复。
- AC3：同步初始化失败时，不会直接进入一个数据未就绪的私有详情可编辑态。

#### 3.2.6 测试与验证

- 单元测试：路由分类 helper。
- 集成/冒烟：刷新到 `/n/:id`、`/home`、公开路由三类场景，验证恢复顺序。
- 回归关注点：桌面端刷新恢复当前上下文的既有体验不能被破坏。

#### 3.2.7 风险与未决问题

- 风险：延后恢复可能引入一次轻微加载感知。
- 未决问题：是否要在 UI 上显式展示“正在恢复私有会话”。

### 3.3 F3.6 缺失 note 防误创建

#### 3.3.1 目标与边界

- 用户场景：用户进入私有详情页后编辑内容，系统只能在“明确新建态”下创建笔记，不能因为本地 miss 而把旧 note 当新建。
- 范围内：`NoteDetail` 新建/更新判定逻辑。
- 范围外：公开详情页只读逻辑。

#### 3.3.2 现状证据

- 代码事实：
  - `E1` [src/views/NoteDetail.vue:217]：当前先通过 `getNote(id)` 判断本地是否存在。
  - `E2` [src/views/NoteDetail.vue:236]：本地未命中时直接走“新增笔记”分支。
  - `E3` [src/views/NoteDetail.vue:195]：当前只有 `id === '0'` 会被视为显式新建路由。
- 现状结论：当前“非新建路由但本地 miss”的情况没有被单独保护。
- 推断与待确认：
  - `推断`：在同步上下文未补齐时，这条分支会把缺失旧 note 误创建为新 note。
  - `待确认`：对 miss 状态采用“等待同步/未找到/回到列表”哪种产品文案最合适。

#### 3.3.3 技术方案

- 方案摘要：把“能否创建”从“本地存在性”改为“是否显式新建态”判断；对非新建路由 miss 增加只读保护与缺失提示。
- 架构落点（按层）：
  - UI（`views/components`）：详情页增加“正在同步/未找到笔记”态。
  - 编排（`hooks`）：必要时暴露 `isSessionBootstrapReady` 给详情页判断。
  - 领域（`stores`）：继续通过 `getNote` 获取本地数据，不增加写接口。
  - 持久化（`database`）：无。
  - 云端（`pocketbase`）：无。
  - 路由（`router`）：可选守卫，兜底阻止未准备好的私有详情进入编辑态。
- 数据契约变更：
  - 字段/类型：无 `Note` 契约变更；新增页面状态 `missingReason = waitingSync | notFound`。
  - 兼容与迁移：无。
- 同步与冲突策略：保持本地优先，不改变创建的 `id` 语义。
- 错误处理与回滚策略：若同步后仍 miss，则阻止保存并引导用户返回列表或重试同步。

#### 3.3.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 将创建判定改为仅允许显式新建态触发 | `src/views/NoteDetail.vue` | 无 | 高 |
| T2 | 为非新建路由 miss 增加缺失提示与禁编态 | `src/views/NoteDetail.vue` | T1 | 中 |

#### 3.3.5 验收标准

- AC1：`/n/:id` 且 `id !== '0'` 时，本地 miss 不会触发创建分支。
- AC2：只有 `id === '0'` 或明确新建态时才允许创建笔记。
- AC3：同步完成后仍 miss 的笔记会显示明确的未找到提示，而不是创建同 id 新记录。

#### 3.3.6 测试与验证

- 单元测试：`NoteDetail` 在 `new`、`existing hit`、`existing miss` 三种分支下的保存行为。
- 集成/冒烟：恢复到旧 `/n/:id` -> 同步前编辑 -> 不误创建；新建态正常创建。
- 回归关注点：移动端新建笔记路径不应被误伤。

#### 3.3.7 风险与未决问题

- 风险：如果产品文案不清晰，用户可能误以为笔记被删除。
- 未决问题：是否在 miss 状态下提供“从云端重新拉取此笔记”单独入口。

### 3.4 F3.7 本地缓存健康检查与补齐

#### 3.4.1 目标与边界

- 用户场景：即使本地缓存为空或部分缺失，系统也能发现“当前缓存不可信”，并触发补齐流程，而不是直接沿用旧游标继续增量同步。
- 范围内：同步前健康检查、游标重置/补齐式增量。
- 范围外：服务端全量校验 API。

#### 3.4.2 现状证据

- 代码事实：
  - `E1` [src/hooks/useSync.ts:281]：当前本地增量只取 `updated > cursor`。
  - `E2` [src/pocketbase/notes.ts:16]：当前云端增量也只取 `updated > cursor`。
  - `E3` [src/hooks/useSync.ts:527]：当前每成功一条记录就推进游标。
- 现状结论：当前同步模型默认“本地缓存完整且游标可信”，缺少健康检查环节。
- 推断与待确认：
  - `推断`：在“本地空库 + 非初始游标”或“当前私有详情 id 本地 miss”场景下，应视为缓存不健康。
  - `待确认`：健康检查是自动重置游标，还是弹窗让用户确认后重建。

#### 3.4.3 技术方案

- 方案摘要：在同步开始前增加缓存健康检查；命中异常条件时，自动重置当前用户游标到初始值并执行补齐式增量。
- 架构落点（按层）：
  - UI（`views/components`）：用户面板展示“正在重建本地缓存”状态与结果。
  - 编排（`hooks`）：`useSync` 增加 `checkCacheHealth()` 与 `repairCacheIfNeeded()`。
  - 领域（`stores`）：补充获取本地总量与按 id 校验 helper。
  - 持久化（`database`）：无 schema 变更。
  - 云端（`pocketbase`）：继续用现有增量接口，在需要时把游标回退到初始值。
  - 路由（`router`）：无直接变更。
- 数据契约变更：
  - 字段/类型：新增内部 `CacheHealthState = healthy | emptyWithCursor | currentNoteMissing`。
  - 兼容与迁移：无。
- 同步与冲突策略：异常场景下允许一次“补齐式增量”；补齐完成后继续回到正常增量模式。
- 错误处理与回滚策略：补齐失败时保留旧游标副本并提示用户稍后重试。

#### 3.4.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 增加缓存健康检查状态机 | `src/hooks/useSync.ts`, `src/stores/notes.ts` | 无 | 中 |
| T2 | 支持异常场景下重置当前用户游标并补齐 | `src/hooks/useSync.ts` | T1 | 中 |
| T3 | 用户面板增加“重建本地缓存”反馈 | `src/components/UserProfile.vue` | T2 | 低 |

#### 3.4.5 验收标准

- AC1：本地空库但游标非初始值时，会自动进入补齐流程。
- AC2：当前私有详情路由对应 note 本地 miss 时，会阻止正常保存并优先补齐。
- AC3：补齐成功后，当前用户游标恢复正常推进。

#### 3.4.6 测试与验证

- 单元测试：健康检查命中 `healthy/emptyWithCursor/currentNoteMissing` 三种状态。
- 集成/冒烟：手动制造“空库 + 有游标”后登录，确认历史数据能被重新拉回。
- 回归关注点：首次登录默认初始游标场景不应重复全量补齐。

#### 3.4.7 风险与未决问题

- 风险：重置游标可能造成一次较大增量拉取。
- 未决问题：是否需要在弱网场景下把补齐过程拆成分页。

### 3.5 F3.8 离开页本地落盘兜底

#### 3.5.1 目标与边界

- 用户场景：用户刚编辑完内容就切后台、刷新或关闭页面时，尽量把内存中的最新笔记状态及时落到 Dexie，缩小防抖窗口风险。
- 范围内：页面离开、`pagehide`、必要时 `beforeunload` 的本地 flush。
- 范围外：服务端实时上传保证。

#### 3.5.2 现状证据

- 代码事实：
  - `E1` [src/stores/notes.ts:162]：本地 Dexie 落盘当前通过 `useRefDBSync` 进行，`debounceMs = 300`。
  - `E2` [src/database/sync.ts:190]：`useRefDBSync` 已提供 `manualSync()` 可手动触发。
  - `E3` [src/hooks/useLastVisitedRoute.ts:55]：当前 `beforeunload` 仅保存最后访问路由，不会强制 flush 笔记缓存。
- 现状结论：当前有本地自动落盘能力，但缺少页面离开前的主动 flush。
- 推断与待确认：
  - `推断`：在进程被系统回收或页面瞬时关闭的窗口里，最后一小段编辑可能仍停留在内存态。
  - `待确认`：是否在所有页面统一接入 flush，还是先限定在 `NoteDetail`。

#### 3.5.3 技术方案

- 方案摘要：复用现有 `manualSync()` 能力，在详情页离开和页面隐藏时补一次本地落盘，不改变云端同步策略。
- 架构落点（按层）：
  - UI（`views/components`）：无新增交互。
  - 编排（`hooks`）：可新增 `useLocalFlushOnLeave()`，统一注册 `pagehide`/`beforeunload`。
  - 领域（`stores`）：通过 `getNotesSync()` 暴露手动 flush。
  - 持久化（`database`）：复用现有 `manualSync()`。
  - 云端（`pocketbase`）：无。
  - 路由（`router`）：无。
- 数据契约变更：
  - 字段/类型：无。
  - 兼容与迁移：无。
- 同步与冲突策略：仅补本地落盘，不推进云端游标。
- 错误处理与回滚策略：flush 失败只记录日志，不阻断页面关闭流程。

#### 3.5.4 实施任务拆分

| 任务ID | 任务描述 | 变更文件 | 依赖 | 风险 |
| ------ | -------- | -------- | ---- | ---- |
| T1 | 暴露并复用本地手动 flush 能力 | `src/stores/notes.ts`, `src/database/sync.ts` | 无 | 低 |
| T2 | 在详情页离开/隐藏时触发本地 flush | `src/views/NoteDetail.vue` | T1 | 低 |

#### 3.5.5 验收标准

- AC1：编辑后快速切后台或关闭页面，本地 Dexie 中能尽量保留最新内容。
- AC2：本地 flush 不会错误推进云端同步游标。
- AC3：flush 失败不会阻断页面关闭，只记录可观测日志。

#### 3.5.6 测试与验证

- 单元测试：flush helper 在有无 `notesSync` 两种情况下都可安全调用。
- 集成/冒烟：编辑后立即切后台/刷新，重新进入后本地内容仍存在。
- 回归关注点：频繁切换详情页不会产生明显性能抖动。

#### 3.5.7 风险与未决问题

- 风险：过度 flush 可能增加 IndexedDB 写入频率。
- 未决问题：是否需要对大文档仅在内容变更时触发 flush。
