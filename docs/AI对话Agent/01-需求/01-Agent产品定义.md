# Agent 产品定义

## 1. Agent 定义

fastnote 中的 Agent，不定义为“无限自主”的 AI，而定义为“围绕用户当前目标，在本地笔记域内完成多步任务闭环的任务代理”。

用一句话概括：

`目标理解 + 对象定位 + 分步执行 + 状态可见 + 风险确认 + 结果收口 + 刷新可恢复`

如果只有“用户发一句话，模型调用一次工具，再返回一段文本”，它仍然只是带工具的助手，不是完整 Agent。

## 2. 产品目标

本阶段 Agent 需要解决的问题：

- 让 AI 对话从消息级回答，升级到任务级执行。
- 让“读取 -> 分析 -> 改写/执行 -> 收口”成为稳定闭环。
- 让用户知道当前任务进行到哪一步，而不是看到零散消息。
- 在网页端本地数据域内，优先基于当前页面、当前对象和本地缓存工作。
- 对写操作建立明确的风险分级和确认规则。

## 3. 非目标

本阶段不做下面这些能力：

- 通用网页浏览、系统自动化、跨应用操作。
- 服务端常驻 Agent、定时任务、无人值守任务。
- 企业级 MCP 平台或外部工具市场。
- 首期多对象批量无人确认写入。

## 4. 能力分层

为了避免“已经做了很多，但边界不清”，这里把能力拆成四层：

### 4.1 L0 对话助手

- 能回答问题。
- 能基于 Prompt 给出建议。
- 不具备任务状态和工具闭环。

### 4.2 L1 工具助手

- 能按结构化 JSON 请求本地工具。
- 能搜索、读取、更新、移动、删除、加锁。
- 能对部分高风险操作先确认。

### 4.3 L2 任务型 Agent

- 一条用户消息可以生成一个任务。
- 任务具有状态、步骤、终止原因和恢复能力。
- 工具执行后可以自动续跑，直到完成、失败或等待确认。
- 用户可以看到当前步骤和任务收口结果。

### 4.4 L3 具备记忆与技能的 Agent

- 对长对话和长正文有上下文压缩层。
- 对用户偏好、最近任务、常用对象有记忆层。
- 能按技能模板组织 Prompt、工具白名单和输出格式。
- 能通过统一的应用内方法调用协议扩展能力。

当前 fastnote 大致位于 `L1 -> L2` 之间，下一步目标是稳定进入 L2，并为 L3 预留结构。

## 5. 首期范围

首期版本建议范围如下：

- 单会话单任务
- 单任务单线程
- 以本地笔记和文件夹为主要操作域
- 支持读取型任务、改写型任务、单对象写入任务
- 支持当前对象、当前目录、fastnote 链接驱动的目标定位
- 支持刷新后恢复待确认态与中断态

## 6. 下一阶段新增范围

用户本次新增关注点，应进入下一阶段正式范围：

- 上下文压缩
- 记忆层
- 技能层
- 应用内 MCP

其中，“应用内 MCP”在 fastnote 网页端的实际含义是：

- 不引入独立 MCP Server
- 使用前端应用内注册的方法目录、参数 Schema、风险分级和执行适配器
- 让模型调用的是“受控的应用能力”，不是直接碰底层 store

## 7. 关键场景

### 7.1 链接驱动读取与改写

用户粘贴 `/n/<noteId>`，要求“读取这个备忘录并帮我重写”，Agent 需要：

1. 识别目标对象
2. 读取本地正文
3. 生成改写结果
4. 根据用户意图决定“先建议”还是“直接写回”
5. 返回收口结果

### 7.2 当前对象连续操作

用户在当前详情页里说“把这条笔记总结成三点并直接写回摘要”，Agent 需要：

1. 识别当前活动对象
2. 读取正文
3. 生成摘要
4. 按风险规则执行写回
5. 反馈结果与影响对象

### 7.3 刷新恢复

用户在待确认或执行中刷新页面，Agent 需要：

- 恢复任务摘要
- 恢复已知对象与路由快照
- 不重复执行高风险写操作
- 在必要时要求重新确认或继续任务

## 8. 验收口径

文档层面，这套 Agent 设计至少应满足：

- 能清楚区分当前已有能力与后续目标能力。
- 能说明上下文压缩、记忆、技能、应用内 MCP 分别解决什么问题。
- 能把网页端约束下的 MCP 定义为应用内方法调用，而不是抽象概念。
- 能把历史 PRD、技术方案、开发计划组织到统一入口。
