# 上下文压缩规则

## 1. 目标

这份规则用于把“上下文压缩”从原则落成执行标准，适用于 FastNote 的网页端 AI 对话 Agent。

核心目标只有四个：

- 防止长任务把上下文窗口撑爆
- 保留任务目标和最近操作的精确信息
- 压缩中间历史而不破坏调用事务合法性
- 让压缩后的上下文仍然可继续执行，而不是只剩抽象总结

可概括为一句话：

`掐头去尾，浓缩中间，修复格式，平滑续跑`

## 2. 适用范围

本规则适用于以下场景：

- 长对话
- 多轮方法调用
- 读取大正文后的连续续跑
- 长线开发型任务
- 页面刷新后的上下文恢复

不适用于：

- 首轮短问答
- 无方法调用的极短会话
- 上下文量明显远低于预算的任务

## 3. 默认参数

建议首期默认参数如下：

```ts
interface ContextCompressionConfig {
  triggerRatio: 0.5
  protectFirstN: 3
  summaryTargetRatio: 0.2
  tailMinMessages: 3
  hardSafetyMarginRatio: 0.1
  focusDetailRatio: 0.65
}
```

对应含义：

- `triggerRatio = 0.5`
  - 当当前上下文累计 token 达到模型窗口的 50% 时，触发压缩
- `protectFirstN = 3`
  - 固定保留最前面的 3 条消息
- `summaryTargetRatio = 0.2`
  - 用于计算尾部保留 token 预算
- `tailMinMessages = 3`
  - 无论预算如何，最近至少保留 3 条消息
- `hardSafetyMarginRatio = 0.1`
  - 给最终输出和计数误差预留至少 10% 安全余量
- `focusDetailRatio = 0.65`
  - 如果指定焦点主题，则约 65% 摘要预算优先保留该主题细节

## 4. 触发规则

### 4.1 自动触发

满足任一条件即触发压缩：

- 当前累计 token `>= maxContext * 0.5`
- 单次读取正文后，工具结果体积显著膨胀
- 连续方法调用达到 2 到 3 步，并且历史消息预算明显下降
- 页面刷新前准备持久化当前长任务

### 4.2 手动触发

用户可以主动触发压缩，适用于：

- 长线开发任务切阶段
- 需要保留某个专题的精细上下文
- 准备把上下文切给后续子任务

手动触发时允许传入：

- `focusTopic`
- `reason`
- `expectedNextTask`

## 5. 上下文预算规则

### 5.1 基础公式

压缩前先算可用预算：

`可用动态上下文 = 模型最大上下文窗口 - 预留输出 - 固定提示词 - 方法定义 - 安全余量`

### 5.2 历史预算公式

历史消息预算按下面公式计算：

`historyBudget = maxContext - (system + skill + methodSchemas + memory + toolResults + userInput + reservedOutput + safetyMargin)`

如果 `historyBudget <= 0`，必须立即压缩。

### 5.3 尾部预算

尾部保留区不按固定条数，而按 token 预算计算：

`tailBudget = thresholdTokens * summaryTargetRatio`

其中：

- `thresholdTokens = maxContext * triggerRatio`
- `summaryTargetRatio` 默认取 `0.2`

例如：

- `128k` 模型
- `triggerRatio = 0.5`
- `summaryTargetRatio = 0.2`

则：

- `thresholdTokens = 64k`
- `tailBudget = 12.8k`

这意味着最近上下文大致保留到约 `10%` 左右的总窗口量级。

## 6. 头尾保留规则

### 6.1 头部保留

固定保留前 `N` 条消息，默认 `N = 3`。

通常包括：

- system prompt
- 用户的初始目标
- 早期关键约束或任务边界

头部保留内容禁止参与普通压缩，除非显式进入“全量重建上下文”流程。

### 6.2 尾部保留

尾部保留区按 token 预算截取，并满足两个约束：

- 满足 `tailBudget`
- 至少保留最近 `3` 条消息

尾部内容通常包括：

- 最近一轮用户输入
- 最近一次工具结果
- 最近一次 assistant 判断
- 待确认写入草案

### 6.3 中段压缩

位于头部和尾部之间的全部消息，进入压缩区。

压缩区不直接传给主模型，而是先生成结构化摘要，再替换原始中段历史。

## 7. 摘要生成规则

### 7.1 摘要代理

压缩区应交给独立的摘要步骤处理，逻辑上等价于一个“摘要代理”。

这个摘要步骤的职责不是自由总结，而是严格按模板提炼：

- 目标
- 约束
- 已完成
- 进行中
- 受阻点
- 关键决策
- 相关文件
- 方法调用结果
- 具体错误
- 下一步建议

### 7.2 强制结构化模板

摘要输出必须使用固定结构，建议格式如下：

```md
## 任务目标

## 当前约束

## 已完成

## 进行中

## 受阻点

## 关键决策

## 相关文件与对象

## 关键错误与日志

## 待确认事项

## 下一步
```

### 7.3 信息保留优先级

摘要时按以下优先级保留信息：

1. 当前任务目标和成功标准
2. 用户显式约束
3. 最近仍有效的关键决策
4. 尚未解决的报错、失败命令、对象 ID、文件路径
5. 已完成但会影响后续判断的步骤
6. 纯过程型噪声

最后一类应最先被压缩掉。

## 8. 迭代摘要规则

如果系统已经有旧摘要，则新一轮压缩不能简单覆盖，而应做增量合并。

合并输入包括：

- 旧摘要
- 本轮新增的压缩区消息
- 当前仍保留的尾部上下文

合并规则：

- 已完成项从“进行中”移到“已完成”
- 已解决的错误从“受阻点”移除
- 已失效的临时判断不再保留
- 当前活跃对象、当前待确认事项必须更新到最新版本

摘要不是 append-only 日志，而是持续刷新的任务状态快照。

## 9. 焦点加权规则

如果压缩时指定 `focusTopic`，则摘要预算应向该主题倾斜。

建议规则：

- 约 `60% ~ 70%` 的摘要 token 预算优先服务 `focusTopic`
- 与焦点相关的文件路径、命令、报错、数值、对象 ID 尽量精确保留
- 与焦点无关的历史过程允许大幅概括，必要时直接省略

适用场景：

- 只保留某个失败测试链路
- 只保留某个模块重构上下文
- 只保留某条笔记的改写和写回过程

## 10. 调用事务合法性修复规则

这是压缩流程里的强制步骤，不能省略。

这里的“事务”默认指内部 `method_call / method_result`。

如果上游模型接口使用了 OpenAI `tool_calls` 或 Anthropic `tool_use`，应先适配成内部方法帧，再进入压缩和持久化流程。

原因：

- 调用消息和结果消息往往要求成对出现
- 截断头尾时，很容易留下“孤儿调用”或“孤儿结果”
- 如果直接把不合法序列发给模型 API，会报格式错误

### 10.1 清理对象

需要扫描并修复：

- 有 `method_call` 但没有对应 `method_result`
- 有 `method_result` 但找不到对应 `method_call`
- 同一 `methodCallId` 被重复或错配
- 若保留了供应商原生 ID，则 `vendorCallId` 也不能跨事务错配

### 10.2 清理策略

默认策略：

1. 优先保留完整成对的 `method_call / method_result`
2. 删除孤立的 `method_call`
3. 删除孤立的 `method_result`
4. 若摘要中已保留结果结论，可丢弃旧的原始工具明细

### 10.3 结果要求

压缩完成后，最终消息序列必须满足：

- 角色顺序合法
- 方法调用链闭合
- `methodCallId` 一一对应
- 不残留被截断的半条方法事务

## 11. 摘要注入规则

压缩完成后，摘要不能随意插入消息流。

默认注入方式：

- 把结构化摘要拼接到尾部保留区中的第一条 user 消息前面
- 在摘要末尾插入明确分隔线
- 分隔线后紧接真正需要响应的最新用户消息

推荐分隔线：

`--- END OF CONTEXT SUMMARY — respond to the message below, not the summary above ---`

### 11.1 注入目标

优先注入到：

- 尾部第一条 user 消息

不建议优先注入到：

- system 消息末尾
- assistant 消息中间
- 独立插入一条新的摘要消息但破坏角色交替

### 11.2 注入后效果

主模型看到的不是完整旧历史，而是：

- 头部目标和核心约束
- 一段结构化中期摘要
- 最近几轮精确上下文

这样既保留全局任务脉络，也能对最近动作继续推理。

## 12. FastNote 场景下的专项规则

结合当前项目，以下内容默认视为高价值信息，应尽量避免被早压缩：

- 当前目标 noteId / folderId
- 当前路由快照
- 最近一次 `get_note_detail` 的结论摘要
- 最近一次待确认写回草案
- 最近一次失败的方法调用原因
- 用户显式给出的写入意图，如“直接写回”或“先给建议”

以下内容可以优先进入压缩或丢弃：

- 已过时的搜索候选列表全文
- 已完成且不再影响后续判断的中间推理
- 重复出现的同类日志
- 历史成功结果的原始长正文

## 13. 首期实现建议

首期不必一步做满，建议先分三步：

1. 实现预算计算、头尾切分和孤儿工具清理
2. 实现结构化摘要生成和迭代更新
3. 实现 `focusTopic` 加权和摘要注入策略

## 14. 验收规则

压缩机制完成后，至少满足：

- 长任务不会因为上下文膨胀而频繁撞窗口
- 被压缩后仍能继续执行，不丢当前目标
- 方法调用序列在压缩后仍然合法
- 刷新恢复后仍能看出任务目标、当前进展和最近阻塞
- 指定 `focusTopic` 时，焦点相关细节明显保留得更多
