# 上下文压缩规则

## 1. 目标

这份规则用于把“上下文压缩”从原则落成执行标准，适用于 FastNote 的网页端 AI 对话 Agent。

这份规则参考了 Hermes Agent 已公开的上下文压缩思路，但按 FastNote 的网页端实现做了收敛，不直接照搬 provider cache、网关层压缩器和多 agent 编排。

核心目标只有五个：

- 防止长任务把上下文窗口撑爆
- 先用低成本清理控制膨胀，再决定是否进入摘要
- 保留任务目标和最近操作的精确信息
- 压缩中间历史而不破坏调用事务合法性
- 让压缩后的上下文仍然可继续执行，而不是只剩抽象总结

可概括为一句话：

`掐头去尾，浓缩中间，修复格式，平滑续跑`

## 2. 适用范围

本规则适用于以下场景：

- 长对话
- 多轮方法调用
- 读取大正文后的连续续跑
- 长线开发型任务
- 页面刷新后的上下文恢复

不适用于：

- 首轮短问答
- 无方法调用的极短会话
- 上下文量明显远低于预算的任务

## 3. 默认参数

建议首期默认参数如下：

```ts
interface ContextCompressionConfig {
  summaryTriggerRatio: 0.6
  hygieneTriggerRatio: 0.85
  protectFirstN: 3
  protectLastN: 8
  tailTargetRatio: 0.2
  minSummaryTokens: 1200
  maxSummaryRatio: 0.05
  hardSafetyMarginRatio: 0.1
  focusDetailRatio: 0.65
  allowLossyCompressionOnSummaryFailure: false
}
```

对应含义：

- `summaryTriggerRatio = 0.6`
  - 当动态上下文累计 token 达到预算的 60% 左右时，允许进入结构化摘要
- `hygieneTriggerRatio = 0.85`
  - 当动态上下文累计 token 达到预算的 85% 左右时，先执行卫生清理
- `protectFirstN = 3`
  - 固定保留最前面的 3 条消息
- `protectLastN = 8`
  - 无论预算如何，最近至少保留 8 条消息或等价消息块
- `tailTargetRatio = 0.2`
  - 用于计算尾部保留 token 预算
- `minSummaryTokens = 1200`
  - 低于这个规模时，不值得调用摘要模型
- `maxSummaryRatio = 0.05`
  - 摘要本身默认不应超过总窗口的 5%
- `hardSafetyMarginRatio = 0.1`
  - 给最终输出和计数误差预留至少 10% 安全余量
- `focusDetailRatio = 0.65`
  - 如果指定焦点主题，则约 65% 摘要预算优先保留该主题细节
- `allowLossyCompressionOnSummaryFailure = false`
  - 摘要失败时禁止直接删除中段原始历史

## 4. 触发规则

### 4.1 自动触发

满足任一条件即触发“卫生清理”：

- 当前累计 token `>= dynamicBudget * 0.85`
- 单次读取正文后，旧工具结果和旧正文块体积显著膨胀
- 连续方法调用达到 2 到 3 步，并且历史预算明显下降
- 页面刷新前准备持久化当前长任务，且原始历史已偏长

满足任一条件即允许进入“结构化摘要”：

- 当前累计 token `>= dynamicBudget * 0.6`
- 已完成卫生清理，但历史预算仍然不足
- 当前任务进入待确认、中断、完成，需要形成恢复摘要

### 4.2 手动触发

用户可以主动触发压缩，适用于：

- 长线开发任务切阶段
- 需要保留某个专题的精细上下文
- 准备把上下文切给后续子任务

手动触发时允许传入：

- `focusTopic`
- `reason`
- `expectedNextTask`

## 5. 两段式压缩流程

这是参考 Hermes Agent 后首期必须明确的核心流程：

1. 先做卫生清理
2. 再做结构化摘要
3. 最后做恢复快照落盘

### 5.1 卫生清理

卫生清理只做低成本、可解释、尽量无损的收缩，不调用摘要模型。

优先处理：

- 保护区之外的旧工具结果长文本
- 保护区之外的旧正文原文
- 已经过时的搜索候选全文
- 重复出现的同类日志与状态播报

推荐动作：

- 用 `summary / code / objectId` 替代完整结果体
- 对超长正文只保留标题、对象 ID、核心结论、截断标记
- 仅保留最近一次失败错误的完整上下文，其余历史错误转为摘要

### 5.2 结构化摘要

如果卫生清理后仍超预算，再对中段历史做结构化摘要。

摘要输入包括：

- 旧摘要
- 中段待压缩消息
- 当前尾部保护区
- 当前活动对象和待确认写回草案

### 5.3 落盘快照

压缩结束后，摘要不应只存在于一次请求中，还要更新 working memory。

working memory 至少包含：

- `taskSummary`
- `activeTargetSummary`
- `latestToolResultSummary`
- `pendingMutationSummary`
- `lastCompressionAt`
- `compressionReason`

## 6. 上下文预算规则

### 6.1 基础公式

压缩前先算可用预算：

`可用动态上下文 = 模型最大上下文窗口 - 预留输出 - 固定提示词 - 方法定义 - 安全余量`

### 6.2 历史预算公式

历史消息预算按下面公式计算：

`historyBudget = maxContext - (system + skill + methodSchemas + memory + toolResults + userInput + reservedOutput + safetyMargin)`

如果 `historyBudget <= 0`，必须立即压缩。

### 6.3 尾部预算

尾部保留区不按固定条数，而按 token 预算计算：

`tailBudget = dynamicBudget * tailTargetRatio`

其中：

- `dynamicBudget = maxContext - fixedCosts`
- `tailTargetRatio` 默认取 `0.2`

例如：

- `128k` 模型
- `tailTargetRatio = 0.2`

则：

- 如果 `dynamicBudget = 96k`
- 则 `tailBudget = 19.2k`

这意味着最近上下文大致保留到约 `15%` 左右的总窗口量级；如果按条数换算，至少还要满足 `protectLastN`。

### 6.4 摘要预算

摘要预算不应写死，而应同时满足上下限：

`summaryBudget = min(maxSummaryRatio * maxContext, compressibleTokens * 0.3)`

并且：

- 不低于 `minSummaryTokens`
- 不高于 `maxSummaryRatio * maxContext`

这样可以避免“为了压缩 3k 内容，反而生成 4k 摘要”。

## 7. 头尾保留规则

### 7.1 头部保留

固定保留前 `N` 条消息，默认 `N = 3`。

通常包括：

- system prompt
- 用户的初始目标
- 早期关键约束或任务边界

头部保留内容禁止参与普通压缩，除非显式进入“全量重建上下文”流程。

### 7.2 尾部保留

尾部保留区按 token 预算截取，并满足两个约束：

- 满足 `tailBudget`
- 至少保留最近 `8` 条消息或等价消息块

尾部内容通常包括：

- 最近一轮用户输入
- 最近一次工具结果
- 最近一次 assistant 判断
- 待确认写入草案

这个规则参考了 Hermes 的“保守保护尾部”原则。对于连续工具调用型任务，保留过短的尾部会明显损害续跑稳定性。

### 7.3 中段压缩

位于头部和尾部之间的全部消息，进入压缩区。

压缩区不直接传给主模型，而是先生成结构化摘要，再替换原始中段历史。

## 8. 摘要生成规则

### 8.1 摘要代理

压缩区应交给独立的摘要步骤处理，逻辑上等价于一个“摘要代理”。

这个摘要步骤的职责不是自由总结，而是严格按模板提炼：

- 目标
- 约束
- 已完成
- 进行中
- 受阻点
- 关键决策
- 相关文件
- 方法调用结果
- 具体错误
- 下一步建议

### 8.2 强制结构化模板

摘要输出对内应保存为结构化对象，对外可渲染成固定 Markdown 模板。

建议内部结构：

```ts
interface AiCompressedContext {
  taskSummary: string
  constraints: string[]
  completed: string[]
  inProgress: string[]
  blockers: string[]
  decisions: string[]
  relatedObjects: string[]
  errors: string[]
  pendingConfirmations: string[]
  nextSteps: string[]
}
```

建议渲染格式如下：

```md
## 任务目标

## 当前约束

## 已完成

## 进行中

## 受阻点

## 关键决策

## 相关文件与对象

## 关键错误与日志

## 待确认事项

## 下一步
```

### 8.3 信息保留优先级

摘要时按以下优先级保留信息：

1. 当前任务目标和成功标准
2. 用户显式约束
3. 最近仍有效的关键决策
4. 尚未解决的报错、失败命令、对象 ID、文件路径
5. 已完成但会影响后续判断的步骤
6. 纯过程型噪声

最后一类应最先被压缩掉。

## 9. 迭代摘要规则

如果系统已经有旧摘要，则新一轮压缩不能简单覆盖，而应做增量合并。

合并输入包括：

- 旧摘要
- 本轮新增的压缩区消息
- 当前仍保留的尾部上下文

合并规则：

- 已完成项从“进行中”移到“已完成”
- 已解决的错误从“受阻点”移除
- 已失效的临时判断不再保留
- 当前活跃对象、当前待确认事项必须更新到最新版本

摘要不是 append-only 日志，而是持续刷新的任务状态快照。

## 10. 焦点加权规则

如果压缩时指定 `focusTopic`，则摘要预算应向该主题倾斜。

建议规则：

- 约 `60% ~ 70%` 的摘要 token 预算优先服务 `focusTopic`
- 与焦点相关的文件路径、命令、报错、数值、对象 ID 尽量精确保留
- 与焦点无关的历史过程允许大幅概括，必要时直接省略

适用场景：

- 只保留某个失败测试链路
- 只保留某个模块重构上下文
- 只保留某条笔记的改写和写回过程

## 11. 摘要失败与回退规则

这是参考 Hermes Agent 后必须补上的硬规则。

### 11.1 禁止静默丢失

以下任一情况发生时，不得删除中段原始历史：

- 摘要模型调用失败
- 摘要模型超时
- 摘要模型可用上下文小于待压缩输入
- 旧摘要与新摘要合并失败

### 11.2 回退顺序

默认回退顺序：

1. 保留已完成的卫生清理结果
2. 放弃本轮结构化摘要
3. 标记 `compressionStatus = partial`
4. 在任务状态里记录 `compressionFailureReason`

### 11.3 结果要求

压缩流程失败后，也必须满足：

- 当前目标对象不丢失
- 待确认写回草案不丢失
- 最近失败原因不丢失
- 页面刷新后仍能恢复到“未完成深度压缩”的状态

## 12. 调用事务合法性修复规则

这是压缩流程里的强制步骤，不能省略。

这里的“事务”默认指内部 `method_call / method_result`。

如果上游模型接口使用了 OpenAI `tool_calls` 或 Anthropic `tool_use`，应先适配成内部方法帧，再进入压缩和持久化流程。

原因：

- 调用消息和结果消息往往要求成对出现
- 截断头尾时，很容易留下“孤儿调用”或“孤儿结果”
- 如果直接把不合法序列发给模型 API，会报格式错误

### 12.1 清理对象

需要扫描并修复：

- 有 `method_call` 但没有对应 `method_result`
- 有 `method_result` 但找不到对应 `method_call`
- 同一 `methodCallId` 被重复或错配
- 若保留了供应商原生 ID，则 `vendorCallId` 也不能跨事务错配

### 12.2 清理策略

默认策略：

1. 优先保留完整成对的 `method_call / method_result`
2. 删除孤立的 `method_call`
3. 删除孤立的 `method_result`
4. 若摘要中已保留结果结论，可丢弃旧的原始工具明细

### 12.3 结果要求

压缩完成后，最终消息序列必须满足：

- 角色顺序合法
- 方法调用链闭合
- `methodCallId` 一一对应
- 不残留被截断的半条方法事务

## 13. 摘要注入规则

压缩完成后，摘要不能随意插入消息流。

FastNote 首期不建议把摘要文本直接写回普通用户消息，而应：

- 先把摘要作为 working memory 单独持久化
- 在组装请求时，把它转成一条隐藏系统摘要块
- 这条摘要块位于系统提示词之后、尾部保护区之前

推荐分隔线：

`--- END OF CONTEXT SUMMARY — respond to the message below, not the summary above ---`

### 13.1 注入目标

优先注入到：

- 单独的隐藏摘要块

不建议优先注入到：

- 最后一条真实用户消息正文里
- 现有活动对象 system prompt 的正文里
- 任意覆盖普通会话消息原文

### 13.2 注入后效果

主模型看到的不是完整旧历史，而是：

- 头部目标和核心约束
- 一段结构化中期摘要
- 最近几轮精确上下文

这样既保留全局任务脉络，也能对最近动作继续推理。

## 14. FastNote 场景下的专项规则

结合当前项目，以下内容默认视为高价值信息，应尽量避免被早压缩：

- 当前目标 noteId / folderId
- 当前路由快照
- 最近一次 `get_note_detail` 的结论摘要
- 最近一次待确认写回草案
- 最近一次失败的方法调用原因
- 用户显式给出的写入意图，如“直接写回”或“先给建议”

以下内容可以优先进入压缩或丢弃：

- 已过时的搜索候选列表全文
- 已完成且不再影响后续判断的中间推理
- 重复出现的同类日志
- 历史成功结果的原始长正文

再补一条从 Hermes 迁移来的特殊规则：

- 摘要模型的上下文窗口不能小于主执行模型的主工作区预期
  - 如果摘要模型更小，只能处理卫生清理后的子块，不得吞整段历史

## 15. 首期实现建议

首期不必一步做满，建议先分四步：

1. 实现预算计算、头尾切分和卫生清理
2. 实现结构化摘要生成、迭代更新和失败回退
3. 实现独立 working memory 持久化与请求装配
4. 最后补 `focusTopic` 加权和更细的摘要注入策略

## 16. 验收规则

压缩机制完成后，至少满足：

- 长任务不会因为上下文膨胀而频繁撞窗口
- 旧工具结果会先被卫生清理，而不是无条件进入摘要模型
- 被压缩后仍能继续执行，不丢当前目标
- 方法调用序列在压缩后仍然合法
- 刷新恢复后仍能看出任务目标、当前进展和最近阻塞
- 指定 `focusTopic` 时，焦点相关细节明显保留得更多
- 摘要失败时不会静默删除中段历史
