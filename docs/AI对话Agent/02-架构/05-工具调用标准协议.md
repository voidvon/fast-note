# 工具调用标准协议

## 1. 目标

这份协议用于规范 FastNote AI 对话 Agent 的工具调用链。

本阶段目标不是实现完整 MCP，也不是同时支持所有厂商特性，而是先解决三个问题：

- 让工具调用从“自由文本 JSON 解析”升级到“标准调用帧”
- 让 OpenAI、Anthropic、现有 JSON envelope 都能落到同一套内部结构
- 让后续压缩、恢复、确认执行和重复调用拦截有稳定的数据基座

一句话定义：

`对外多协议适配，对内统一 tool_call / tool_result 帧`

## 2. 当前问题

当前实现主要依赖模型输出一段 JSON envelope，再由前端解析：

- `mode: "tool_calls"`
- `toolCalls: [...]`
- `answer: "..."`

这套做法能跑，但有明显问题：

- 模型一旦多说一句、漏一个字段、包错 Markdown，解析就不稳
- 工具调用不是标准消息帧，压缩时更容易切成半条事务
- 刷新恢复时缺少稳定的 `tool_call_id`
- 无法优雅兼容 OpenAI 原生 `tool_calls` 和 Anthropic 原生 `tool_use`

## 3. 本阶段边界

### 3.1 范围内

- 定义内部统一工具调用协议
- 设计 OpenAI / Anthropic / JSON envelope 三种输入适配
- 统一风险分级、确认语义、结果回流格式
- 为现有 8 个工具建立标准注册表接口

### 3.2 不在范围

- 完整 MCP client / server
- 技能层裁剪
- 长期记忆
- 多 Agent 并行协作

## 4. 总体原则

### 4.1 对外兼容

外部模型可以继续使用不同风格：

- OpenAI `tool_calls`
- Anthropic `tool_use`
- 当前 JSON envelope fallback

### 4.2 对内统一

所有外部格式进入 FastNote 后，都要先转成内部统一帧：

- `ToolCallFrame`
- `ToolResultFrame`

### 4.3 执行闭环

一次工具事务至少包含：

1. 模型发起 `tool_call`
2. 本地执行并产出 `tool_result`
3. 结果回流模型或进入确认态

### 4.4 强约束优于自由文本

能结构化的地方都不要继续依赖自由文本解析。

## 5. 内部统一数据结构

### 5.1 方法名

首期方法名继续沿用当前工具名：

- `search_notes`
- `get_note_detail`
- `list_folders`
- `create_note`
- `update_note`
- `move_note`
- `delete_note`
- `set_note_lock`

### 5.2 Tool Call Frame

建议内部统一为：

```ts
type AiToolProtocolVendor = 'openai' | 'anthropic' | 'json_envelope' | 'internal'

interface ToolCallFrame<TInput = unknown> {
  id: string
  toolName: string
  input: TInput
  vendor: AiToolProtocolVendor
  status: 'requested' | 'validated' | 'rejected' | 'executing' | 'awaiting_confirmation' | 'completed' | 'failed'
  createdAt: number
  taskId?: string
  requestId?: string
  riskLevel?: 'none' | 'low' | 'medium' | 'high'
  requiresConfirmation?: boolean
  dryRun?: boolean
  confirmed?: boolean
  sourceMessageId?: string
}
```

### 5.3 Tool Result Frame

```ts
interface ToolResultFrame<TOutput = unknown> {
  id: string
  toolCallId: string
  toolName: string
  vendor: AiToolProtocolVendor
  status: 'success' | 'error' | 'blocked' | 'confirmation_required'
  ok: boolean
  code: string
  message: string | null
  output?: TOutput
  preview?: {
    title: string
    summary: string
    affectedNoteIds: string[]
  }
  affectedNoteIds?: string[]
  syncQueued?: boolean
  humanActionRequired?: boolean
  createdAt: number
}
```

### 5.4 事务关系

一条 `ToolResultFrame` 必须绑定一条 `ToolCallFrame`：

- `toolCallId` 必填
- `toolName` 必须一致
- `tool_call / tool_result` 必须成对闭合

## 6. 方法注册表

首期先做最小注册表，不上完整 MCP。

```ts
interface AiMethodDefinition<TInput = unknown, TOutput = unknown> {
  name: string
  description: string
  inputSchema: unknown
  outputSchema: unknown
  riskLevel: 'none' | 'low' | 'medium' | 'high'
  requiresConfirmation: boolean
  execute: (input: TInput) => Promise<TOutput>
}
```

注册表职责：

- 校验方法名是否合法
- 校验输入结构
- 提供默认风险分级
- 提供默认确认策略
- 调用真正执行器

## 7. 协议适配层

### 7.1 OpenAI 适配

OpenAI 风格通常包含：

- `tool_calls`
- `id`
- `function.name`
- `function.arguments`

映射规则：

- `tool_calls[].id -> ToolCallFrame.id`
- `function.name -> toolName`
- `function.arguments -> input`
- `vendor = 'openai'`

### 7.2 Anthropic 适配

Anthropic 风格通常包含：

- `tool_use`
- `id`
- `name`
- `input`

映射规则：

- `tool_use.id -> ToolCallFrame.id`
- `tool_use.name -> toolName`
- `tool_use.input -> input`
- `vendor = 'anthropic'`

### 7.3 JSON Envelope 适配

当前兼容格式：

```json
{
  "mode": "tool_calls",
  "answer": "可选文本",
  "toolCalls": [
    {
      "tool": "search_notes",
      "payload": {
        "query": "xxx"
      }
    }
  ]
}
```

映射规则：

- 若没有显式 `id`，本地生成 `toolCallId`
- `tool -> toolName`
- `payload -> input`
- `vendor = 'json_envelope'`

这是 fallback，不再作为主协议。

## 8. 风险与确认规则

### 8.1 风险来源

风险级别不应只靠模型声明，而应由本地注册表和策略层共同决定。

优先级建议：

1. 本地方法注册表默认值
2. 本地变更策略修正
3. 模型附带建议

### 8.2 默认确认规则

建议首期沿用现有语义：

- `search_notes`：不确认
- `get_note_detail`：不确认
- `list_folders`：不确认
- `create_note`：可直执行
- `update_note`：按策略判断，默认中风险
- `move_note`：必须确认
- `delete_note`：必须确认
- `set_note_lock`：必须确认

### 8.3 帧级状态

当需要确认时：

- `ToolCallFrame.status = 'awaiting_confirmation'`
- `ToolResultFrame.status = 'confirmation_required'`

确认后不应创建一条全新的语义调用，而应复用原 `toolCallId` 延续事务。

## 9. 执行流程

建议标准流程如下：

1. 接收模型输出
2. 识别协议来源
3. 转换为 `ToolCallFrame[]`
4. 通过方法注册表校验
5. 应用风险/确认策略
6. 进入执行或待确认
7. 产出 `ToolResultFrame[]`
8. 将结果回流模型或直接收口

## 10. 压缩与恢复要求

由于后续要做上下文压缩，这里先定义要求：

- 上下文中尽量保留 `toolCallId`
- 压缩前后必须能识别哪一条结果属于哪一条调用
- 孤立的 `tool_call` 或 `tool_result` 必须清理
- 刷新恢复时，待确认态优先恢复 `ToolCallFrame`

换句话说，这套协议不是单纯为了“调用更规范”，而是为了让压缩、恢复、重复执行拦截都能基于同一事务 ID 工作。

## 11. 与现有类型的关系

当前已有：

- `AiNoteToolCall`
- `AiToolResult`

建议关系如下：

- `AiNoteToolCall`
  - 继续作为“方法入参级别类型”
- `AiToolResult`
  - 继续作为“方法执行结果级别类型”
- `ToolCallFrame`
  - 新增，作为“协议事务层”
- `ToolResultFrame`
  - 新增，作为“协议事务层结果”

也就是说：

- `AiNoteToolCall` / `AiToolResult` 偏执行器
- `ToolCallFrame` / `ToolResultFrame` 偏编排协议

两层不要混在一起。

## 12. 推荐的目录落点

建议新增如下文件：

- `fastnote/src/shared/types/ai-tool-protocol.ts`
- `fastnote/src/features/ai-chat/model/tool-protocol-adapter.ts`
- `fastnote/src/features/ai-chat/model/tool-protocol-openai.ts`
- `fastnote/src/features/ai-chat/model/tool-protocol-anthropic.ts`
- `fastnote/src/features/ai-chat/model/tool-protocol-envelope.ts`
- `fastnote/src/features/ai-chat/model/method-registry.ts`

现有文件角色建议：

- `ai-note-tools.ts`
  - 保留工具 payload/result 类型
- `assistant-envelope.ts`
  - 逐步退化为 envelope fallback 解析器
- `use-ai-chat.ts`
  - 改为围绕协议帧编排，而不是直接围绕 envelope 编排

## 13. 首期实施顺序

建议按下面顺序推进：

1. 新增 `ToolCallFrame` / `ToolResultFrame`
2. 写 JSON envelope -> 标准帧转换
3. 写 OpenAI `tool_calls` -> 标准帧转换
4. 把 `useAiChat()` 的执行入口改成消费标准帧
5. 把 `useAiChatSession()` 的待确认存储改成存标准帧
6. 最后再补 Anthropic 适配器

原因：

- 现有系统已经是 envelope 驱动
- OpenAI 风格是首个最值得接入的主路径
- Anthropic 可以在内部协议稳定后再补

## 14. 最小验收标准

协议层完成后，至少满足：

- 同一轮工具调用都有稳定 `toolCallId`
- OpenAI / envelope 两条路径都能落到同一内部帧
- 待确认执行可以基于帧恢复，而不是只恢复 payload
- 压缩时可以识别并清理孤立工具事务
- `useAiChat()` 不再依赖自由文本 JSON 才能进入工具编排
