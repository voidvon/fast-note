# 上下文压缩与记忆

## 1. 为什么现在必须补这一层

当前实现已经能做连续工具调用，但随着任务变长，会逐步出现下面的问题：

- 消息越来越长，模型看到的是堆叠文本，不是任务摘要。
- `get_note_detail`、搜索结果、执行结果混在一起，续跑成本越来越高。
- 页面刷新后只能恢复部分状态，不能恢复“任务为什么停在这里”。
- 同一个用户重复表达偏好时，没有稳定复用。

所以“上下文压缩”和“记忆”不该继续塞进系统提示词，而应该独立成层。

## 2. 先区分三个概念

### 2.1 请求上下文

指当前这一轮请求附带的页面信息。

例子：

- 当前路由
- 当前活动笔记
- 当前目录
- 候选对象
- 最近对象

这部分是“即时上下文”，已有基础实现。

### 2.2 压缩上下文

指对长对话、长正文、历史工具结果做裁剪后，仍保留任务所需关键信息的摘要层。

例子：

- 最近 8 轮对话的任务摘要
- 上一轮读取到的目标对象和正文结论
- 最近一次搜索命中的 3 条候选对象

这部分当前缺失。

### 2.3 记忆

指可跨轮次、跨刷新、甚至跨会话复用的信息。

例子：

- 用户偏好“改写默认先给建议，不直接写回”
- 用户常用目录“工作 / 会议 / 个人”
- 最近一次成功完成的任务摘要

这部分当前也缺失。

### 2.4 参考 Hermes Agent 后的补充结论

参考 Hermes Agent 现有文档与实现，FastNote 在“上下文压缩”上不应只做一层“中段摘要”，而应至少拆成两段：

- 请求前卫生清理
  - 在真正调用摘要模型前，先裁掉明显过时、又体积很大的旧工具结果和旧正文块。
- 运行中精确压缩
  - 当历史消息仍然超预算时，再对中段做结构化摘要。

Hermes 的经验对 FastNote 最有价值的不是某个固定阈值，而是下面四条工程规律：

- 先做便宜的裁剪，再做昂贵的 LLM 摘要。
- 头尾保护要足够保守，不能只剩最近 2 到 3 条。
- 新摘要应增量合并旧摘要，而不是每次重写整段历史。
- 摘要失败时绝不能静默丢失中段上下文，必须回退到“不压缩”或“只做无损裁剪”。

FastNote 不直接照搬 Hermes 的地方也要写清楚：

- Hermes 还讨论了网关层和 provider cache；FastNote 首期仍以网页端请求组装层为主，不额外引入网关压缩器。
- Hermes 会处理更通用的多工具 agent 序列；FastNote 首期重点是笔记读取、搜索、写回、待确认执行这几类对象。
- Hermes 会根据模型家族做更细的上下文与缓存策略；FastNote 首期先统一成模型无关的预算与回退规则。

## 3. 建议的记忆分层

### 3.1 会话工作记忆

作用：

- 支撑当前任务的多步执行
- 控制 token 膨胀
- 提升续跑稳定性

建议内容：

- 当前任务摘要
- 当前唯一目标对象
- 最近一次工具调用结果摘要
- 最近一次待确认写入草案

建议生命周期：

- 当前会话内有效
- 页面刷新后可恢复
- 任务完成后可以衰减或归档

### 3.2 短期任务记忆

作用：

- 支撑“刚做完什么、为什么停住、是否可继续”

建议内容：

- 任务输入摘要
- 步骤列表摘要
- 终止原因
- 最终结果摘要
- 路由对象快照

建议生命周期：

- 保留最近若干个任务
- 优先本地存储

### 3.3 用户偏好记忆

作用：

- 减少每次都要重说一遍的操作偏好

建议内容：

- 默认回复风格
- 是否偏好先预览再写回
- 常用目录别名
- 语言偏好

建议规则：

- 只保存明确、稳定、用户可解释的偏好
- 不自动保存高风险推断

### 3.4 领域记忆

作用：

- 为后续技能提供稳定背景

建议内容：

- 常用工作流别名
- 常见文档类型
- 常用文件夹映射

这一层不是首期必做，但结构上要预留。

## 4. 压缩策略

### 4.1 上下文容量通常怎么计算

讨论压缩规则前，先要把“可用上下文”拆成预算，而不是只看模型标称窗口。

最常见的计算方式是：

`可用动态上下文 = 模型最大上下文窗口 - 预留输出 - 固定提示词 - 方法/工具定义 - 安全余量`

对应到 FastNote 这类 Agent，可拆成：

- 模型最大上下文窗口
  - 例如 `32k`、`64k`、`128k`
- 预留输出
  - 给模型最终回答、工具请求 JSON、续跑收口留空间
- 固定提示词
  - 系统提示词、产品规则、Agent 通用约束
- 方法/工具定义
  - 工具 Schema、应用内 MCP 方法目录、技能附加约束
- 安全余量
  - 防止流式补充、JSON 膨胀、计数误差导致撞满窗口

真正能拿来放“历史消息 + 工作记忆 + 工具结果 + 当前输入”的，只是剩下的这一部分。

### 4.2 一轮请求的预算拆法

把一次请求拆开后，通常是下面这几个输入块：

- `system`
  - 通用系统提示词
- `skill`
  - 技能附加提示词
- `methodSchemas`
  - 方法定义、工具说明、Schema
- `memory`
  - 压缩后的工作记忆、短期记忆、偏好记忆
- `history`
  - 最近几轮消息
- `toolResults`
  - 最近一次或最近几次工具结果
- `userInput`
  - 当前用户输入
- `reservedOutput`
  - 给模型输出预留的 token
- `safetyMargin`
  - 安全余量

可写成更具体的公式：

`historyBudget = maxContext - (system + skill + methodSchemas + memory + toolResults + userInput + reservedOutput + safetyMargin)`

如果 `historyBudget <= 0`，说明不能再继续堆历史消息，必须先压缩。

### 4.3 典型阈值怎么定

参考 Hermes Agent 的压缩思路，FastNote 更适合用“两段阈值”，而不是单一阈值：

- 卫生清理阈值
  - 一般设在总窗口或动态上下文的 `80% ~ 85%`
  - 先触发“旧工具结果 / 旧正文块”的便宜裁剪
- 摘要压缩阈值
  - 一般设在动态上下文的 `50% ~ 70%`
  - 用于触发真正的中段摘要生成
- 硬阈值
  - 一般设在动态上下文的 `85% ~ 90%`
  - 超过后禁止继续堆原文，只允许保留摘要、关键对象、最近尾部和待确认草案

建议再单独留一块输出保护区：

- 输出保护区
  - 通常不低于总窗口的 `10%`
  - 避免模型没空间完成回答或输出合法 JSON

### 4.4 一个具体例子

假设使用 `128k` 上下文模型，可以这样分：

- 最大窗口：`128k`
- 预留输出：`8k`
- 固定系统提示词：`4k`
- 技能与方法定义：`10k`
- 安全余量：`10k`

那么可用动态上下文大约是：

`128k - 8k - 4k - 10k - 10k = 96k`

这 `96k` 才是给下面这些内容共用的：

- 历史消息
- 压缩记忆
- 工具结果
- 当前用户输入

如果再设：

- 摘要压缩阈值：`96k * 0.6 = 57.6k`
- 卫生清理阈值：`96k * 0.85 = 81.6k`

那就可以定义：

- 小于 `57k`：正常保留近期对话
- `57k ~ 81k`：允许增量摘要旧消息
- 大于 `81k`：先做旧工具结果和旧正文块清理，再决定是否生成新摘要

### 4.5 token 通常怎么估算

最准的方法永远是用目标模型对应的 tokenizer 做实际计数。

如果讨论方案阶段还没有接 tokenizer，通常会先用粗算：

- 英文文本
  - 可粗略按 `1 token ≈ 3 到 4 个字符`
- 中文文本
  - 粗略上通常比英文更贵
  - 保守估算可以按 `1 个汉字 ≈ 1 到 2 个 token` 看待
- JSON / Schema / URL / id
  - 往往比自然语言更费 token
  - 不能按“字符少就一定便宜”处理

所以在中文笔记场景里，最稳妥的做法不是按字符数卡死，而是：

- 方案阶段先用保守粗算
- 实现阶段接入真实 tokenizer
- 线上再加一层安全余量

### 4.6 对压缩规则真正有用的不是总量，而是分配比例

讨论压缩规则时，应该讨论这几个比例，而不是只说“模型有 128k”：

- 固定提示词占比
- 方法定义占比
- 记忆层占比
- 历史消息占比
- 工具结果占比
- 预留输出占比

因为上下文压缩本质上是在争夺这些预算。

### 4.7 针对 FastNote 的建议口径

FastNote 的任务里，最容易膨胀的不是普通聊天，而是下面三块：

- 长正文 `get_note_detail`
- 多轮工具结果回流
- 过长的系统提示词和方法定义

所以这里建议优先压缩顺序是：

1. 旧消息明细
2. 旧工具结果明细
3. 长正文原文
4. 只保留任务摘要、关键对象、最近一步结果

不建议优先压缩当前任务的目标对象和待确认写入草案，因为这两类信息最影响正确性。

这里再补一条 Hermes 给出的工程启发：

- 旧工具结果明细不一定都要进入摘要模型
  - 先把过长的旧结果块替换成稳定占位或短摘要，再决定是否做 LLM 摘要
  - 否则摘要模型本身也会被旧结果撑爆

### 4.8 触发时机

建议在以下时机触发压缩：

- 对话轮数超过阈值
- 单次读取正文过长
- 工具调用连续超过 2 到 3 步
- 页面刷新前持久化任务
- 进入待确认或中断态

### 4.9 压缩单位

不要直接压缩整段会话，应该按对象压缩：

- 用户意图摘要
- 目标对象摘要
- 工具轨迹摘要
- 结果摘要
- 待确认写入摘要

### 4.10 压缩输出格式

建议统一成结构化对象，而不是自由文本：

```ts
interface AiCompressedContext {
  taskSummary: string
  activeTargetSummary?: string
  recentSearchSummary?: string[]
  latestToolResultSummary?: string
  pendingMutationSummary?: string
}
```

这样可以直接拼进 Prompt，也方便恢复和调试。

### 4.11 摘要失败保护

这是参考 Hermes Agent 后必须补上的强约束：

- 如果摘要模型调用失败，不得直接删除中段历史
- 如果摘要模型的可用上下文反而小于待压缩内容，不得强行进入摘要流程
- 如果旧摘要与新压缩区合并失败，必须保留旧摘要并放弃本轮重压缩
- 如果只能做到卫生清理，也要把当前任务标记为“已做轻量裁剪，未完成深度压缩”

核心原则只有一句：

`压缩失败时，宁可保留较长上下文，也不要静默损坏任务状态。`

## 5. 记忆读写规则

### 5.1 读取规则

- 先取当前页面上下文
- 再取当前任务工作记忆
- 再按需要补短期记忆
- 最后才注入用户偏好记忆

不要一上来把所有记忆都塞给模型。

### 5.2 写入规则

- 只写结论，不写 chain-of-thought
- 只写可解释摘要，不写模型噪声
- 高风险内容不自动进入长期记忆
- 用户显式否认的信息要可覆盖或清除

### 5.3 清理规则

- 会话工作记忆按任务结束自动衰减
- 短期任务记忆保留最近 N 条
- 用户偏好允许显式清空

## 6. 存储建议

结合当前 fastnote 的网页端架构，建议分两层：

- `localStorage`：保存当前任务恢复态、最近任务摘要、轻量偏好
- IndexedDB/Dexie：如果后续需要更大容量的长期记忆，再引入持久化表

首期不建议把长期记忆直接混进现有 `ai-chat-conversation` 或 `ai-chat-session` 的旧 key 里，最好独立命名。

结合 Hermes 的压缩实现经验，首期建议新增独立快照，而不是把摘要文本混写进普通消息数组：

- `ai-chat-working-memory:<scope>`
  - 保存当前任务的压缩摘要、关键对象摘要、最近工具结果摘要、待确认写回摘要
- `ai-chat-task-memory:<scope>`
  - 保存最近若干个已完成或已中断任务的摘要记录
- `ai-chat-preference-memory:<scope>`
  - 保存少量显式偏好

这样做的好处是：

- 压缩摘要和原始消息可以分别演进
- 刷新恢复时可以先恢复 working memory，再决定是否补原始消息
- 后续改摘要格式时，不必破坏旧消息持久化结构

## 7. 与当前实现的衔接

当前已经能复用的基础：

- `AiChatRequestContext`
- `AiAgentTask`
- `AiChatPendingExecution`
- `routeTargetSnapshot`

下一步建议新增的对象：

- `AiCompressedContext`
- `AiWorkingMemory`
- `AiTaskMemoryRecord`
- `AiUserPreferenceMemory`

## 8. 首期落地建议

如果按最小实现推进，建议先做下面三件事：

1. 在任务进入待确认、完成、中断时生成 `taskSummary`。
2. 在续跑前先对旧工具结果做卫生清理，不再默认回传整段历史结果。
3. 在真正超预算时再生成结构化摘要，并单独持久化为 working memory。
4. 为用户增加极少量、显式可见的偏好记忆，例如“默认先预览再写回”。

如果要继续往前一步，建议马上补第 5 件事：

5. 在实现里引入统一的 `contextBudget` 计算，先按粗算规则跑，再接真实 tokenizer。
