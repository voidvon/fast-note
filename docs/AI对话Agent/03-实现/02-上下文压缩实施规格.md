# 上下文压缩实施规格

## 1. 目的

这份文档把“上下文压缩规则”进一步收敛成可直接开发的实现规格，适用于 FastNote 网页端 AI 对话 Agent 首期落地。

它回答四个问题：

- 压缩相关数据结构具体长什么样
- 压缩结果存到哪里、何时读写
- 请求发出前如何组装上下文
- 首期需要补哪些测试

本文件是 `02-架构/04-上下文压缩规则.md` 的实现侧补充，不替代架构规则。

## 2. 当前代码入口

首期实现必须优先挂到下面几个真实入口：

- `fastnote/src/features/ai-chat/model/use-ai-chat.ts`
  - `hydrateConversation()`
  - `hydrateAgentTask()`
  - `persistConversation()`
  - `persistAgentTask()`
  - `createChatRequestBody()`
  - `continueAssistantAfterToolResults()`
  - `sendMessage()`
  - `resumeInterruptedTask()`
  - `regenerate()`
- `fastnote/src/features/ai-chat/model/openai-compatible-chat-transport.ts`
  - `buildOpenAiMessages()`
  - `requestOpenAiCompatibleCompletion()`
- `fastnote/src/features/ai-chat/model/request-context.ts`
  - 当前页面上下文 system prompt 生成入口

当前真实限制也要明确：

- `createChatRequestBody()` 现在只注入页面上下文，不注入 working memory。
- `continueAssistantAfterToolResults()` 现在会把工具详细结果整体拼进隐藏 system prompt。
- 持久化目前只有 `ai-chat-conversation` 和 `ai-chat-agent-task` 两个 key。
- `buildOpenAiMessages()` 现在只拼 `system + contextSystemPrompt + messages`，没有记忆层插槽。

建议新增以下文件：

- `fastnote/src/features/ai-chat/model/memory/context-compression.ts`

  - working memory 类型、读写、清理
- `fastnote/src/features/ai-chat/model/memory/task-memory.ts`
  - 短期任务记忆记录
- `fastnote/src/features/ai-chat/model/memory/preference-memory.ts`
  - 用户偏好记忆
- `fastnote/src/features/ai-chat/model/memory/request-assembly.ts`

如果首期要更保守，也可以先只落三个文件：

- `context-compression.ts`
- `working-memory.ts`
- `request-assembly.ts`

## 4. 数据结构

### 4.1 压缩配置

```ts
export interface ContextCompressionConfig {
  summaryTriggerRatio: number
  hygieneTriggerRatio: number
  protectFirstN: number
  protectLastN: number
  tailTargetRatio: number
  minSummaryTokens: number
  maxSummaryRatio: number
  hardSafetyMarginRatio: number
  focusDetailRatio: number
  allowLossyCompressionOnSummaryFailure: boolean
}
```

### 4.2 压缩状态

```ts
export type AiCompressionStatus =
  | 'idle'
  | 'hygiene_only'
  | 'summarized'
  | 'partial_failed'
```

### 4.3 工作记忆

```ts
export interface AiStructuredHistorySummary {
  constraints: string[]
  progress: string[]
  blockers: string[]
  pending: string[]
}

export interface AiWorkingMemory {
  version: 1
  taskId: string
  scope: string
  status: AiCompressionStatus
  taskSummary: string
  historySummary?: AiStructuredHistorySummary
  activeTargetSummary?: string
  latestToolResultSummary?: string
  pendingMutationSummary?: string
  recentErrorSummary?: string
  focusTopic?: string
  lastCompressionReason?: string
  lastCompressionAt: string
  sourceMessageIds: string[]
}
```

字段约束：

- `taskSummary` 必填，不能为空字符串。
- `historySummary` 必须保存为结构化对象，不再保存自由文本长段落。
- `historySummary.constraints / progress / blockers / pending` 每个 section 都允许为空，但整体至少有一个 section 非空时才算有效摘要。
- `sourceMessageIds` 只记录仍被摘要覆盖的消息 ID，不记录全文。
- `status = partial_failed` 时必须同时存在 `lastCompressionReason`。
- 为兼容旧存档，读取阶段允许把历史字符串摘要归一化为 `historySummary.progress[0]`，但新的持久化结果一律写结构化对象。

### 4.4 短期任务记忆

```ts
export interface AiTaskMemoryRecord {
  version: 1
  taskId: string
  scope: string
  input: string
  taskSummary: string
  resultSummary?: string
  terminationReason: string
  routeTargetSnapshot?: unknown
  createdAt: string
  completedAt?: string
}
```

### 4.5 用户偏好记忆

```ts
export interface AiPreferenceMemory {
  version: 1
  scope: string
  previewBeforeWrite?: boolean
  defaultReplyStyle?: 'concise' | 'structured'
  preferredLanguage?: string
  updatedAt: string
}
```

### 4.6 请求组装块

```ts
export interface AiRequestAssembly {
  systemPrompt: string
  contextPrompt: string
  memoryPrompt: string
  recentMessages: Array<{ role: 'user' | 'assistant' | 'system', content: string }>
}
```

这里要注意：

- `memoryPrompt` 来自 working memory，不直接来自原始 conversation。
- `recentMessages` 只保留保护区消息，不包含已被摘要覆盖的中段原文。

## 5. 存储 key 与生命周期

首期建议新增 3 个 key：

- `ai-chat-working-memory:<scope>`
- `ai-chat-task-memory:<scope>`
- `ai-chat-preference-memory:<scope>`

继续保留现有 key：

- `ai-chat-conversation:<scope>`
- `ai-chat-agent-task:<scope>`

生命周期规则：

- `ai-chat-working-memory`
  - 当前任务内有效
  - 任务完成或显式清空后可删除
- `ai-chat-task-memory`
  - 保留最近 `10 ~ 20` 条
  - 超出后按时间淘汰最旧记录
- `ai-chat-preference-memory`
  - 长期保留
  - 用户显式修改时覆盖

版本规则：

- 每个新结构都加 `version: 1`
- 后续改结构时只升本结构版本，不改 conversation 旧结构

## 6. 请求装配顺序

首期请求装配必须固定为下面顺序：

1. `systemPrompt`
2. `contextPrompt`
3. `memoryPrompt`
4. `recentMessages`
5. 当前用户输入

实现要求：

- `contextPrompt` 继续由 `buildAiChatContextSystemPrompt()` 生成。
- `memoryPrompt` 由 `AiWorkingMemory` 渲染，不允许直接把 working memory 原始 JSON 发给模型。
- `recentMessages` 只取头部保护区和尾部保护区。
- 当前用户输入必须保持原文，不得因为压缩被改写。

### 6.1 建议的请求体扩展

建议给 `createChatRequestBody()` 增加内部字段：

```ts
{
  _fastnoteRequestContext: AiChatRequestContext | null
  _fastnoteWorkingMemory?: AiWorkingMemory | null
}
```

随后在 transport 层统一抽离：

- `extractAiChatRequestContext()`
- `extractAiWorkingMemory()`

如果首期不想改 transport body 结构，也可以在 `buildOpenAiMessages()` 之前先把 memory block 转成隐藏 `system` 消息。

## 7. 卫生清理规则

首期卫生清理只做三类动作：

1. 裁旧工具结果
2. 裁旧正文原文
3. 去重历史错误和重复状态播报

建议输出：

```ts
export interface AiHygieneResult {
  trimmedMessages: UIMessage[]
  droppedMessageIds: string[]
  latestToolResultSummary?: string
  latestErrorSummary?: string
}
```

禁止行为：

- 不得改写最新一轮用户消息
- 不得删除待确认写回草案
- 不得删除当前目标 noteId / folderId

### 7.1 首期预算估算落地方式

当前实现已经接入真实 tokenizer，但仍保留 heuristic 回退层，不能继续只按“消息条数 + 文本长度”硬编码触发。

实现要求：

- 新增独立的 `context-budget.ts`
- 统一根据 `systemPrompt + contextPrompt + memoryPrompt + visibleMessages` 估算输入 token
- 使用模型名推断保守的 `maxContextTokens`
- 允许通过前端设置或环境变量显式覆盖上下文窗口
- 对常见 provider / model family 维护一份内置 `model context profile` 表
- OpenAI family 优先走真实 tokenizer 计数
- 其他暂未接入真实 tokenizer 的 family 继续走 profile 对应的 heuristic tokenizer
- 推导：
  - `reservedOutputTokens`
  - `safetyMarginTokens`
  - `dynamicBudgetTokens`
  - `usageRatio`

触发规则建议：

- `usageRatio >= 0.6`
  - 允许进入结构化摘要
- `usageRatio >= 0.85`
  - 标记已逼近预算上沿，应优先卫生清理或强制摘要

补充说明：

- 推荐优先级：`用户设置 contextWindowTokens > 环境变量 VITE_AI_CHAT_CONTEXT_WINDOW_TOKENS > 模型名推断`
- “模型名推断”不应只靠字符串后缀，建议优先匹配内置 profile，例如 `GPT-4.1 / GPT-4o / Claude / Gemini / Qwen / DeepSeek` 等 family，再退化到 `32k / 128k` 这类后缀提示。
- 真实 tokenizer 必须走懒加载，不允许把 `js-tiktoken` rank 静态打进主 chunk。
- 请求前预算判断允许走异步真实 tokenizer；本地同步状态刷新可继续走同步 heuristic 估算，以保证状态更新链路稳定。
- tokenizer 加载失败时必须自动回退到 heuristic 估算，不得因为预算模块失败阻断主对话请求。

## 8. 结构化摘要规则

### 8.1 摘要输入

摘要输入必须是卫生清理后的中段消息，而不是整段会话。

```ts
export interface AiSummaryInput {
  previousSummary?: AiWorkingMemory | null
  compressibleMessages: UIMessage[]
  tailMessages: UIMessage[]
  currentTargetSummary?: string
  pendingMutationSummary?: string
  focusTopic?: string
}
```

### 8.2 摘要输出

```ts
export interface AiSummaryOutput {
  historySummary: AiStructuredHistorySummary
  consumedMessageIds: string[]
}
```

实现要求：

- 摘要器对内返回结构化对象，对外渲染成固定文本模板注入 prompt。
- 固定模板顺序为：`约束 -> 进展 -> 阻塞 -> 待处理`。
- LLM 摘要器必须按上述四段输出；若输出不规范，前端需要做 section 解析和兜底归一化。
- 首期允许 deterministic summarizer 只用启发式把中段消息映射到这些 section，不要求语义完全准确，但要求稳定、可复现。

建议的 prompt 渲染格式：

```md
已压缩的历史消息摘要：
约束：
- ...
进展：
- ...
阻塞：
- ...
待处理：
- ...
```

### 8.3 失败回退

摘要失败时按下面顺序执行：

1. 保留卫生清理结果
2. 不更新 `consumedMessageIds`
3. 把 working memory 标成 `partial_failed`
4. 记录失败原因
5. 继续请求，但只带已清理后的原始历史

### 8.4 请求时缓存与并发去重

为了避免同一轮请求重复调用摘要模型，首期实现应增加两层保护：

- 摘要缓存
  - key 建议由 `summarizerName + taskId + sourceMessageIds` 组成
  - 同一任务、同一压缩区再次请求时优先复用摘要结果
- in-flight 去重
  - 若相同 key 的异步摘要尚未完成，后续请求直接等待同一个 Promise
  - 不允许同一片历史在一次用户操作中并发触发多次摘要调用

补充规则：

- 如果当前 working memory 已经是 `summarized`，且 `sourceMessageIds` 与本轮压缩区完全一致，并且 `lastCompressionAt` 仍在 freshness 窗口内，可直接复用现有 `historySummary`，不必再次请求模型。
- freshness 窗口建议首期默认 `30 ~ 60s`；超过后允许重新摘要，避免长任务中长期复用陈旧摘要。
- 摘要缓存不应永久有效，建议增加 TTL；首期可取 `5min` 左右，超时后重新请求摘要模型。
- 异步 LLM 摘要失败时必须自动回退到 deterministic summarizer，不得阻断主请求链路。

## 9. 触发时机与接入点

### 9.1 用户发送消息前

接入点：

- `sendMessage()`
- `resumeInterruptedTask()`
- `regenerate()`

执行顺序：

1. 读取 conversation、agent task、working memory
2. 计算预算
3. 触发卫生清理或摘要
4. 更新 working memory
5. 再调用 `createChatRequestBody()`

### 9.2 工具结果续跑前

接入点：

- `continueAssistantAfterToolResults()`

这是首期最关键的改动点。当前实现会把 `buildDetailedToolResultsPrompt(results)` 全量塞回 prompt，必须改为：

1. 先把 `results` 转为 `resultsSummary`
2. 仅对最近结果保留必要明细
3. 若历史工具结果过长，先做卫生清理
4. 需要时写入 `latestToolResultSummary`
5. 再生成 follow-up prompt

### 9.3 持久化前

接入点：

- `persistConversation()`
- `persistAgentTask()`

新增：

- `persistWorkingMemory()`
- `persistTaskMemory()`
- `persistPreferenceMemory()`

### 9.4 刷新恢复后

接入点：

- `hydrateConversation()`
- `hydrateAgentTask()`

新增：

- `hydrateWorkingMemory()`

恢复顺序建议：

1. conversation
2. agent task
3. working memory

原因：

- 任务态先恢复，才能判断 working memory 是否仍然归属当前任务
- 若 taskId 不匹配，应丢弃 working memory，而不是强行恢复

### 10.3 最小验收场景

首期至少要覆盖下面 8 个场景：

1. 长对话超过摘要阈值后，生成 working memory，不丢最新用户消息。
2. 长正文读取后，旧正文块被裁剪，但当前目标 noteId 仍保留。
3. 多轮工具调用后，旧工具结果被卫生清理，而不是无上限堆积。
4. 摘要失败时，任务仍可继续，且中段历史未被静默删除。
5. 进入待确认态时，`pendingMutationSummary` 被写入 working memory。
6. 刷新恢复时，taskId 不匹配的 working memory 会被丢弃。
7. 恢复后继续任务时，会优先带 working memory 而不是整段旧历史。
8. 关闭 Agent feature flag 时，不生成 working memory key。

## 11. 研发拆分建议

建议按 4 个任务拆：

1. `T-COMP-001`
   - 实现预算计算和卫生清理
2. `T-COMP-002`
   - 实现 working memory 类型、持久化和恢复
3. `T-COMP-003`
   - 实现请求装配与工具续跑接入
4. `T-COMP-004`
   - 实现摘要失败回退和集成测试

这样拆的原因：

- 先把“无损清理”做出来，系统就已经能明显降噪
- working memory 单独落地后，刷新恢复能力才算真正闭环
- 最后再接摘要失败回退，便于定位问题边界

## 12. 与 Hermes Agent 的收敛和差异

本实现规格借鉴 Hermes 的地方：

- 两段式压缩
- 保守保护尾部
- 摘要预算上限
- 摘要失败禁止静默丢历史

本实现规格没有照搬 Hermes 的地方：

- 不引入 provider cache
- 不引入网关层压缩器
- 不引入多 agent 共享记忆
- 不把网页端会话直接抽象成通用轨迹引擎

首期目标不是做成 Hermes 的等价实现，而是把它的有效压缩经验迁移成适合 FastNote 的网页端内存层。
