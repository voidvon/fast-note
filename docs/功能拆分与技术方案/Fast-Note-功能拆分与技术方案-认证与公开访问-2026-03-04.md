## 3. 功能点状态

### 3.1 F4.1 登录注册与会话生命周期

- 当前状态：已支持。
- 已落地能力：
  - 登录页和注册页都已完成输入校验、提交 loading、错误与成功反馈。
  - 注册成功后会自动登录并返回上一页。
  - 应用启动时会先恢复本地认证用户，再尝试 `authRefresh()` 校验会话。
  - token 失效时会清空本地登录态，避免脏会话残留。
  - 登录态变化后会统一触发会话上下文重建，而不是让各页面各自处理。
- 当前真实边界：
  - 还没有 OAuth、多因子认证或“记住我”配置。
  - 认证失败时当前主要通过页内错误提示展示，没有单独的会话过期中断页。

### 3.2 F4.2 公开用户内容读取与本地隔离缓存

- 当前状态：已支持。
- 已落地能力：
  - 路由守卫会对 `UserHome / UserFolder / UserNote` 预先执行 `ensurePublicNotesReady()`。
  - 公开页页面层仍会在进入和手动刷新时主动拉取一次公开数据，保证页面可自愈。
  - 公开用户资料和公开笔记都已接入错误态与空态展示。
  - 每个公开用户名使用独立的 `UserPublicNotes_<username>` Dexie 库，避免公开缓存互串。
- 当前真实边界：
  - 公开数据初始化仍同时存在“路由守卫预热 + 页面 init 再拉取”两处触发，语义上有重复。
  - 目前没有显式 TTL / LRU 的公开缓存清理策略。

### 3.3 F4.3 公开详情桌面态参数对齐

- 当前状态：已支持。
- 真实代码事实：
  - 桌面公开页现在已经改为 `<NoteDetail :note-id="state.noteUuid" />`。
  - `FolderPage` 也已同步接入 `selected-note-id`，分栏列表和详情选中契约与私有首页桌面模式保持一致。
  - 单测已经覆盖“公开主页列表选择 -> FolderPage -> NoteDetail”的 props 传递链路。
- 当前剩余边界：
  - 移动端公开详情仍主要依赖路由参数，不复用这套桌面分栏装配。

## 5. 真实边界与剩余风险

- 公开数据初始化在守卫与页面层双触发，虽能保证可用，但后续维护时容易产生重复请求或状态心智分裂。
- 公开缓存当前按用户名不断创建独立 Dexie 库，长期使用后的清理策略仍未定义。
- 会话恢复虽然已落地，私有 deferred 恢复也已启用，但当前仍缺少更细的恢复阶段可视化。

## 6. 迁移状态说明

- 迁移结果：已迁入目标架构，但整体完成度仍为“部分支持”。
- 当前迁移口径：
  - 认证主链路已经由 `processes/session` 统一编排，不再是页面直连认证 SDK。
  - 公开页数据初始化已经有 `processes/public-notes` 统一入口，但守卫和页面层仍有双触发问题。
  - 当前真实状态应描述为“主链路已可用，桌面公开详情参数链路已修复，但公开缓存与预热治理仍未完全稳定”。
- 后续仍可继续迁移和补强的点：
  - 收敛公开数据初始化触发点。
  - 持续补公开缓存治理策略。
