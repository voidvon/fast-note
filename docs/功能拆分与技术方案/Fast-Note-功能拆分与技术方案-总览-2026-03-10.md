## 4. 里程碑与发布建议

- 里程碑划分：
  - M1：补齐覆盖策略与确认交互
  - M2：补齐 `metadata` / `user_info` 冲突策略与更细粒度交互语义
  - M3：补自动化测试与登录/同步流程联动验证
- 上线策略：在现有“检测 + 合并 + 共存”能力基础上继续渐进增强，无需推翻当前登录初始化结构
- 灰度/回滚：
  - 若游客态处理出现异常，可暂时保留 scope 切换能力并关闭弹窗决策入口
  - 若合并逻辑回归，可退回“仅共存不迁移”策略

## 5. 跨模块依赖与风险

- 依赖关系：
  - `会话初始化` -> `游客态处理`：决策流程挂在 `use-session-bootstrap` 的登录成功分支
  - `本地数据库` -> `游客态处理`：跨库读写依赖 scope 隔离数据库能力
  - `同步与实时` -> `游客态处理`：合并后的数据仍需走既有同步链路上传到云端
- 关键风险：
  - 当前只支持 `merge` / `coexist` 两个分支，没有 `overwrite`
  - `mergeGuestDataIntoCurrent()` 已补齐 `metadata` / `user_info` 的 guest-only 合并，但同 key / id 冲突仍默认保留登录态版本
  - 游客态资产统计已覆盖 `notes` / `note_files` / `metadata` / `user_info`，后续新增资产表仍需同步扩展
- 需要业务决策的问题：
  - 是否要补“覆盖登录态”入口及二次确认
  - `metadata` / `user_info` 遇到同 key / id 冲突时是否允许覆盖登录态
