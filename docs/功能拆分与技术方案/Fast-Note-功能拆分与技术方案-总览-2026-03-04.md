## 4. 里程碑与发布建议

- 里程碑划分：
  - M1（P0 稳定性）：已完成公开页参数错配修复；剩余重点是补齐删除递归策略和同步失败隔离与重试能力
  - M2（P1 体验）：完善搜索、导航恢复边界、手动同步可观测性
  - M3（P1 安全）：PIN / 生物识别提示、设备差异说明与异常提示优化
- 上线策略：先灰度内部用户（同步与公开路径），再全量
- 灰度/回滚：
  - 同步异常时关闭自动同步入口，仅保留本地编辑
  - 公开路由异常时回退到只保留 `/:username` 首页入口

## 5. 跨模块依赖与风险

- 依赖关系：
  - `认证` -> `同步与实时`（登录态是同步前置）
  - `笔记编辑` -> `附件`（内容结构内嵌 file-upload）
  - `路由导航` -> `公开访问`（用户名路由触发公开数据初始化）
- 关键风险：
  - 公开数据初始化仍存在“守卫预热 + 页面 init”双触发，重复请求成本和状态心智仍待收敛（`src/app/router/index.ts:13`, `src/pages/user-public-notes/ui/user-public-notes-page.vue:57`）
  - 私有详情 deferred 恢复已启用，但当前应用级阻塞态仍缺更细的 loading / 错误可视化（`src/processes/navigation/model/use-last-visited-route.ts:15`, `src/App.vue:22`）
  - 同步批次当前仍是逐条串行且一条失败即中断，弱网下会放大人工重试成本（`src/processes/sync-notes/model/use-sync-notes.ts:582`, `src/processes/sync-notes/model/use-sync-notes.ts:705`）
  - 生物识别凭据仍是设备本地能力，不随账号跨设备同步，需持续做好产品提示（`src/features/note-lock/model/use-note-lock.ts:840`）
  - `useDeviceType` 全局永久监听 `resize`，长期运行下可能产生资源开销（`src/shared/lib/device/use-device-type.ts:10`）
- 需要业务决策的问题：
  - 扩展系统入口是否下线（代码已标注废弃，`src/features/extension-manager/model/use-extensions.ts:15`）
  - 公开分享权限是否需要“范围级别”而非二值开关

### 8.3 里程碑与发布建议增补

- 建议里程碑：`M1` 完成 P0 完整性约束补硬；`M2` 完成 P1 本地落盘兜底与观测增强。
- 上线策略：先灰度给多账号/高频同步用户，观察误创建和同步失败率后再全量。
- 灰度/回滚：
  - 开发版按新结构直改，不保留旧 key/旧库兼容分支。
  - 若路由恢复时序影响体验，可暂时保留 `/home` 立即恢复，继续保留详情页防误创建保护。

### 8.4 跨模块依赖与风险增补

- 依赖关系：
  - `同步与实时` -> `路由与导航`：私有详情路由恢复时机需要与现有桌面刷新恢复逻辑对齐。
  - `同步与实时` -> `笔记编辑与组织`：`NoteDetail` 新建/更新判定需要保持移动端新建链路兼容。
  - `同步与实时` -> `测试发布`：P0 任务必须补自动化验证，避免引入误创建回归。
- 关键风险：
  - 本地库按用户隔离会改变当前开发环境本地数据布局，需要接受开发期缓存重建成本。
  - 私有路由延后恢复可能带来一次轻微加载感知，需要 UI 兜底。
  - 详情页 miss 防误创建若文案不清晰，用户可能误判为“笔记丢失”。
- 需要业务决策的问题：
  - 本地缓存补齐是自动触发还是需要用户确认。
  - 切账号后是否提供“清理旧账号本地缓存”入口。
