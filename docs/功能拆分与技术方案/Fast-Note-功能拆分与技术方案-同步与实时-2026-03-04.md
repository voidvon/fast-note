  - `部分支持`：按用户作用域的双向增量同步已经落地，缓存补齐和缺失私有笔记修复也已接入。

  - `部分支持`：Realtime 已作为在线补丁流接入，但用户面板仍主要展示结果，而不是阶段状态。
  - `部分支持`：同步批次仍是一条失败即中断，尚未补“失败隔离 + 部分成功继续执行”。

## 3. 当前实现拆解

### 3.1 F3.1 双向增量同步与游标推进

#### 3.1.1 目标与边界

- 用户场景：登录用户在多设备编辑后，可通过自动或手动同步获得一致内容。
- 范围内：增量拉取、游标推进、用户作用域隔离、缓存补齐、缺失私有笔记修复。
- 范围外：协作编辑冲突合并、附件安全策略细节。

#### 3.1.2 当前实现

- 同步游标
  - 统一使用 `pocketbaseUpdated` 前缀。
  - 游标存储 key 已通过 `createScopedStorageKey()` 绑定当前用户。
- 增量模型
  - 本地侧读取 `updated > cursor` 的记录。
  - 云端侧读取 `updated > cursor && user_id = auth.id` 的记录。
  - 同步操作按 `updated` 升序串行执行，成功一条推进一条游标。
- 缓存补齐
  - “空库 + 非初始游标”时会自动重置游标并进入补齐模式。
  - 私有详情页本地 miss 且游标非初始值时，可触发补齐式静默同步。
- 附件链路
  - 同步前会把正文中的本地 hash 替换为临时标识符。
  - 服务端返回真实文件名后，再把正文回填并补一次 update。

#### 3.1.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 当前同步主模型已经稳定落在 `processes/sync-notes + shared/api/pocketbase + shared/lib/storage`。
  - 核心问题不再是“有没有增量同步”，而是失败恢复、阶段可观测性和边界场景兜底仍不完整。

#### 3.1.4 剩余风险与建议

- 风险：当前批次仍是一条失败即中断，弱网或单条附件异常会阻断后续记录。
- 风险：局部断裂场景主要靠现有 repair 条件兜底，仍不是完整对账式同步。
- 建议：后续优先补失败记录聚合和用户可见的补齐模式提示，不建议回退到全量同步。

#### 3.1.5 验证建议

- 冒烟：A 端编辑 + B 端编辑 + 登录态恢复 + 手动同步后，确认数据最终一致。
- 冒烟：制造“空库 + 有游标”与“私有详情本地 miss”两类场景，确认都会进入补齐流程。
- 回归关注点：未登录静默同步必须直接返回 `null`，不能污染同步状态。

### 3.2 F3.2 冲突裁决与删除保留策略

#### 3.2.1 目标与边界

- 用户场景：同一笔记在多端被修改或删除时，系统给出可预测的最终结果。
- 范围内：时间戳冲突处理、30 天删除保留、本地/云端清理动作。
- 范围外：协作编辑场景下的段落级合并。

#### 3.2.2 当前实现

- 冲突裁决
  - 继续采用 `updated` 单因子裁决，新者覆盖旧者。
  - 本地与云端都存在时，按本地时间与云端时间比较决定上传或下载。
- 删除保留
  - 已删除记录会参与 30 天窗口判断。
  - 30 天内保留为可恢复墓碑；超窗后进入本地清理或云端删除态同步。
- 规则沉淀
  - 领域层已经存在 `NOTE_DELETION_RETENTION_DAYS = 30` 和 `isDeletedNoteRetained()`。
  - 但页面层仍有局部直接写 `Date.now() - 30 天` 的实现，尚未完全收口。

#### 3.2.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 冲突裁决与删除保留主规则都已存在并在运行。
  - 当前缺口主要是规则落点仍不够统一，以及失败反馈不够完整。

#### 3.2.4 剩余风险与建议

- 风险：客户端时间漂移仍会影响 `updated` 裁决和删除窗口判断。
- 风险：展示层与同步层对 30 天窗口仍有重复实现，后续容易产生边界漂移。
- 建议：后续优先把页面层删除窗口判断统一到领域 helper，而不是改同步主规则。

#### 3.2.5 验证建议

- 冒烟：本地软删 -> 同步 -> 跨端恢复或清理，确认窗口内外行为一致。
- 回归关注点：29/30/31 天边界不能出现展示与同步结果不一致。

### 3.3 F3.3 Realtime 与手动同步操作面板

#### 3.3.1 目标与边界

- 用户场景：登录后自动建立会话级 Realtime 连接，并在需要时手动触发同步查看结果。
- 范围内：Realtime 生命周期、登录后自动静默同步、手动同步按钮、结果展示。
- 范围外：消息推送通知体系、后台常驻同步。

#### 3.3.2 当前实现

- 会话启动
  - 登录态恢复后会先初始化 `PocketBaseRealtimeService`。
  - 若未连接，会先连接 Realtime，再执行一次静默同步。
- Realtime
  - 订阅限定到当前用户的 `notes` 集合。
  - `create/update/delete` 会直接驱动本地 `addNote/updateNote/deleteNote`。
  - 断线后采用指数退避重连；登出后会主动断开。
- 手动同步
  - 用户面板保留显式同步入口。
  - 当前会展示上传、下载、删除数量与错误信息。
  - `syncStatus` 已有 `currentStep/progress/error/lastSyncTime` 字段，但 UI 还没完整消费。

#### 3.3.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 自动入口和手动入口都已存在，Realtime 也已融入主链路。
  - 当前更像“结果可见”，不是“过程可见”，所以仍然只能记为部分支持。

#### 3.3.4 剩余风险与建议

- 风险：用户现在只能较晚看到最终错误，重连中、处理中和补齐模式都不够透明。
- 风险：Realtime 是在线补丁流，但 UI 还没有清楚区分“Realtime 已连接”和“历史同步已完成”。
- 建议：后续优先把 `syncStatus.currentStep/progress` 和 Realtime 连接态映射到用户面板。

#### 3.3.5 验证建议

- 冒烟：登录 -> 自动同步 -> 手动同步 -> 退出登录断连。
- 回归关注点：Realtime 失败时不应阻塞手动同步；退出登录必须断开连接，避免串账号。
