## 2. 模块目标与现状

- 业务目标：
  - 用户在列表中不仅能识别“这篇 note 已启用锁”，还要能识别“这篇 note 在当前设备上是否仍处于临时已解锁状态”。
- 当前状态（已支持/部分支持/未支持）：已支持
- 当前判断：
  - `已支持`：`widgets/note-list` 已经支持 `locked / unlocked / placeholder` 三态渲染。
  - `已支持`：`features/note-lock` 已经把列表展示态抽成独立 hook，不需要页面自己订阅解锁事件。
  - `已支持`：PIN 解锁、生物识别解锁、立即重锁和会话自然过期都能驱动列表刷新。
  - `部分支持`：当前自动化覆盖到组件、hook、首页混合列表，但还没有真实跨页时间流逝 e2e。
- 架构意图：
  - `features/note-lock` 负责会话真相、事件分发和列表三态映射。
  - `widgets/note-list` 只消费状态 map 并渲染图标，不直接访问会话存储。
  - `pages/note-detail` 与锁管理面板只通过 feature API 触发解锁或重锁，不再承担列表刷新的编排责任。

## 3. 当前实现拆解

### 3.1 F1.1 列表项会话感知三态渲染

#### 3.1.1 目标与边界

- 用户场景：
  - 用户在详情页输入正确 PIN 或生物识别成功后返回列表，希望立刻看到对应 note 变成解锁图标。
  - 用户在桌面分栏中直接在右侧详情区解锁，左侧列表也应同步切换图标。
- 范围内：
  - note 行三态展示。
  - `is_locked=1` 且当前设备存在有效会话时显示解锁图标。
  - `is_locked=1` 且没有有效会话或处于冷却时显示锁住图标。
  - `is_locked=0` 或缺失锁字段时保留占位。
- 范围外：
  - 锁图标点击交互。
  - 非 note 条目新增解锁语义。

#### 3.1.2 当前实现

- `features/note-lock`
  - `useNoteLockIndicatorState()` 会扁平化当前列表可见 note，并对每个已锁 note 调用 `getLockViewState()`。
  - 当 `viewState === 'unlocked'` 时映射为 `unlocked`，其余已锁 note 一律回退为 `locked`，未锁定 note 则为 `placeholder`。
- `widgets/note-list`
  - `note-list.vue` 统一拿到状态 map，再将结果下发给 `NoteListItem`。
  - `note-list-item.vue` 只关心渲染三态，不关心会话来源，也不直接读取 Dexie。
- `entities/shared`
  - 没有新增任何 `Note` 字段，列表三态仍是 UI 派生态，不写回笔记实体。

#### 3.1.3 迁移状态

- 状态：已支持
- 当前结论：
  - 列表三态展示已经迁入 `features -> widgets` 的目标路径。
  - 页面层不再自己拼接“是否已解锁”的判断逻辑。

#### 3.1.4 剩余风险与建议

- 风险：当前 `cooldown` 会被当作锁住态展示，语义正确，但列表层不会额外提示“为什么是锁住”。
- 建议：如果后续要补锁状态说明，仍应从 `useNoteLock` 的视图语义继续向上暴露，不要在列表组件里重新解释冷却状态。

#### 3.1.5 验证建议

- 冒烟：解锁一篇已锁 note，确认列表图标立刻从 `lockClosed` 变为 `lockOpen`。
- 回归关注点：未锁定和旧数据 note 仍只保留占位，不会误显示解锁图标。

### 3.2 F1.2 解锁事件、重锁动作与会话到期刷新

#### 3.2.1 目标与边界

- 用户场景：
  - 用户刚在详情页解锁某篇 note，列表应立即同步。
  - 用户点击“立即重新锁定”后，列表应立即回到锁住态。
  - 用户停留在列表页不操作，等到会话过期后，图标应自动恢复锁住态。
- 范围内：
  - 列表图标刷新触发源。
  - PIN、生物识别、重锁、过期四类事件联动。
  - 移动返回列表与桌面分栏的最小兜底刷新。
- 范围外：
  - 锁详情页整体流程重构。
  - 轮询式全局 store。

#### 3.2.2 当前实现

- `features/note-lock`
  - `verifyPin()` 成功后会落本地会话并派发 `verified` 事件。
  - `tryBiometricUnlock()` 成功后会派发 `biometric` 事件。
  - `relock()` 走统一清会话链路，列表刷新不依赖页面手写事件。
  - `useNoteLockIndicatorState()` 只维护一个最近过期时间定时器，到期时重新计算可见 note 的锁展示态。
  - hook 还监听 `visibilitychange` 和 `focus`，在返回页面或重新聚焦时做兜底刷新。
- `pages / ui`
  - `note-lock-manage-modal.vue` 的“立即重新锁定”已经走 `noteLock.relock()`，因此能自动接入同一条刷新链路。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 会话写入、事件派发、定时回锁和列表重算已经闭环。
  - 当前实现不是轮询补丁，而是基于会话事件和过期时间的增量刷新。

#### 3.2.4 剩余风险与建议

- 风险：当前主要缺口是没有更贴近真实页面切换和时间流逝的 e2e，而不是核心业务能力缺失。
- 建议：后续若继续补测试，优先补跨页返回和短 TTL 场景，不建议引入更重的全局刷新机制。

#### 3.2.5 验证建议

- 冒烟：详情页解锁后回到列表，确认图标立即变为 `lockOpen`。
- 冒烟：锁管理面板点击“立即重新锁定”，确认列表图标立即切回 `lockClosed`。
- 冒烟：缩短 TTL 或用 mock 等待会话过期，确认列表无需手动刷新即回锁。

### 3.3 F1.3 自动化覆盖与剩余缺口

#### 3.3.1 目标与边界

- 用户场景：
  - 该能力虽然只改图标，但依赖本地会话、事件和定时器，需要自动化防止跨模块回退。
- 范围内：
  - 组件单测。
  - hook 单测。
  - 首页混合列表集成测试。
- 范围外：
  - e2e。
  - 真实系统时间和真实 WebAuthn 联调测试。

#### 3.3.2 当前实现

- `tests/unit/components/note-list-item-lock-indicator.spec.ts`
  - 已覆盖锁住、解锁、占位三态。
- `tests/unit/hooks/useNoteLockIndicatorState.spec.ts`
  - 已覆盖可见 note 状态映射、会话事件重算和解锁到期回锁。
- `tests/integration/home/note-list-lock-indicator.spec.ts`
  - 已覆盖首页混合列表中三态图标的数量、顺序和深色模式展示。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - 当前自动化已经覆盖三态渲染和核心刷新机制。
  - 剩余空白主要在真实跨页链路，而不是组件或 hook 层没有测试。

#### 3.3.4 剩余风险与建议

- 风险：文件夹页、未来新增列表入口和真实跨页时间流逝场景，仍主要依赖共享组件复用和人工冒烟。
- 建议：如果下一步继续迁移锁能力，优先补 e2e 或更贴近真实页面装配的集成验证。

#### 3.3.5 最小人工冒烟建议

- 首页混合列表：
  - 同时存在锁住、已解锁、未锁定 note，确认三态顺序正确。
- 桌面分栏：
  - 右侧详情页解锁后，左侧列表图标同步刷新。
- 停留等待：
  - 列表停留到会话过期，确认图标自动回锁。
