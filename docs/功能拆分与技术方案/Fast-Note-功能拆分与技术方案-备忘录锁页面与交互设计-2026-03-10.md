## 2. 迁移结论

- 当前状态：已支持。
- 交互结论：锁能力的实际 UI 已经收口为 3 个界面：
  - 首次锁定 / 已有 PIN 锁定弹层：`NoteLockSetupModal`
  - 已锁笔记的设置弹层：`NoteLockManageModal`
  - 详情页解锁面板：`NoteUnlockPanel`
- 需要特别说明的真实现状：
  - 首次设置全局 PIN 当前只有一个 6 位输入框，没有“二次确认 PIN”字段。
  - 已有全局 PIN 时仍会打开同一个 setup modal，但标题会切到“锁定这篇备忘录”，不再要求输入 PIN。
  - 解锁面板当前文案仍使用“备忘录密码”表述，尚未完全统一成“全局 PIN”术语。

## 3. 当前交互原则

- 用户只需要理解两个核心概念：
  - 这篇备忘录是否上锁。
  - 当前账号是否已经设置全局 PIN。
- 生物识别始终是当前设备的快捷解锁方式，不是独立密码体系。
- 页面与弹层只负责表达当前动作，不直接解释底层 Dexie / PocketBase 细节。
- PC 与移动端共用同一套业务流程，差异主要体现在 modal 呈现方式。

## 4. 当前界面与流程

### 4.1 锁定弹层 `NoteLockSetupModal`

- 真实触发入口：
  - 用户在 [note-more.vue](/root/fast-note/src/widgets/note-more/ui/note-more.vue) 点击“锁定”。
- 实际标题：
  - 无全局 PIN：`设置全局 PIN`
  - 已有全局 PIN：`锁定这篇备忘录`
- 实际表单：
  - 无全局 PIN 时：
    - 1 个 `6 位数字` 输入框
    - 1 个“生物识别快捷解锁”复选开关
  - 已有全局 PIN 时：
    - 不展示 PIN 输入框
    - 仅保留“生物识别快捷解锁”开关
- 实际按钮：
  - 次按钮：`取消`
  - 主按钮：`确认`
- 关键代码证据：
  - [note-lock-setup-modal.vue:114](/root/fast-note/src/features/note-lock/ui/note-lock-setup-modal.vue#L114) 标题在“设置全局 PIN / 锁定这篇备忘录”之间切换。
  - [note-lock-setup-modal.vue:127](/root/fast-note/src/features/note-lock/ui/note-lock-setup-modal.vue#L127) 只有在 `!hasGlobalPin` 时才展示 PIN 输入。

### 4.3 详情页解锁面板 `NoteUnlockPanel`

- 真实触发条件：

## 5. PC / 移动端布局现状

### 5.1 PC

- setup / manage modal 在桌面端不走底部断点 sheet，而是居中弹层。
- 解锁面板直接占据右侧详情区域，不遮挡左侧列表上下文。
- `立即重新锁定` 和解锁成功后列表锁图标联动，符合桌面分栏心智。

### 5.2 移动端

- setup / manage modal 使用 `IonModal` 的底部 sheet 断点。
- 解锁面板仍在详情页主内容区域显示，不额外开新页面。
- 错误与冷却信息保留在输入框下方，不走短暂 toast。

## 6. 当前文案口径

- 已落地的核心文案：
  - `设置全局 PIN`
  - `锁定这篇备忘录`
  - `PIN 设置`
  - `修改全局 PIN`
  - `立即重新锁定`
  - `关闭这篇备忘录的锁`
  - `尝试生物识别`
  - `解锁`
- 仍建议后续统一的点：
  - 解锁面板里的“备忘录密码 / 输入密码”最好统一为“全局 PIN / 输入全局 PIN”。
  - setup modal 的确认按钮目前统一写作 `确认`，若要强化操作心智，可进一步区分“设置并锁定 / 锁定这篇备忘录”。

## 7. 当前状态机

| 状态 | 含义 | 当前 UI | 可执行动作 | 结束条件 |
| ---- | ---- | ------- | ---------- | -------- |
| `unlocked` | note 未锁或当前会话有效 | 编辑器可见 | 编辑、打开锁设置 | 手动重锁、会话过期 |
| `locked` | note 已锁且无会话 | `NoteUnlockPanel` | 输入 PIN、尝试生物识别 | PIN 成功 / 生物识别成功 |
| `unlocking` | 正在校验 | 提交按钮 loading、输入禁用 | 等待结果 | 成功 / 失败 |
| `cooldown` | PIN 连续失败过多 | 错误区显示倒计时 | 等待 | 倒计时结束 |

## 8. 仍需注意的交互差异

- 首次设置全局 PIN 当前没有“确认 PIN”二次输入，这和早期设计稿不同，后续若要加强防误输，需要改真实实现而不是沿用旧文档。
- 锁管理弹层当前没有单独的“影响全部已锁备忘录”提示文案，只是通过“修改全局 PIN”动作名表达范围。
- setup modal 对“已存在全局 PIN 的锁定动作”仍沿用同一个 modal，而不是完全无弹层直接锁定。

## 9. 本轮口径结论

- 不再按“每篇备忘录独立密码”设计页面。
- 当前真实交互已经建立在“全局 PIN + 当前设备生物识别快捷解锁 + 当前 note 本地会话”之上。
- 这篇文档现阶段应作为“当前实现说明”使用，而不是继续作为未落地设计稿。
