## 4. 里程碑与发布建议

- 里程碑状态：
  - M1：本地数据模型从“note 自带 PIN 字段”切到 `security_settings + device_security_state + note_unlock_sessions`，已完成。
  - M2：`useNoteLock` 收口全局 PIN、note 会话、生物识别与锁管理主链路，已完成。
  - M3：`users` 表字段接入账号级 PIN 同步，已完成最小闭环。
  - M4：多端一致性、云端字段可观测性与自动化补强，部分完成。
- 上线策略：
  - 当前已属于现网真实实现说明，后续应优先补测试和异常观测，而不是继续扩散方案稿描述。
  - 若继续增强，应优先补测试和云端异常可观测性，而不是回退安全模型。
- 回滚边界：
  - 若后续锁云端同步出现问题，可先回退 `users` PIN 同步相关逻辑，保留本地全局 PIN + note 会话主链路。

## 5. 跨模块依赖与风险

- 依赖关系：
  - 依赖 `src/shared/lib/storage/dexie.ts` 提供 `security_settings`、`device_security_state`、`note_unlock_sessions` 三张本地表。
  - 依赖 `src/shared/api/pocketbase/users.ts` 承接 `note_lock_pin_salt/hash/version` 云端字段。
  - 依赖 `src/pages/note-detail/ui/note-detail-page.vue` 与 `src/widgets/note-more/ui/note-more.vue` 作为详情页和更多菜单入口。
  - 依赖 `src/shared/lib/security/use-web-authn.ts` 提供设备生物识别能力。
- 当前风险：
  - 云端同步依赖 `users` 表字段已正确开放；缺字段时当前会明确报错，但仍是部署层风险。
  - 修改全局 PIN 不会主动清理所有既有 note 本地会话，这属于当前保留边界。
  - 当前缺少更完整的跨设备、多账号切换和真实云端异常链路自动化验证。
- 已明确保持的不变量：
  - `note.is_locked` 仍只表达 note 是否需要解锁，不再承载 PIN 本体。
  - 生物识别仍是设备本地快捷解锁能力，不进入云端。
  - 锁能力仍是门禁，不是内容加密。
