## 3. 当前实现拆解

### 3.1 F1.1 路由编排与公开入口初始化

#### 3.1.1 目标与边界

- 用户场景：用户通过 URL 直接进入私有首页、私有详情、公开主页、公开文件夹或公开笔记。
- 范围内：路由表、路由守卫、公开用户首次初始化、公开页错误与刷新入口。
- 范围外：公开笔记的数据同步算法本身。

#### 3.1.2 当前实现

- `app/router`
  - `routes.ts` 已注册 `/home`、`/n/:id`、`/f/:pathMatch(.*)*`、`/deleted`、`/:username`、`/:username/f/:pathMatch(.*)*`、`/:username/n/:noteId` 等核心入口。
  - `index.ts` 的 `beforeEach` 会先处理桌面重定向，再执行公开路由预热。
- `processes/public-notes`
  - `ensurePublicNotesRouteReady(to)` 只对 `UserHome`、`UserFolder`、`UserNote` 三类公开路由执行初始化。
  - `ensurePublicNotesReady(username)` 会初始化公开用户缓存并同步公开笔记列表，且通过 `initializedUsers` 避免重复初始化。
- `pages/user-public-notes`
  - 页面本身已有加载、错误、空态和刷新交互。
  - 桌面布局下会把文件夹列表和详情并排展示。

#### 3.1.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 路由装配和公开入口初始化主体已经迁入目标架构：`app/router + processes/public-notes + pages/user-public-notes`。
  - 公开入口并不是“未实现”，但当前也不能写成完全稳定，因为守卫错误处理和重复预热仍有真实成本。

#### 3.1.4 剩余风险与建议

- 风险：路由守卫对公开数据初始化失败仍只做 `console.error`，缺少统一日志、埋点或用户可见降级说明。
- 风险：守卫和页面都可能触发公开数据准备，存在重复请求成本。
- 建议：这一块后续优先补守卫可观测性与预热去重，而不是改路由表结构。

#### 3.1.5 验证建议

- 冒烟：访问 `/:username`、`/:username/f/*`、`/:username/n/:noteId`，确认初始化、错误态与刷新入口仍可用。
- 回归关注点：私有 `/home`、`/f/*`、`/n/:id`、`/deleted` 不应被公开路由初始化误伤。

### 3.2 F1.2 桌面重定向与智能返回

#### 3.2.1 目标与边界

- 用户场景：桌面端使用双栏布局时，详情深链进入需要统一回到 `/home`；用户返回时优先回到真实上一页。
- 范围内：桌面重定向、导航历史、智能返回、文件夹返回时的页面状态恢复。
- 范围外：详情页具体编辑与保存行为。

#### 3.2.2 当前实现

- 桌面重定向
  - `shouldRedirectDesktopNoteRoute(path, viewportWidth)` 以 `viewportWidth >= 640` 判定桌面端。
  - 命中桌面私有 `/n/*` 或 `/f/*` 时，守卫直接返回 `/home`。
- 导航历史
  - `useNavigationHistory()` 统一保存业务历史，最多保留 15 条。
  - 若历史只有一条且是“孤儿详情页”，跳回首页或文件夹时会替换成可用 fallback 历史。
- 智能返回
  - 详情返回优先使用真实上一页，否则回退到父文件夹或 `/home`。
  - 文件夹返回会按顶层/子层级计算 fallback。
- 页面状态恢复
  - `useRouteStateRestore()` 会监听 `popstate`，把文件夹页进入模式区分为 `restore` 或 `reset`。
  - `registerRouterDependencies(router)` 会在应用启动时把导航历史和状态恢复绑定到路由实例。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 桌面重定向、导航历史、智能返回与文件夹页状态恢复已经形成明确的 `app -> processes/navigation -> pages` 分层。
  - 这一块的主要问题已经不是“能力缺失”，而是阈值统一和长期可观测性。

#### 3.2.4 剩余风险与建议

- 风险：桌面阈值当前仍固定写死为 `640`，尚未与 `useDeviceType()` 统一收口。
- 风险：导航历史与智能返回目前缺少更明确的诊断日志，边界问题仍偏向手工复现。
- 建议：若继续迁移这一块，应优先统一设备阈值来源，再考虑补调试开关。

#### 3.2.5 验证建议

- 冒烟：桌面端访问私有 `/n/:id` 或 `/f/*`，确认会被重定向到 `/home`。
- 冒烟：列表 -> 详情 -> 返回、文件夹多层前进/回退，确认返回优先命中真实上一页。
- 回归关注点：登录/注册页不应污染业务导航历史。

### 3.3 F1.3 全局搜索与最后访问路由恢复

#### 3.3.1 目标与边界

- 用户场景：用户输入关键词搜索笔记，并在重启或登录态切换后恢复到上次业务页。
- 范围内：搜索输入防抖、中文输入法兼容、树形递归搜索、最后访问路由保存与恢复。
- 范围外：全文索引、远端搜索、复杂排序推荐。

#### 3.3.2 当前实现

- `features/global-search`
  - 搜索 UI 已迁入 feature 层，不再由页面直连实现。
  - 搜索执行使用 `300ms` 防抖。
  - 已处理 `compositionstart/compositionend`，避免中文输入法合成阶段误触发搜索。
- `entities/note`
  - `searchNotesByParentId()` 支持在当前目录树内递归匹配标题和正文。
- `processes/navigation`
  - 最后访问路由会排除登录/注册后保存。
  - 恢复模式已经拆成 `all / immediate / deferred`。
  - 当前 `/n/:id`（排除 `/n/0`）会进入 deferred，而 `/home`、公开路由与新建态仍保持 immediate。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - 全局搜索已经基本迁完，当前属于“可用并已归位”的状态。
  - 最后访问路由的保存与作用域隔离也已经落地。
  - 私有详情延迟恢复已经落地，当前剩余问题更多转向加载可见性和搜索体验细化。

#### 3.3.4 剩余风险与建议

- 风险：搜索结果当前更偏“可用结果列表”，还没有标题优先级和命中摘要这类体验优化。
- 风险：私有详情刷新期间当前只有应用级阻塞态，缺少更明确的恢复阶段反馈。
- 建议：这块后续优先级应放在恢复阶段反馈和搜索体验优化，而不是再改路由分类。

#### 3.3.5 验证建议

- 冒烟：首页搜索中文关键词，确认输入法合成阶段不误触发，停止输入后约 `300ms` 返回结果。
- 冒烟：刷新后恢复上次业务页，确认登录/注册页不会污染最后访问路由。
- 回归关注点：用户显式访问的新 URL 不能被历史恢复强制覆盖。
