- 业务目标：让“新建文件夹”和“详情自动保存”两条高频编辑链路具备稳定输入体验和轻量状态反馈，减少键盘闪断、成功提示打扰和失败不可感知问题。
- 当前状态（已支持/部分支持/未支持）：已支持
- 当前判断：
  - `已支持`：首页与文件夹页的新建文件夹弹窗都已接入 `did-present + focus()` 焦点兜底。
  - `已支持`：详情页保存中状态已经沉到 `features/note-save`，页面只负责呈现 spinner。
  - `已支持`：本地保存失败与同步失败都统一走顶部错误提示。
  - `范围外`：重命名弹窗仍是 `alertController.create()` 默认输入框，尚未复用同一套焦点兜底。
- 关键证据：

  - `E6` [src/features/note-save/model/use-note-save.ts:203]：同步失败统一通过 `presentTopError()` 提示。
  - `E7` [src/features/note-save/model/use-note-save.ts:209]：本地保存失败统一通过 `presentTopError()` 提示。

## 3. 当前实现拆解

### 3.1 F2.5 新建文件夹弹窗输入聚焦稳定化

#### 3.1.1 目标与边界

- 用户场景：用户在首页或文件夹页点击“新建文件夹”后，弹窗出现时输入框应自动获得焦点，并尽量避免键盘闪断。
- 范围内：首页和文件夹页的 `IonAlert` 聚焦治理。
- 范围外：重命名弹窗、`IonModal` 替代方案、原生键盘插件层面的兼容治理。

#### 3.1.2 当前实现

- `pages/home`
  - `focusFolderAlertInput()` 会在 `did-present` 事件触发后延迟 50ms 查询 `input.alert-input` 并执行 `focus()`。
  - 聚焦后会把光标移动到输入末尾，避免覆盖已有文本。
- `pages/folder`
  - 复用与首页相同的焦点兜底函数和 `IonAlert` 事件接入方式。
- 交互策略
  - 目前仍保留 `IonAlert`，没有改成自定义 modal。
  - 兜底失败时仍保留用户手动点击输入框的自然退路。

#### 3.1.3 迁移状态

- 状态：已支持
- 当前结论：
  - 本次范围内的两条主路径已经稳定接入焦点兜底。
  - 重命名弹窗未统一不应反向拉低本模块状态，因为它明确属于范围外事项。

#### 3.1.4 剩余风险与建议

- 风险：Ionic overlay 焦点时序若变动，当前 `50ms` 延迟参数可能需要重调。
- 建议：后续若继续迁移这一块，应提炼一个可复用的 overlay focus helper，优先覆盖重命名、移动等文本输入型弹窗。

#### 3.1.5 验证建议

- 冒烟：首页首次打开、关闭后二次打开、文件夹页打开三条路径，确认输入框都能自动获得焦点。
- 回归关注点：弹窗二次打开时仍能重新聚焦，不出现键盘闪断。

### 3.2 F2.6 详情保存状态顶部可视化与错误提示统一

#### 3.2.1 目标与边界

- 用户场景：用户编辑笔记触发非静默保存时，希望看到轻量“保存中”反馈；保存成功时不被打扰；保存失败时得到清晰错误提示。
- 范围内：`use-note-save` 保存状态、`note-detail-page` 头部 spinner、顶部错误提示。
- 范围外：保存重试按钮、全局通知中心、静默保存多阶段进度拆分。

#### 3.2.2 当前实现

- `features/note-save`
  - 以 `isSaving` 维护非静默保存中的 feature 级编排态。
  - 非静默保存开始时置为 `true`，在 `finally` 中统一回收。
  - 本地保存成功后会继续触发一次静默同步。
  - 同步失败时提示“同步失败，请检查网络连接”。
  - 本地保存失败时提示“保存失败，请重试”。
- `pages/note-detail`
  - `presentTopError()` 负责展示顶部 danger toast。
  - 头部更多按钮左侧通过 `IonSpinner` 呈现保存中状态。
- 交互语义
  - 非静默保存显示 spinner。
  - 静默保存不显示 spinner，避免桌面端频繁切换时闪动。
  - 保存成功不再弹成功提示，只保留失败反馈。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 保存反馈核心状态已经从页面临时变量迁到 `features/note-save`。
  - 页面层现在只负责呈现 spinner 和 toast，不再承载保存规则本身。

#### 3.2.4 剩余风险与建议

- 风险：当前只区分“保存失败”和“同步失败”，还没有更细的“本地已保存、云端待重试”状态表达。
- 风险：顶部错误提示目前仍是页面内 helper，没有继续抽成跨页面统一能力。
- 建议：后续若要继续迁移，应优先补更细的状态文案与复用层，而不是回退到页面内保存状态管理。

#### 3.2.5 验证建议

- 冒烟：编辑已有笔记、断网同步失败、桌面切换笔记三种路径，确认 spinner 与顶部错误提示行为一致。
- 回归关注点：异常分支下 spinner 必须在 `finally` 正常关闭，且本地已保存内容不能丢失。
