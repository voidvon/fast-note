# 同步完整性保障技术方案（fastnote）

## 1. 文档信息

- 模块名称：同步完整性保障
- 输出日期：2026-03-09
- 输出文件：`docs/功能拆分与技术方案/fastnote-功能拆分与技术方案-同步完整性保障-2026-03-09.md`

## 2. 模块目标与现状

- 业务目标：在保持 offline-first 与增量同步模型不变的前提下，把“同步完整性依赖的前置条件”沉到当前实现中，降低串账号、误创建、旧游标配空库、离开页丢写等风险。
- 当前状态（已支持/部分支持/未支持）：部分支持
- 当前判断：
  - `已支持`：同步游标、本地 Dexie 库、最后访问路由 key 已按用户作用域隔离。
  - `已支持`：非新建态私有详情在本地 miss 时不会误走新建保存链路。
  - `部分支持`：缓存健康检查与缺失 note 补齐已经接入，但用户侧反馈仍偏弱。
  - `已支持`：私有路由恢复已经按 `immediate/deferred` 双阶段生效，私有 `/n/:id` 会等待会话同步上下文 ready 后再恢复。
  - `部分支持`：离开页本地落盘兜底已覆盖详情页主链路，但仍属于 best effort。
- 关键证据：
  - `E1` [src/processes/sync-notes/model/use-sync-notes.ts:26]：同步游标 key 通过 `createScopedStorageKey()` 按用户作用域生成。

  - `E3` [src/processes/navigation/model/use-last-visited-route.ts:9]：最后访问路由 key 已按用户作用域生成。
  - `E4` [src/processes/session/model/prepare-session-context.ts:4]：会话切换时会重新初始化数据库与笔记上下文。
  - `E5` [src/processes/navigation/model/use-last-visited-route.ts:15]：`isDeferredPrivateRoute()` 已按 `/n/:id`（排除 `/n/0`）启用 deferred 分类。
  - `E6` [src/features/note-save/model/use-note-save.ts:157]：非新建态且本地 miss 时会设置 `missingPrivateNote` 并阻止创建。
  - `E7` [src/features/note-detail-private/model/use-note-detail-private.ts:28]：私有详情 miss 后会异步触发补齐式增量同步。

## 3. 当前实现拆解

### 3.1 F3.4 同步上下文按用户隔离

#### 3.1.1 目标与边界

- 用户场景：用户切换账号、重新登录或在 guest 与登录态之间切换时，本地缓存、同步游标和最后访问路由互不污染。
- 范围内：Dexie 库名、同步游标、最后访问路由 key、会话切换时的上下文初始化。
- 范围外：服务端多租户模型、公开用户缓存独立库重构。

#### 3.1.2 当前实现

- `processes/sync-notes`
  - `getSyncCursorStorageKey(userId)` 以用户作用域拼接游标 key。
  - `ensureSyncScopeReady()` 会在用户变化时切换当前游标视图。
- `shared/lib/storage`
  - `initializeDatabase(userId)` 基于 `getScopedDatabaseName(userId)` 打开对应 Dexie 库。
  - 账号变化时不会复用旧账号数据库实例。
- `processes/navigation`
  - `getLastVisitedRouteStorageKey(userId)` 已按用户作用域保存最后路由。
- `processes/session`
  - `prepareSessionContext(userId)` 会在登录态变化时重新初始化数据库和笔记上下文。

#### 3.1.3 迁移状态

- 状态：已支持
- 当前结论：
  - 同步上下文的三个关键持久化入口已经按用户隔离。
  - 现阶段不再保留旧全局 key 的兼容读取，文档口径应视为“开发期已切到新模型”。

#### 3.1.4 剩余风险与建议

- 风险：项目当前是“按用户切库”，不是“切账号后清理旧账号缓存”，本地仍可能保留历史账号数据副本。
- 风险：guest 上下文与登录上下文虽然隔离，但缺少显式的“本机缓存管理”入口。
- 建议：若后续补缓存治理，应优先补“查看/清理当前与历史上下文缓存”的产品能力，而不是回退到共享库。

#### 3.1.5 验证建议

- 冒烟：A 账号登录并同步 -> 退出 -> B 账号登录，确认游标、路由与本地数据各自独立。
- 回归关注点：guest 模式新建数据后再登录，不应污染登录态游标与最后访问路由。

### 3.2 F3.5 私有路由恢复时序治理

#### 3.2.1 目标与边界

- 用户场景：用户刷新页面或重新打开应用时，私有详情路由应建立在“当前用户同步上下文已就绪”的前提上再恢复。
- 范围内：`/n/:id`、受登录态保护的详情路由恢复时机。
- 范围外：公开路由恢复策略调整。

#### 3.2.2 当前实现

- `useSessionBootstrap()`
  - 启动时会先执行 `restoreImmediateLastVisitedRoute()`。
  - 登录后会执行 `prepareSessionContext()`、建立 Realtime、执行 `sync(true)`，再调用 `restoreDeferredLastVisitedRoute()`。
  - `shouldBlockPrivateRoute` 已经预留了“私有路由未 ready 时阻塞”的计算属性。
- `useLastVisitedRoute()`
  - 已抽出 `immediate/deferred/all` 三种恢复模式。
  - `getRouteRestoreMode(path)` 完全依赖 `isDeferredPrivateRoute(path)`。
  - 当前 `/n/:id`（排除 `/n/0`）会进入 deferred，`/home`、公开路由和新建态 `/n/0` 仍保持 immediate。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 双阶段恢复框架已经完成落地。
  - 私有详情恢复现在会在 `prepareSessionContext()`、Realtime、`sync(true)` 与 PIN 配置同步之后再进入 deferred 恢复。
  - `shouldBlockPrivateRoute` 已经成为真实生效的阻塞防线，而不只是预留逻辑。

#### 3.2.4 剩余风险与建议

- 风险：刷新后若最后停在私有详情页，当前会先进入应用级阻塞态，缺少更细的骨架屏或阶段提示。
- 风险：同步初始化失败时虽然不会过早打开详情，但失败原因目前仍主要依赖日志定位。
- 建议：下一步优先补更明确的 loading / 错误可见性，而不是再改 deferred 分类规则。

#### 3.2.5 验证建议

- 冒烟：分别从 `/home`、公开路由、私有 `/n/:id` 刷新，验证恢复顺序是否符合 immediate/deferred 设计。
- 回归关注点：同步初始化失败时不能把用户直接丢进一个可编辑但数据未准备好的私有详情页。

### 3.3 F3.6 缺失 note 防误创建

#### 3.3.1 目标与边界

- 用户场景：用户进入私有详情页继续编辑旧笔记时，系统只能在显式新建态下创建笔记，不能因为本地 miss 就把旧 note 当成新 note。
- 范围内：详情页加载缺失态、保存链路新建/更新判定、缺失后补齐触发。
- 范围外：公开详情页只读逻辑。

#### 3.3.2 当前实现

- `features/note-save`
  - 保存前先查 `getNote(noteId)`。
  - 若本地存在，走更新分支。
  - 若本地不存在且 `wasNewNote` 为 `false`，会设置 `missingPrivateNote`、清空当前 note，并提示“当前备忘录不存在或尚未同步完成”，不会创建同 id 新记录。
- `features/note-detail-private`
  - 本地 miss 时先进入缺失态，再异步调用 `repairMissingPrivateNoteIfNeeded(id)`。
  - 补齐成功后会重新读取本地 note 并回填页面。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - “非新建态本地 miss 导致误创建”这个高风险问题已经被关闭。
  - 当前剩下的主要问题是体验层面的提示细化，不再是同步完整性主链路缺口。

#### 3.3.4 剩余风险与建议

- 风险：用户当前只能看到通用错误提示，尚未区分“正在补齐中”和“同步后仍不存在”两类场景。
- 建议：后续若继续迁移这一块，应优先补缺失态文案与 CTA，而不是再改创建判定规则。

#### 3.3.5 验证建议

- 冒烟：恢复到旧 `/n/:id`，在本地 miss 场景下尝试编辑并保存，确认不会生成同 id 新记录。
- 回归关注点：新建态 `/n/0` 仍应正常创建，不应被缺失保护误伤。

### 3.4 F3.7 本地缓存健康检查与补齐

#### 3.4.1 目标与边界

- 用户场景：即使本地缓存为空或当前详情笔记缺失，系统也能识别“当前游标与本地状态断裂”，转入补齐流程，而不是继续沿用旧游标做普通增量同步。
- 范围内：同步前健康检查、游标重置、私有详情缺失补齐。
- 范围外：服务端全量校验 API、用户可视化缓存重建面板。

#### 3.4.2 当前实现

- `ensureCacheHealth()`
  - 若本地笔记数为 `0` 且当前游标不是初始值，会调用 `scheduleCacheRepair('empty-local-stale-cursor')`。
  - `scheduleCacheRepair()` 会先把当前用户游标重置回初始值，再记录待执行补齐原因。
- `repairMissingPrivateNoteIfNeeded(noteId)`
  - 仅在已登录且当前不在同步中时触发。
  - 若当前游标已是初始值，会直接跳过，避免重复重置。
  - 否则会安排 `private-note-miss` 补齐并立刻执行一次 `sync(true)`。
- `sync()`
  - 在真正执行 `syncNote()` 前先跑健康检查，所以补齐属于当前同步主链路的一部分，而不是旁路脚本。

#### 3.4.3 迁移状态

- 状态：部分支持
- 当前结论：
  - “空库 + 旧游标”与“私有详情本地 miss”这两类关键断裂场景已经有修复入口。
  - 当前还没有更细的健康状态枚举，也没有把补齐原因反馈给用户面板。

#### 3.4.4 剩余风险与建议

- 风险：补齐过程对用户几乎不可见，用户更容易感知为“先丢了，再恢复”，而不是“系统在做缓存修复”。
- 风险：当前健康检查以 `notes` 总量和目标 note 缺失为主，尚未覆盖更复杂的局部断裂情形。
- 建议：下一步优先补“用户面板展示本次同步是否为缓存补齐模式”；更细的完整性扫描可以放在后续。

#### 3.4.5 验证建议

- 冒烟：制造“空库 + 非初始游标”后重新同步，确认游标被回退并能重新拉回历史数据。
- 冒烟：恢复到缺失的私有详情页，确认会进入缺失态并触发一次补齐式同步。
- 回归关注点：首次登录默认初始游标场景不应重复触发补齐。

### 3.5 F3.8 离开页本地落盘兜底

#### 3.5.1 目标与边界

- 用户场景：用户编辑后立刻切后台、刷新、关闭页面或离开详情页时，尽量把内存中的最新内容写入 Dexie，缩小防抖窗口造成的数据丢失风险。
- 范围内：详情页离开、`pagehide`、`beforeunload`、Ion View 离场时的本地 flush。
- 范围外：云端强一致上传保证、所有页面统一 flush 策略。

#### 3.5.2 当前实现

- `shared/lib/storage/sync`
  - `useRefDBSync` 仍以防抖写入为主，但额外暴露了 `manualSync()`。
- `features/note-detail-leave`
  - 离开详情页时会调用 `manualSync()`，只做本地落盘，不推进云端游标。
  - `triggerLeavePageLocalFlush(reason)` 会先清理挂起的防抖保存，再转入静默保存链路。
- `features/note-detail-leave-lifecycle`
  - 已在 `pagehide`、`beforeunload`、`view-leave` 接入本地 flush。

#### 3.5.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 详情页主链路已经具备本地落盘兜底。
  - 这套能力仍是 best effort，无法保证浏览器/系统强杀时 100% 成功。
  - 目前覆盖重点在 `NoteDetail`，不是应用范围的统一能力。

#### 3.5.4 剩余风险与建议

- 风险：大文档或极端关闭场景下，最后一小段改动仍可能停留在内存态。
- 风险：当前没有额外指标说明 flush 是否成功，问题定位仍主要依赖日志。
- 建议：这块后续优先级低于路由恢复时序；如果继续迁移，应先补可观测性，再考虑全局推广。

#### 3.5.5 验证建议

- 冒烟：编辑后立即刷新、切后台、切走详情页，重新进入后确认 Dexie 中仍保留最新内容。
- 回归关注点：频繁切换详情页时不应引入明显性能抖动或重复保存。
