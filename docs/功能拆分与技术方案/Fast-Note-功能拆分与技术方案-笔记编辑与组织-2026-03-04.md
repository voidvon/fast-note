## 2. 模块目标与现状

- 业务目标：支持稳定的“创建 -> 编辑 -> 移动/归档 -> 删除/恢复”闭环，保障本地离线优先可用。
- 当前状态（已支持/部分支持/未支持）：部分支持
- 当前判断：
  - `已支持`：保存主流程已经沉到 `features/note-save`，空草稿不落库、缺失 note 保护、保存状态反馈都已闭环。
  - `已支持`：移动、删除/恢复、回收站页面和最近删除窗口都已经迁入当前 FSD 分层。
  - `部分支持`：文件夹树、父级计数和桌面选中恢复已经在运行，但移动后的全量校准能力仍不明确。
  - `已支持`：文件夹深层删除/恢复已收口到 `entities/note`，子树软删与同批恢复测试已补齐。
- 架构意图：
  - `entities/note` 负责文件夹树、计数、删除窗口等本地笔记域状态与规则。
  - `features/note-save`、`features/note-move`、`features/note-delete`、`features/note-actions-menu` 负责用户动作编排。
  - `pages/home`、`pages/folder`、`pages/note-detail`、`pages/deleted` 只负责页面装配和交互接线。

## 3. 当前实现拆解

### 3.1 F2.1 文件夹树与计数一致性

#### 3.1.1 目标与边界

- 用户场景：用户在多层文件夹下组织笔记并查看准确计数。
- 范围内：文件夹树构建、父级计数更新、移动后计数修正、桌面当前选中恢复。
- 范围外：公开用户文件夹树。

#### 3.1.2 当前实现

- `entities/note`
  - 已维护 `notesMap`、`parentIdMap` 等索引。
  - 已提供 `getFolderTreeByParentId()`、`getNoteCountByParentId()`、`updateParentFolderSubcount()`。
  - 文件夹树构建已经沉到 `folder-tree` 领域函数。
- `features/note-move`
  - 移动后会更新当前 note 的 `parent_id` 与 `updated`。
  - 当前会补刷旧父级和新父级的 `note_count`。
- `pages/home`
  - 桌面端已引入活动 note 快照恢复与持久化，尽量保持当前选中上下文。
- `pages/folder`
  - 文件夹列表依赖实体仓储数据和状态恢复管理，不再自行维护独立树结构。

#### 3.1.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 文件夹树、索引和父级计数已经进入 `entities/note`，主链路已归位。
  - 当前问题不在“有没有计数”，而在“跨层移动后是否有足够强的校准能力和回归证据”。

#### 3.1.4 剩余风险与建议

- 风险：当前移动流程主要补旧父级和新父级计数，祖先链全量校准能力仍不明确。
- 风险：多层文件夹跨层移动后，现有逻辑缺少专门的自动化回归证据。
- 建议：后续优先补“计数全量重算 helper”或等价校准入口，再补多层移动测试。

#### 3.1.5 验证建议

- 冒烟：多层文件夹间移动笔记，确认旧父级与新父级计数正确。
- 回归关注点：重启应用后计数与列表展示仍一致。

### 3.2 F2.2 编辑自动保存与新建笔记流程

#### 3.2.1 目标与边界

- 用户场景：用户编辑笔记时无需手动保存，切换笔记不丢内容。
- 范围内：新建 ID 分配、标题/摘要提取、防抖保存、空内容不落库、缺失 note 保护、保存状态反馈。
- 范围外：附件上传与回填。

#### 3.2.2 当前实现

- `features/note-save`
  - 已统一新增、更新、空草稿不落库、缺失 note 保护、本地保存后静默同步。
  - 非静默保存会维护 `isSaving`，失败时统一转为顶部错误提示。
- `pages/note-detail`
  - 保留 800ms 防抖保存入口。
  - 负责移动端草稿 URL 替换、顶部错误 toast 和头部 spinner 呈现。
- `features/note-detail-*`
  - 详情初始化、离页保存、缺失 note 补齐和视图状态都已拆到对应 feature。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 保存主流程已经从页面散点逻辑迁到 `features/note-save`。
  - 当前缺口主要不在保存链路本身，而在同步、附件和锁等外围链路的可观测性。

#### 3.2.4 剩余风险与建议

- 风险：保存成功后的同步失败仍只展示通用错误提示，未细分“本地已保存 / 云端失败”。
- 建议：这块当前优先级低于子树删除和计数校准，不建议再回退到页面内保存状态管理。

#### 3.2.5 验证建议

- 冒烟：编辑已有笔记、切换桌面当前笔记、移动端新建空草稿，确认保存行为符合预期。
- 回归关注点：缺失 note 不会误创建，空草稿不会落库。

### 3.3 F2.3 删除、回收站、恢复与永久删除

#### 3.3.1 目标与边界

- 用户场景：用户可安全删除并在 30 天窗口恢复，必要时永久删除。
- 范围内：软删除、恢复、永久删除标记、回收站展示。
- 范围外：超过 30 天的云端清理细节。

#### 3.3.2 当前实现

- `features/note-delete`
  - 已处理单 note 软删除与父级计数刷新。
- `widgets/note-more`
  - 当前通过 `useNoteDelete()` 接入删除动作。
  - 列表更多操作仍承载删除、恢复、永久删除、移动和锁入口。
- `features/note-actions-menu`
  - 删除/恢复动作已委托 `entities/note` 的统一软删入口，不再在组件里手工遍历一层 children。
- `pages/deleted`
  - 已按最近 30 天窗口展示回收站列表。
- `entities/note`
  - 已沉淀最近删除保留窗口规则，并统一处理子树软删、祖先路径恢复与同批后代恢复。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - 单 note、文件夹子树删除、回收站展示与恢复主链都已闭环。
  - 删除/恢复规则已沉到 `entities/note`，页面和菜单层只保留用户动作入口。

#### 3.3.4 剩余风险与建议

- 风险：回收站当前仍是扁平列表，深层恢复路径虽已自动补回，但层级感知与批量操作体验仍偏弱。
- 建议：若继续演进这块，优先补回收站分组/层级展示和更贴近真实交互的集成验证。

#### 3.3.5 验证建议

- 冒烟：删除普通 note 后可在回收站中看到并恢复。
- 回归关注点：删除多层文件夹时，所有活跃后代都应进入回收站；恢复深层 note 时，祖先路径应补回，同时不应误恢复更早独立删除的兄弟节点。
