## 2. 模块目标与现状

- 业务目标：
  - 让用户在备忘录列表中一眼识别哪些 note 已启用锁，同时保证锁定与未锁定条目的标题起始位稳定，不因图标出现与否发生抖动。
- 当前状态（已支持/部分支持/未支持）：已支持
- 当前判断：
  - `已支持`：`Note` 契约已经内建 `is_locked`，列表展示不需要新增数据字段。
  - `已支持`：`widgets/note-list` 已经把锁住态和未锁定占位统一收口到固定前导槽位。
  - `已支持`：锁图标颜色和尺寸已经接入现有主题 token，不是临时硬编码样式。
  - `部分支持`：自动化覆盖了组件和首页混合列表，但文件夹页仍主要依赖共享组件复用来兜底，没有单独入口级集成测试。
- 架构意图：
  - `shared/types` 负责锁字段标准化，保证旧数据缺失字段时也能按未锁定处理。
  - `widgets/note-list` 负责固定宽度状态槽位、图标渲染和文本对齐。
  - `features/note-lock` 负责更高层的会话感知三态；本模块只关注其基础回退语义，即 `locked / placeholder`。
- 关键证据：
  - `E1` `src/shared/types/index.ts:14`：`NoteLockFields` 已定义 `is_locked`。
  - `E2` `src/shared/types/index.ts:75`：`normalizeNoteLockFields()` 会把缺失锁字段归一成未锁定。
  - `E3` `src/widgets/note-list/ui/note-list-item.vue:49`：列表项会先解析锁展示态，再决定显示图标还是占位。
  - `E4` `src/widgets/note-list/ui/note-list-item.vue:160`：note 行已具备独立前导槽位。
  - `E5` `src/widgets/note-list/ui/note-list-item.vue:267`：锁状态槽位颜色已接入 `var(--c-icon)`。
  - `E6` `src/widgets/note-list/ui/note-list-item.vue:277`：锁图标尺寸已独立收口到 `note-lock-icon`。
  - `E7` `src/css/var.scss:47`：浅色主题已定义 `--c-icon`。
  - `E8` `tests/unit/components/note-list-item-lock-indicator.spec.ts:75`：组件单测已覆盖锁住态渲染。

## 3. 当前实现拆解

### 3.1 F1.1 列表项固定前导槽位与锁住态回退

#### 3.1.1 目标与边界

- 用户场景：
  - 用户在首页或文件夹页浏览 note 列表时，需要快速识别已锁定 note，同时不希望标题因为图标有无而左右跳动。
- 范围内：
  - note 行左侧固定宽度状态槽位。
  - `is_locked=1` 时显示锁图标。
  - 未锁定或缺失锁字段时隐藏图标内容但保留占位。
- 范围外：
  - 文件夹行增加锁状态。
  - 锁图标点击交互、提示文案或跳转行为。

#### 3.1.2 当前实现

- `shared/types`
  - `NoteLockFields` 和 `normalizeNoteLockFields()` 已经统一定义锁字段和旧数据归一语义。
- `widgets/note-list`
  - `note-list-item.vue` 的 `resolvedLockIndicatorState` 会优先消费会话态 map；没有 map 时回退到 `is_locked === 1 ? 'locked' : 'placeholder'`。
  - note 行模板固定渲染 `note-leading-slot`，图标只在非 `placeholder` 时出现，因此文本布局不会因图标显隐而改变。
  - 该逻辑通过 `note-list.vue` 被首页和文件夹页共享，不是某一个页面单独维护。
- `features/note-lock`
  - 本模块没有把会话规则写进组件，只是预留了会话态 map 的接入位，保持列表项仍可独立回退到基础锁住态。

#### 3.1.3 迁移状态

- 状态：已支持
- 当前结论：
  - 列表锁住态展示已经迁入目标架构。
  - `widgets` 只负责 UI 展示，锁字段语义仍由共享契约层和锁 feature 提供，没有反向污染实体结构。

#### 3.1.4 剩余风险与建议

- 风险：当前文件夹页没有独立集成测试，但它复用同一份 `widgets/note-list` 渲染链路。
- 建议：如果后续继续扩锁图标能力，优先保持三态语义都从 feature 输出，不要在页面层再补一套锁状态判断。

#### 3.1.5 验证建议

- 冒烟：首页混合显示锁定 note、未锁定 note、文件夹，确认只有 note 行出现前导槽位。
- 回归关注点：锁定与未锁定条目混排时，标题左边界保持一致。

### 3.2 F1.2 图标样式、主题 token 与视觉一致性

#### 3.2.1 目标与边界

- 用户场景：
  - 用户在浅色/深色主题之间切换时，锁图标需要持续可见，并和现有列表前导图标保持一致量级。
- 范围内：
  - 锁图标颜色 token 复用。
  - 前导槽位宽度、图标字号和顶部对齐。
- 范围外：
  - 新建独立锁图标主题体系。
  - 全局主题系统重构。

#### 3.2.2 当前实现

- `widgets/note-list`
  - `note-leading-slot` 负责固定宽度、纵向对齐和点击区域。
  - `note-lock-icon` 单独收口字号和轻微顶部偏移，避免和文本首行错位。
- `shared/css`
  - 列表锁图标直接复用 `--c-icon`，主题切换时由现有 token 体系统一生效。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 视觉规范已经固化在共享样式 token 和组件样式内，不依赖页面分支临时覆盖。
  - 这部分已经属于“当前实现”，不是待落地方案。

#### 3.2.4 剩余风险与建议

- 风险：当前自动化只能证明 DOM 结构和主题类名切换，不会真正校验不同平台字体渲染差异。
- 建议：如果后续重做列表视觉，继续复用 `note-leading-slot` 这个结构，不要把锁图标直接塞回标题文本流。

#### 3.2.5 验证建议

- 冒烟：切换浅色/深色主题，确认锁图标都清晰可见。
- 回归关注点：长标题换行时，图标仍贴合首行顶部且不挤压摘要文本。

### 3.3 F1.3 自动化覆盖与剩余缺口

#### 3.3.1 目标与边界

- 用户场景：
  - 该能力虽然只改列表展示，但命中通用 `widgets`，需要最小自动化覆盖避免在首页或文件夹页回退。
- 范围内：
  - 组件单测。
  - 首页混合列表集成验证。
- 范围外：
  - e2e。
  - 锁管理流程和详情页解锁流程回归。

#### 3.3.2 当前实现

- `tests/unit/components/note-list-item-lock-indicator.spec.ts`
  - 已覆盖锁住态渲染、占位态渲染、旧数据缺失锁字段时的回退行为。
- `tests/integration/home/note-list-lock-indicator.spec.ts`
  - 已覆盖首页混合列表中锁图标和空白占位的数量、顺序以及深色模式下的可见性。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - 当前不是“没有验证”，而是“验证主要集中在共享组件和首页入口”。
  - 这对锁住态基础展示已经够用，但还没延伸到每个列表入口逐页建集成用例。

#### 3.3.4 剩余风险与建议

- 风险：文件夹页、搜索结果页若未来引入不同列表容器，仍可能绕开当前共享测试面。
- 建议：后续若新增新的 note 列表容器，优先复用现有 `NoteList`/`NoteListItem`，而不是复制一个新的锁图标实现。

#### 3.3.5 最小人工冒烟建议

- 首页混合列表：
  - 确认锁定 note 显示 `lockClosed`，未锁定 note 仅保留占位。
- 文件夹页：
  - 确认长标题、多行摘要和父文件夹信息不受前导槽位影响。
