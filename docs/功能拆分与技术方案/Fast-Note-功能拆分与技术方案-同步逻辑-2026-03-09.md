## 2. 模块目标与现状

- 业务目标：保持 Fast-Note 的 offline-first 语义，在本地 Dexie 为即时真相源的前提下，实现登录后的云端增量同步与在线回写。
- 当前状态（已支持/部分支持/未支持）：部分支持
- 当前判断：
  - `部分支持`：自动同步、手动同步、登录后首轮同步都已接入；私有详情 deferred 恢复也已生效，但恢复阶段反馈仍偏弱。
  - `部分支持`：当前同步主体是按用户作用域的双向增量，不是全量校验式同步。
  - `部分支持`：冲突裁决、软删除 30 天窗口、附件两阶段同步都已存在，但失败聚合与用户反馈仍偏弱。
  - `部分支持`：Realtime 已作为在线补丁流接入，但还没有清晰的 UI 状态与首轮同步分层表达。
- 关键证据：

## 3. 当前实现拆解

### 3.1 F3.1 同步触发入口与会话启动

#### 3.1.1 目标与边界

- 用户场景：用户打开应用、登录账号、编辑笔记或手动点击同步时，系统能够在合适时机触发同步。
- 范围内：应用启动恢复、登录态会话初始化、保存后静默同步、手动同步入口。
- 范围外：首次安装引导、推送通知。

#### 3.1.2 当前实现

- `processes/session`
  - `initializeSession()` 会初始化认证服务、恢复立即可访问路由、在登录态变化后切换本地上下文。
  - `bootstrapLoggedInSession()` 会串行执行 `prepareSessionContext()`、Realtime 连接、`sync(true)`、PIN 配置同步，并在最后尝试恢复 deferred 路由。
- `features/note-save`
  - 笔记保存成功并完成本地 flush 后，会执行一次静默同步。
  - 未登录时静默同步会被跳过，不会中断本地保存。
- `widgets/user-profile`
  - 用户面板保留显式手动同步入口，可展示上传、下载、删除统计。

#### 3.1.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 同步入口已经分布在启动、保存、手动触发三条主链路上。
  - “先同步后恢复私有详情”的设计框架已经落地，当前主要缺口转为恢复阶段可视化而不是路由分类本身。

#### 3.1.4 剩余风险与建议

- 风险：刷新后若最后停在私有详情页，当前仍可能先进入缺失态，再等待补齐式同步修复。
- 风险：当前没有显式的“会话初始化阶段”对外状态，UI 很难区分“正在准备私有上下文”和“同步失败”。
- 建议：下一步优先补恢复阶段 loading / 错误反馈，而不是再扩展新的触发入口。

#### 3.1.5 验证建议

- 冒烟：刷新应用 -> 恢复到 `/n/:id` -> 登录 -> 完成首轮同步，确认没有误走新建链路。
- 回归关注点：离线打开已有本地笔记时，不应被登录态同步阻塞。

### 3.2 F3.2 双向增量同步与游标推进

#### 3.2.1 目标与边界

- 用户场景：多设备编辑后，系统通过增量拉取和推送逐步达成最终一致。
- 范围内：本地增量、云端增量、游标推进、缓存补齐入口。
- 范围外：服务端全量对账、历史版本回放。

#### 3.2.2 当前实现

- 游标模型
  - 初始游标固定为 `2010/01/01 00:00:00`。
  - 真实游标通过按用户作用域的 localStorage key 读取和写回。
- 增量读取
  - 本地侧依赖 `getNotesByUpdated(updated)`。
  - 云端侧依赖 `updated > cursor && user_id = auth.id`。
- 游标推进
  - 同步中每成功处理一条记录，都会比较 `syncedUpdatedAt` 并推进游标。
- 缓存补齐
  - `ensureCacheHealth()` 会在“空库 + 非初始游标”场景下重置游标。
  - `repairMissingPrivateNoteIfNeeded()` 会在私有详情 miss 时安排一次补齐式同步。

#### 3.2.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 现有实现是标准的“按游标双向增量”，不是“每次全量扫描整库”。
  - 当前已经补上两个关键的完整性修复入口，但同步正确性仍然高度依赖 `updated` 和本地缓存连续性。

#### 3.2.4 剩余风险与建议

- 风险：当本地局部数据断裂但又不满足现有两个 repair 条件时，旧记录仍可能因为游标已前进而不会重新出现。
- 风险：游标按单条记录成功推进，单条失败会中断后续流程，也不会输出更细的恢复建议。
- 建议：同步逻辑下一步优先补“补齐模式可观测性”和“更细的失败结果”，不是改成全量同步。

#### 3.2.5 验证建议

- 冒烟：A 账号同步后退出 -> B 账号登录，确认游标独立。
- 冒烟：制造“空库 + 有游标”后重新同步，确认历史数据会被重新拉回。
- 回归关注点：首次登录默认初始游标场景不能重复触发无意义的全量补齐。

### 3.3 F3.3 同步操作队列与冲突裁决

#### 3.3.1 目标与边界

- 用户场景：本地和云端都有变更时，系统按统一规则决定上传、下载、删除或本地清理。
- 范围内：操作队列构建、`updated` 冲突裁决、30 天删除窗口。
- 范围外：多人协作冲突合并。

#### 3.3.2 当前实现

- 操作队列构建
  - 先以本地增量记录为主，结合 `cloudNotesMap` 判断 `upload/update/delete/deleteLocal/skip`。
  - 再以云端增量记录为主，补充本地不存在或云端更新较新的下载动作。
- 冲突裁决
  - 非删除记录主要依赖 `updated` 时间戳的新者覆盖旧者。
  - 已删除记录保留 30 天窗口，窗口外触发本地物理删除或云端删除态同步。
- 执行策略
  - 所有操作按 `updated` 升序串行执行。
  - 任何单条操作异常都会 `throw`，本轮同步立即停止。
- 结果聚合
  - 当前只返回 `uploaded/downloaded/deleted` 三类计数。

#### 3.3.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 当前冲突裁决模型和软删除窗口已经是运行中的正式实现。
  - 缺口不在“有没有规则”，而在“失败时怎样把剩余状态暴露给用户和后续重试”。

#### 3.3.4 剩余风险与建议

- 风险：完全依赖客户端时间戳，设备时钟漂移仍可能影响裁决结果。
- 风险：单条失败会终止整轮同步，但返回值没有 `failed` 维度，用户面板也无法精确表达部分成功。
- 建议：后续如果继续迁移这一块，应优先补失败统计与失败原因暴露，而不是先动冲突规则本身。

#### 3.3.5 验证建议

- 冒烟：双端同时编辑同一笔记，确认时间较新的内容覆盖较旧内容。
- 冒烟：软删除记录在 30 天内继续参与同步，超过 30 天后转入本地清理。
- 回归关注点：单条异常不会回滚已完成记录，但需要确认游标推进仍符合预期。

### 3.4 F3.4 云端 upsert 与附件同步

#### 3.4.1 目标与边界

- 用户场景：笔记内容和附件在本地编辑后，能够可靠地同步到 PocketBase。
- 范围内：云端 `create/update/auto` 写入、附件上传、富文本 URL 回填。
- 范围外：文件压缩、图片转码、对象存储迁移。

#### 3.4.2 当前实现

- 云端 upsert
  - `create` 模式遇到“已存在”错误会 fallback 到 `update`。
  - `update` 模式遇到 404 会 fallback 到 `create`。
  - `auto` 模式默认先尝试 `update`，404 时再 `create`。
- 附件同步
  - 上传前会把富文本中的本地 hash 替换成临时文件标识符。
  - PocketBase 返回真实文件名后，会再次回填正文并补一次 `update`。
  - 同步过程中也会把云端返回的 `user_id`、`updated`、`files` 回写到本地 note。
- 保存链路
  - 本地保存和本地 flush 成功后才会触发同步。
  - 若同步失败，当前只通过顶部错误提示告诉用户“同步失败，请检查网络连接”。

#### 3.4.3 迁移状态

- 状态：部分支持
- 当前结论：
  - 客户端 upsert 和附件两阶段同步都已经落地，不再是设计稿能力。
  - 当前剩余问题集中在失败反馈与两阶段窗口期，不是“还没有附件同步”。

#### 3.4.4 剩余风险与建议

- 风险：附件同步是两阶段写入，若第二次正文回填失败，会出现短暂的正文与 `files` 元数据不一致。
- 风险：当前用户只能收到“同步失败”的通用提示，缺少“本地已保存、云端失败”的更精细表达。
- 建议：后续如果继续迁移这一块，应优先补用户反馈与失败重试，而不是推翻现有 upsert 模型。

#### 3.4.5 验证建议

- 冒烟：新建笔记插入附件 -> 保存 -> 同步 -> 刷新，确认正文里的附件引用已替换为 PocketBase 文件名。
- 回归关注点：同一文件在正文中多次引用时不应重复上传。

### 3.5 F3.5 Realtime 回写与本地索引一致性

#### 3.5.1 目标与边界

- 用户场景：另一端完成创建、更新或删除后，当前设备能通过 Realtime 快速收到变化并写回本地。
- 范围内：建立连接、订阅过滤、消息处理、重连。
- 范围外：后台保活策略细化、首轮历史数据恢复。

#### 3.5.2 当前实现

- 连接与订阅
  - 登录态初始化时会创建 `PocketBaseRealtimeService` 并连接。
  - 订阅目标为 `notes` 集合，且按 `user_id` 过滤。
- 消息处理
  - `create` 映射为本地 `addNote`。
  - `update` 会比较本地 `updated` 与远端 `updated`，仅在远端更新较新时覆盖本地。
  - `delete` 会执行本地 `deleteNote`。
- 重连
  - 断线后会采用指数退避重连。
  - 登出时会主动断开连接。

#### 3.5.3 迁移状态

- 状态：部分支持
- 当前结论：
  - Realtime 已经是当前线上同步链路的一部分，用来缩短多端一致性的延迟。
  - 它仍然只是“在线补丁流”，不能替代登录后的首轮历史同步。

#### 3.5.4 剩余风险与建议

- 风险：当前没有用户可见的 Realtime 状态，用户无法区分“已连上 Realtime”和“历史数据已完成同步”。
- 风险：虽然已有限制重复连接的保护，但乱序消息与重复推送更多依赖时间戳和日志兜底。
- 建议：若继续迁移，应优先补“历史同步完成”和“Realtime 已连接”的状态区分，再考虑更细的诊断信息。

#### 3.5.5 验证建议

- 冒烟：设备 A 修改 -> 设备 B 在线收到更新并写回本地。
- 回归关注点：登出后必须断开订阅，避免串账号数据。
