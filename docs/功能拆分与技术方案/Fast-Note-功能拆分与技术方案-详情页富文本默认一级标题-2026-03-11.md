## 2. 模块目标与现状

- 业务目标：
  - 让用户在新建详情页时直接从一级标题开始输入，减少一次格式切换，并保持已有内容、离线保存和同步链路不回退。
- 当前状态（已支持/部分支持/未支持）：已支持
- 当前判断：
  - `已支持`：默认新建标题骨架已经沉到 `features/note-editor`，并且是幂等能力。
  - `已支持`：新建草稿入口已经在 `features/note-detail-editor` 中统一接入默认标题初始化和聚焦。
  - `已支持`：保存链路已经改成语义空内容判断，默认空 `h1` 不会误创建空白笔记。
  - `部分支持`：当前测试已覆盖主要 helper 和页面接线，但还没有真实 Tiptap 实例级 e2e。
- 架构意图：
  - `features/note-editor` 负责默认标题文档骨架和“有意义内容”语义判断。
  - `widgets/editor` 负责把编辑器能力暴露给页面。
  - `features/note-detail-editor` 负责新建草稿初始化。
  - `features/note-save` 负责保存前是否落库的决策，不把“空标题骨架”误当成真实内容。

## 3. 当前实现拆解

### 3.1 F1.1 新建详情页首块默认一级标题

#### 3.1.1 目标与边界

- 用户场景：
  - 用户进入空白详情页后，不需要打开格式面板，也不需要先输入正文再改样式，首个输入位置就是一级标题。
- 范围内：
  - 私有详情页新建场景（`id === '0'`）的编辑器初始化。
  - 新建空白编辑器首块的默认块类型设定。
- 范围外：
  - 已有笔记、公开只读详情页、锁定态详情页内容重写。
  - 文本格式面板交互改造。

#### 3.1.2 当前实现

- `features/note-editor`
  - 定义默认一级标题文档骨架 `DEFAULT_NEW_NOTE_HEADING_CONTENT`。
  - 提供 `applyDefaultHeadingIfEmptyToEditor()`，只在空编辑器上生效。
- `widgets/editor`
  - 通过 exposed API 向上层暴露默认标题初始化能力。
- `features/note-detail-editor`
  - `showNewDraft()` 已统一执行清空内容、设置可编辑、应用默认标题、再聚焦。
- `pages/note-detail`
  - 页面只负责进入新建草稿分支，不再直接拼编辑器命令细节。

#### 3.1.3 迁移状态

- 状态：已支持
- 当前结论：
  - 默认一级标题能力已经迁入目标架构。
  - 主链路不再依赖页面临时逻辑，而是由 editor feature 和 detail feature 协作完成。

#### 3.1.4 剩余风险与建议

- 风险：当前主要风险不在业务语义，而在不同浏览器和真实 Tiptap 实例上的命令链联动。
- 建议：后续若继续增强，优先补更真实的编辑器实例测试，不建议把默认标题逻辑重新塞回页面。

#### 3.1.5 验证建议

- 冒烟：打开空白详情页，确认首行直接为一级标题且自动聚焦。
- 回归关注点：已有笔记内容不会被自动改写，重复进入新建态也不会插入多个空标题块。

### 3.2 F1.2 默认标题骨架的空内容判定与保存兜底

#### 3.2.1 目标与边界

- 用户场景：
  - 用户进入新建详情页后未输入任何有效内容就离开，不应因为系统预置了一个空一级标题而自动生成一条空白笔记。
- 范围内：
  - 保存前的内容有效性判断。
  - 默认空 `h1`、空 `p` 等骨架节点的语义空内容识别。
- 范围外：
  - 附件上传、表格编辑、同步重试策略重构。

#### 3.2.2 当前实现

- `features/note-editor`
  - `hasMeaningfulEditorContent()` 会把非空文本、附件、表格、任务列表视为有效内容。
  - `isDefaultNewNoteHeadingDocument()` 可识别“只有空一级标题骨架”的文档。
- `widgets/editor`
  - 已向页面暴露 `isMeaningfulContent()`。
- `features/note-save`
  - 保存时不再仅看 HTML 字符串真假。
  - 新建草稿且无真实内容时，直接跳过新增、删除和同步。

#### 3.2.3 迁移状态

- 状态：已支持
- 当前结论：
  - 现在的保存链路已经能把“默认空标题骨架”与“真实内容”区分开。
  - 这个能力已经和默认标题初始化一起形成闭环，不再是单点修补。

#### 3.2.4 剩余风险与建议

- 风险：当前测试主要基于 helper 和 exposed API，没有覆盖更真实的浏览器输入与编辑器插件组合。
- 建议：后续优先补编辑器实例级回归，而不是再扩展更多默认标题规则。

#### 3.2.5 验证建议

- 冒烟：进入新建详情页但不输入任何有效内容后离开，确认不会新增空白笔记。
- 冒烟：只输入一级标题文本后保存，确认会正常创建笔记。
- 回归关注点：附件、表格、任务列表等非纯文本节点仍会被视为有效内容。

### 3.3 F1.3 新建默认标题回归测试补齐

#### 3.3.1 目标与边界

- 用户场景：
  - 该需求的主价值集中在编辑器初始化与保存边界，需要自动化测试覆盖新建、编辑、离开页三类核心路径。
- 范围内：
  - 单元测试和集成测试补齐。
- 范围外：
  - Cypress 端到端大规模补测。

#### 3.3.2 当前实现

- `tests/unit/hooks/useEditor-meaningful-content.spec.ts`
  - 已覆盖默认标题初始化幂等性和语义有效内容判断。
- 现有页面/保存相关测试
  - 已覆盖保存状态、空草稿不落库、桌面切换前静默保存等关键边界。

#### 3.3.3 迁移状态

- 状态：已支持
- 当前结论：
  - 默认标题本身的 helper 级回归已经具备。
  - 当前缺的不是“有没有自动化”，而是“自动化够不够贴近真实编辑器实例”。

#### 3.3.4 剩余风险与建议

- 风险：真实 Tiptap 实例、输入法、插件组合带来的交互差异，目前仍主要依赖手工冒烟。
- 建议：如果继续迁移这一块，优先补一层更接近真实编辑器实例的集成验证。

#### 3.3.5 最小人工冒烟建议

- 新建详情页：
  - 打开空白详情，确认首行直接为一级标题。
- 不输入离开：
  - 直接返回或切页，确认不会生成空白笔记。
- 输入标题保存：
  - 仅输入一级标题文本，确认能正常创建并回写标题。
