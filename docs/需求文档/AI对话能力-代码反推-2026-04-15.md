## 2. 背景与问题陈述

- 当前能力概述（基于代码事实）：
  - 首页已经提供 AI 对话入口，入口挂载在 `GlobalSearch` 组件内，与全局搜索共用同一输入 dock，用户可在“搜索 / AI 对话”两种模式间切换。[证据: E1(A), E2(A)]
  - 当前 AI 对话是浏览器直连 OpenAI-compatible 接口的流式聊天能力，依赖用户配置 `Base URL`、`Model`、`API Key`。[证据: E3(A), E4(A), E5(A)]
  - 当前会话消息和模型配置会按用户 scope 存入浏览器 `localStorage`，支持重新挂载后恢复对话内容。[证据: E3(A), E9(A), E10(A)]
  - 当前对话链路已经接入本地工具执行器与确认会话；模型可通过 JSON envelope 请求 `search_notes`、`get_note_detail`、`list_folders`、`create_note`、`update_note`、`move_note`、`delete_note`、`set_note_lock` 等本地工具。[证据: E3(A), E5(A), E6(A), E7(A), E12(A)]
  - 当前发送给模型的请求已支持注入有限本地上下文，包括活动目录、活动笔记、候选笔记、最近笔记；这些上下文会被转成第二段 system prompt，而不是把全量笔记直接送模。[证据: E2(A), E5(A), E13(A)]
  - 工具结果当前会渲染成结构化卡片，支持从 AI 结果直接打开备忘录和文件夹。[证据: E3(A), E14(A), E15(A), E16(A)]
- 当前痛点或机会点：
  - 当前工具编排依赖模型返回约定 JSON，协议仍偏脆弱，缺少服务端统一约束层。[证据: E5(A), E12(A)]
  - 当前对话持久化仍只保存纯文本消息，结构化卡片、确认态和对象动作在刷新后不会恢复。[证据: E3(A)]
  - 当前上下文注入仍是有限摘要级别，不包含完整目录树、完整候选详情或稳定的引用体系。[证据: E13(A)]
  - 当前浏览器直连模式会把 API Key 保存在本地浏览器，组件本身也显式提示“只在受信任环境中使用”。[证据: E4(A), E3(A)]
- 本次文档目标：
  - 基于现有实现梳理 AI 对话当前已支持能力。
  - 明确当前已接入入口的能力、当前仍未持久化的能力，以及后续优化缺口。
  - 记录当前实现与目标型“AI 对话代理”之间的差距，供后续优化。

## 3. 目标与非目标

### 3.1 目标

- G1：沉淀一份基于代码事实的 AI 对话现状文档，而不是理想化能力清单。
- G2：帮助产品、研发、测试快速识别“已上线能力”“预埋能力”“未打通能力”。
- G3：把当前实现中的问题和后续优化点结构化记录，便于后续排期。

### 3.2 非目标

- NG1：不在本次文档中重新定义理想状态下的完整 AI 代理架构。
- NG2：不在本次文档中给出最终技术方案或接口重构方案。

## 4. 用户与使用场景

- 用户角色 R1：
  - 首页已登录或游客态用户，通过首页底部/侧边的 `GlobalSearch` dock 使用全局搜索或 AI 对话。[证据: E1(A), E2(A), E9(A)]
- 核心场景 S1：
  - 用户切换到 AI 对话模式，配置直连模型后输入消息，接收流式回答并可继续追问。[证据: E2(A), E3(A), E4(A), E10(A)]
- 核心场景 S2：
  - 用户重新进入页面后恢复本地保存的 AI 对话历史，并继续当前会话。[证据: E3(A), E10(A)]
- 核心场景 S3：
  - 用户在 AI 对话中触发本地工具后，可看到结构化预览/结果卡片，并在确认后继续执行高风险操作。[证据: E3(A), E7(A), E12(A), E14(A)]

## 6. 功能需求

### FR-001 首页入口与模式切换

- 用户价值：
  - 用户可以在首页同一入口下在“搜索”和“AI 对话”之间切换，不需要进入独立页面。
- 前置条件：
  - 用户已进入首页。
- 触发方式：
  - 聚焦 `GlobalSearch` 输入区后，点击切换按钮进入 AI 对话模式。
- 主流程：
  - 首页引入 `GlobalSearch` 组件作为统一入口。
  - `GlobalSearch` 内部维护 `search` / `ai` 两种输入模式。
  - 切换到 AI 模式后，面板标题显示“AI 对话”，面板正文渲染 `AiChatPanel`。
- 异常流程：
  - 若关闭面板，则会清空当前搜索态或 AI 输入草稿并隐藏面板。
- 业务规则：
  - AI 对话不是独立页面，而是首页搜索入口中的一种模式。
  - 当前搜索模式和 AI 模式共用同一个输入 dock，但展示体不同。
- 数据变化：
  - 修改 `inputMode`、`aiDraft`、`showGlobalSearch` 等本地状态。
- 权限要求：
  - 代码中未见额外权限校验。
- 依赖项：
  - `GlobalSearch`、`AiChatPanel`、`useGlobalSearch`。
- 证据引用：`[证据: E1(A), E2(A)]`
- 置信度：`高`

### FR-002 模型配置与浏览器直连

- 用户价值：
  - 用户可在前端直接配置 OpenAI-compatible 服务并立刻使用 AI 对话。
- 前置条件：
  - 浏览器环境可访问 `localStorage`。
- 触发方式：
  - 用户首次进入 AI 对话且未配置 Provider，或手动点击配置按钮。
- 主流程：
  - 系统读取环境变量默认值，再尝试读取本地已保存配置。
  - 若未配置完整 Provider，则打开配置弹窗。
  - 配置项包含 `Base URL`、`Model`、`API Key`。
  - 用户保存后，配置写入用户 scope 下的 `localStorage`。
- 异常流程：
  - 本地配置解析失败时，清理损坏记录后重新初始化。
  - 若未配置完整，则不允许提交 AI 请求。
- 业务规则：
  - 当前模式为“浏览器直连”。
  - 配置组件显式提示 API Key 会保存在浏览器，仅适合受信任环境。
  - 配置 key 与对话 key 都按用户 scope 隔离。
- 数据变化：
  - 写入 `ai-chat-settings:<scopeId>`。
- 权限要求：
  - 代码中未见账号级权限限制。
- 依赖项：
  - `createScopedStorageKey()`、`localStorage`、环境变量。
- 证据引用：`[证据: E3(A), E4(A), E9(A), E10(B)]`
- 置信度：`高`

### FR-003 流式对话与本地会话恢复

- 用户价值：
  - 用户可与模型连续对话，并在刷新或重新挂载后恢复已保存的对话历史。
- 前置条件：
  - Provider 已配置完成。
- 触发方式：
  - 用户提交消息。
- 主流程：
  - `sendMessage()` 对输入做 `trim`，配置不足时弹出配置面板。
  - 调用 `@ai-sdk/vue` 的 `Chat` 实例发送消息。
  - 传输层将历史消息和系统提示词转换为 OpenAI-compatible `messages` 数组，请求 `/chat/completions` 流式接口。
  - 流式响应被转换为 `UIMessageChunk`，最终在面板中呈现为用户消息与助手消息。
  - 对话消息变化后，系统将用户/助手纯文本消息持久化到用户 scope 下的 `localStorage`。
  - 再次初始化时，从本地恢复消息列表。
- 异常流程：
  - HTTP 非 2xx 时显示错误弹窗。
  - 响应体为空时抛出“没有返回可读取数据流”错误。
  - 本地会话解析失败时删除损坏数据。
- 业务规则：
  - 当前持久化的会话只保留 `id + role + text`，不保留结构化消息部件。
  - 当前即使存在结构化卡片和确认态，刷新后也只恢复文本，不恢复卡片与动作。
- 数据变化：
  - 写入 `ai-chat-conversation:<scopeId>`。
- 权限要求：
  - 无额外代码证据。
- 依赖项：
  - `Chat`、`OpenAiCompatibleChatTransport`、`localStorage`。
- 证据引用：`[证据: E3(A), E5(A), E9(A), E10(A)]`
- 置信度：`高`

### FR-004 面板交互与错误处理

- 用户价值：
  - 用户可以清空对话、停止生成、重试失败请求，并在流式过程中保持较稳定的滚动体验。
- 前置条件：
  - AI 对话面板已打开。
- 触发方式：
  - 用户点击工具栏按钮或流式过程中滚动面板。
- 主流程：
  - 工具栏显示模型标签、会话状态、配置、新建对话、停止生成等动作。
  - 面板支持空态引导，提供预置 prompt。
  - 发送消息后支持流式显示“思考中”和助手输出。
  - 用户上滑阅读旧内容时，自动滚动会停止；再次发送新消息会重新回到底部。
  - 出错时弹出错误提示，并在可重试时提供“重试”按钮。
- 异常流程：
  - 没有可重试上下文时，错误弹窗只允许关闭。
- 业务规则：
  - 清空对话会同时清除本地持久化对话。
  - 未配置 Provider 时，设置弹窗不可关闭。
- 数据变化：
  - 清理对话消息、更新滚动状态、更新错误状态。
- 权限要求：
  - 无额外代码证据。
- 依赖项：
  - `AiChatPanel`、`AiChatToolbar`、`AiChatErrorBanner`、`AiChatEmptyState`。
- 证据引用：`[证据: E2(A), E3(A), E10(A)]`
- 置信度：`高`

### FR-005 当前对话链路的上下文注入与边界

- 用户价值：
  - 当前实现不会盲读全量本地数据，而是注入少量高价值界面上下文，帮助模型理解当前目标。
- 前置条件：
  - 用户已打开 AI 对话并配置 Provider。
- 触发方式：
  - 用户提交任意消息。
- 主流程：
  - `GlobalSearch` 在发送 AI 消息时会组装 `AiChatRequestContext`。
  - 上下文包含当前入口、当前路由、活动目录、活动笔记、候选笔记、最近笔记。
  - transport 会把该上下文转换为额外的 system prompt，并和默认系统提示词、历史对话消息一起发送给模型。
- 异常流程：
  - 当上下文字段为空或无法规范化时，系统退化为默认系统提示词 + 历史对话消息。
- 业务规则：
  - 当前上下文是“有限摘要注入”，而不是把全量笔记、完整目录树或 IndexedDB 原始数据直接送模。
  - 上下文仅用于理解界面和帮助定位目标，不等于用户已确认执行对象。
- 数据变化：
  - 无额外本地数据变更。
- 权限要求：
  - 无额外代码证据。
- 依赖项：
  - `AiChatRequestContext`、`buildAiChatContextSystemPrompt()`、`buildOpenAiMessages()`。
- 证据引用：`[证据: E2(A), E5(A), E13(A)]`
- 置信度：`高`

### FR-006 AI 笔记工具执行器

- 用户价值：
  - 仓库中已具备面向 AI 的笔记操作执行器，可作为后续“可执行 AI 代理”基础。
- 前置条件：
  - 调用方显式构造 `AiNoteToolCall`。
- 触发方式：
  - 调用 `executeToolCall()` 或 `executeToolCalls()`。
- 主流程：
  - 支持的工具包括：`search_notes`、`get_note_detail`、`list_folders`、`create_note`、`update_note`、`move_note`、`delete_note`、`set_note_lock`。
  - 对 `move_note`、`delete_note`、`set_note_lock` 默认要求确认，可通过 `dryRun` / `confirmed` 控制预览和执行。
  - 对写操作执行成功后，会尝试调用同步流程 `sync(true)`。
- 异常流程：
  - 搜索词为空返回 `invalid_query`。
  - 目标笔记不存在时返回 `note_not_found`。
  - 开启锁但缺少全局 PIN 时返回 `pin_required` 且标记 `humanActionRequired`。
- 业务规则：
  - 工具执行器已经实现预览对象、影响范围和确认需求判断。
  - 删除当前实现是软删除语义。
- 数据变化：
  - 可能新增/更新/移动/删除/锁定笔记，并在成功后排队同步。
- 权限要求：
  - 当前代码未见额外角色权限判断，默认复用现有笔记写接口能力。
- 依赖项：
  - `useNoteWrite`、`useNoteMove`、`useNoteDelete`、`useNoteLock`、`useSync`。
- 证据引用：`[证据: E6(A), E8(A), E11(A)]`
- 置信度：`高`

### FR-007 工具确认会话

- 用户价值：
  - 为高风险工具调用提供“先预览、后确认执行”的会话态管理。
- 前置条件：
  - 调用方显式使用 `useAiChatSession()`。
- 触发方式：
  - 提交一组 `AiNoteToolCall`。
- 主流程：
  - `submitToolCalls()` 先执行工具调用，收集预览结果。
  - 若任一结果需要确认，则写入 `pendingExecution` 并缓存 `lastResults`。
  - 用户确认后，系统会将 `confirmed=true` 回写到所有待执行调用，再真正执行一次。
- 异常流程：
  - 无待确认任务时，`confirmPendingExecution()` 返回空数组。
  - 用户可主动取消待确认任务。
- 业务规则：
  - 当前确认会话是通用流程组件，不依赖具体 UI。
- 数据变化：
  - 更新 `pendingExecution` 和 `lastResults`。
- 权限要求：
  - 无额外代码证据。
- 依赖项：
  - `useAiNoteCommand()`。
- 证据引用：`[证据: E7(A), E8(A)]`
- 置信度：`高`

### FR-008 当前入口与工具链路的集成状态

- 用户价值：
  - 明确当前入口已经不是纯聊天，而是“聊天 + 本地工具 + 结构化回执”的组合能力。
- 前置条件：
  - 用户通过当前首页 AI 对话入口使用功能。
- 触发方式：
  - 用户在首页 AI 模式下发送消息。
- 主流程：
  - 入口调用 `useAiChat().sendMessage()` 发送消息，并携带请求上下文。
  - 当助手消息内容可解析为工具 envelope 时，`useAiChat()` 会把 `toolCalls` 交给 `useAiChatSession()` 执行。
  - 执行结果会被汇总为文本摘要，同时转换成结构化卡片挂载到对应助手消息。
  - 若存在待确认操作，`AiChatPanel` 会显示确认区，用户可确认执行或取消。
  - 若卡片项包含对象动作，`AiChatPanel -> GlobalSearch -> HomePage` 会把动作继续冒泡，最终打开对应备忘录或文件夹。
- 异常流程：
  - 当助手输出不是合法 envelope 时，系统按普通文本回答处理。
  - 当工具结果需要人工步骤时，会以 warning 卡片和文本摘要告知，而不是静默失败。
- 业务规则：
  - 当前集成方式仍依赖模型按约定输出 JSON，而不是使用服务端托管的原生 function calling。
  - 当前结果卡片属于运行时 UI 元数据，不会进入本地会话持久化。
- 数据变化：
  - 当前入口会触发本地工具执行、更新确认会话状态，并在消息态中挂载运行时卡片。
- 权限要求：
  - 无额外代码证据。
- 依赖项：
  - `useAiChat()`、`parseAiAssistantToolEnvelope()`、`useAiChatSession()`、`AiChatPanel`、`GlobalSearch`、`HomePage`。
- 证据引用：`[证据: E2(A), E3(A), E7(A), E12(A), E14(A), E15(A), E16(A)]`
- 置信度：`高`

## 8. 数据与接口

### 8.1 核心数据对象

- 对象名：`OpenAiCompatibleChatSettings`
- 字段与约束：
  - `apiKey:string`
  - `baseUrl:string`
  - `model:string`

## 9. 验收标准

- AC-001（关联 FR-002）：
  - Given 用户切换到 AI 对话模式且 Provider 未配置
  - When 面板打开
  - Then 系统展示“配置直连模型”弹窗，并要求填写 `Base URL / Model / API Key`

- AC-002（关联 FR-003）：
  - Given 用户已配置可用 Provider
  - When 用户发送一条消息
  - Then 系统以流式方式展示助手回复并保留用户消息

- AC-003（关联 FR-003）：
  - Given 用户之前已有本地保存的 AI 对话
  - When 页面重新挂载并再次打开 AI 对话面板
  - Then 系统恢复本地会话内容

- AC-004（关联 FR-006）：
  - Given 调用 `delete_note` 工具且未确认
  - When 执行工具调用
  - Then 返回 `confirmation_required`，并附带预览信息

- AC-005（关联 FR-006）：
  - Given 对锁定笔记执行开启锁操作且缺少全局 PIN
  - When 工具执行完成
  - Then 返回 `pin_required`，并标记 `humanActionRequired=true`

- AC-006（关联 FR-008）：
  - Given 用户通过当前首页 AI 对话入口发送消息
  - When 助手返回合法工具 envelope
  - Then 当前入口会触发本地工具执行，并把结果回写为文本摘要和结构化卡片

- AC-007（关联 FR-008）：
  - Given AI 搜索结果卡片中存在备忘录或文件夹动作
  - When 用户点击卡片动作
  - Then 页面应打开对应备忘录或切换到对应文件夹

## 11. 风险与未决问题

### 11.1 风险清单

- RISK-001：
  - 当前首页入口已接入本地工具执行，但协议依赖模型输出约定 JSON，若格式漂移，工具链会退化为普通文本回答。[证据: E3, E5, E12]
- RISK-002：
  - 当前上下文注入仍是有限摘要，若目标定位依赖更丰富上下文，模型仍可能需要补问或再次检索。[证据: E2, E13]
- RISK-003：
  - 浏览器直连模式下 API Key 保存在本地浏览器，存在安全风险，组件本身已提示仅适用于受信任环境。[证据: E4]
- RISK-004：
  - 当前 transport 仅适配 OpenAI-compatible 文本流，不支持流重连和服务端原生工具调用，后续若迁移到服务端代理仍有改造成本。[证据: E5]
- RISK-005：
  - 当前对话持久化只保存纯文本，刷新后会丢失结构化卡片、确认态和对象动作，影响会话连续性。[证据: E3]
- RISK-006：
  - 当前虽已覆盖工具链入口与对象打开动作，但仍缺少更大范围的端到端持久化与失败补偿验证。[证据: E10, E11, E16]

### 11.2 未决问题

- Q-001（所需决策人：产品/研发）：
  - 当前前端本地编排模式是否继续保留，还是后续迁移到服务端代理 / 原生 function calling？
- Q-002（所需决策人：产品/安全）：
  - 浏览器直连模式是否接受长期保留，还是后续必须迁移到服务端代理？
- Q-003（所需决策人：研发）：
  - 当前确认交互已经由聊天面板承载，后续是否还需要抽离成更独立的确认 UI 或弹层？
- Q-004（所需决策人：研发）：
  - 当前纯文本持久化是否要升级为结构化消息持久化，以支持工具结果卡片、确认态与对象动作恢复？
