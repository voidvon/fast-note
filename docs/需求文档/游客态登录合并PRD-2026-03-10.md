# 游客态备忘录登录后处理 PRD（合并 / 覆盖 / 共存）

## 1. 文档信息

- 文档标题：游客态备忘录登录后处理 PRD（合并 / 覆盖 / 共存）
- 版本：v0.3
- 日期：2026-03-10
- 作者：Codex
- 输入原文：
  - “现在未登录用户和登录用户连接的indexeddb数据库是不同的，但是按照常理来说，未登录用户在登录之后会想要未登录状态下的备忘录。所以需要在登录之后，如果有记录就弹出合并、覆盖、共存（退出登录之后重连indexeddb数据库能恢复未登录的备忘录）”
  - “去重规则按照时间对比，谁比较新就用谁的，如果一样就使用云端的；覆盖是全部；每次登录之后，如果未登录状态下有数据，都需要弹窗提醒；使用 id 匹配。”
- 需求粒度：`功能级`

## 2. 原始需求摘要

- 原始事实：
  - 未登录用户与登录用户使用不同的 IndexedDB 数据库。
  - 登录后需要提示用户对游客态数据进行处理。
  - 处理方式包含：合并、覆盖、共存。
  - 共存时，退出登录应可恢复游客态备忘录。
  - 合并匹配键为 `id`，冲突以 `updatedAt` 新者为准；相同时间戳保留登录态（云端）。
  - 覆盖为全量覆盖。
  - 每次登录只要游客态有数据，必须弹窗提醒。
- 目标问题：
  - 解决游客态创建的备忘录在登录后不可见导致的“数据断层”。
- 期望结果：
  - 登录后对游客态数据进行可控处理，用户明确理解结果并可恢复或保留数据。

## 3. 背景与问题陈述

- 当前背景：
  - 登录态与游客态数据隔离，导致数据无法自动衔接。
- 要解决的问题：
  - 登录后的数据割裂与“误丢失感”。
- 本次目标：
  - 登录后自动检测游客态数据并提供合并 / 覆盖 / 共存选择。
  - 明确且可验收的数据合并与冲突规则。
- 非目标：
  - 不做复杂的正文级冲突编辑器。
  - 不涉及服务端同步体系重构。
  - 不涉及跨设备游客数据迁移。

## 4. 用户与场景

- 用户角色：
  - 游客用户（未登录状态使用过备忘录）
  - 已登录用户
- 核心场景：
  - 用户未登录创建备忘录后登录，系统检测到游客数据并提示处理。
- 使用时机：
  - 每次登录成功后，只要游客态有数据即触发弹窗。

## 5. 范围定义

### 5.1 In Scope

- I1：登录后检测游客态数据库是否存在记录。
- I2：检测到记录后弹窗提示合并 / 覆盖 / 共存。
- I3：合并逻辑按 `id` 匹配，冲突按 `updatedAt` 处理。
- I4：覆盖为全量覆盖。
- I5：共存保留两套数据，登出后恢复游客态数据库。
- I6：异常处理、失败提示与幂等保护。

### 5.2 Out of Scope

- O1：文本细粒度冲突合并。
- O2：跨设备游客数据同步。
- O3：服务端历史数据回滚。

## 6. 功能需求

### FR-001 登录后检测游客态数据

- 来源：`原始事实`
- 用户价值：避免数据断层与“误丢失感”。
- 前置条件：用户登录成功。
- 触发方式：登录成功后初始化阶段。
- 主流程：
  - 读取游客态数据库记录数。
  - 若存在记录，进入弹窗决策流程。
  - 若无记录，进入登录态正常流程。
- 异常流程：读取失败时提示错误并允许进入登录态。
- 业务规则：只要游客态有数据，每次登录都触发。
- 数据变化：无。
- 依赖与约束：需可访问游客态 IndexedDB。
- 待确认点：游客态“有数据”的判定范围是否仅限备忘录，或包含文件夹/标签/附件等（`推断`）。

### FR-002 登录后提示处理弹窗

- 来源：`原始事实`
- 用户价值：用户明确知晓并选择数据处理方式。
- 前置条件：FR-001 检测到游客态存在数据。
- 触发方式：登录成功后立即弹出。
- 主流程：
  - 展示说明文案与三个处理选项。
  - 用户选择后进入对应处理流程。
- 异常流程：弹窗加载失败时记录错误并允许进入登录态。
- 业务规则：覆盖需二次确认。
- 数据变化：无。
- 待确认点：弹窗关闭/取消时的默认处理（建议保持未处理状态并下次登录继续提示）。

### FR-003 合并游客态与登录态数据

- 来源：`原始事实`
- 用户价值：保留游客态内容并与登录态数据融合。
- 前置条件：用户选择“合并”。
- 触发方式：确认合并操作。
- 主流程：
  - 按 `id` 匹配游客态与登录态记录。
  - 若无同 `id`，直接写入登录态。
  - 若同 `id`，比较 `updatedAt`，保留最新；相同时间戳保留登录态。
- 异常流程：迁移失败则提示失败并保持原数据不变。
- 业务规则：冲突按 `updatedAt` 最新覆盖旧的；相同时保留登录态。
- 数据变化：登录态新增或更新记录。
- 依赖与约束：需要登录态数据库可写。
- 待确认点：关联对象（文件夹/标签/附件/草稿）是否按相同规则合并（`推断`）。

### FR-004 覆盖登录态数据（全量）

- 来源：`原始事实`
- 用户价值：当游客态为最新或主要数据时快速替换登录态。
- 前置条件：用户选择“覆盖”并确认。
- 触发方式：二次确认覆盖。
- 主流程：
  - 清空登录态本地数据。
  - 将游客态全量数据写入登录态。
- 异常流程：失败时回滚或保持原登录态数据并提示。
- 业务规则：覆盖范围为全量（含关联数据）。
- 数据变化：登录态数据被游客态全量替换。
- 依赖与约束：需支持批量删除与批量写入。

### FR-005 共存两套数据

- 来源：`原始事实`
- 用户价值：用户可分别保留游客态与登录态数据。
- 前置条件：用户选择“共存”。
- 触发方式：确认共存。
- 主流程：
  - 登录态数据不变。
  - 游客态数据保留不迁移。
  - 退出登录后重连游客态数据库并展示游客数据。
- 异常流程：登出重连失败时提示并记录错误。
- 业务规则：共存不做自动迁移。
- 数据变化：无跨库迁移。

### FR-006 处理失败与幂等

- 来源：`推断`
- 用户价值：避免重复执行导致数据异常。
- 前置条件：执行合并 / 覆盖 / 共存。
- 触发方式：处理流程中发生异常或重复触发。
- 主流程：
  - 对处理流程加幂等保护。
  - 失败时提示错误并保持原数据。
- 业务规则：处理失败不应导致“部分可见、部分丢失”。
- 数据变化：失败时不提交数据写入。

## 7. 非功能需求

- 性能：登录后检测不应明显阻塞进入首页（阈值 `[待确认]`）。
- 可用性：覆盖需二次确认，弹窗文案清晰可理解。
- 安全与权限：仅本地数据处理，不影响其他账号。
- 兼容性：需兼容当前支持的浏览器/WebView IndexedDB 行为。
- 可观测性：记录检测结果、用户选择、处理成功/失败。
- 数据一致性：合并/覆盖过程需保证事务性或可回滚策略（`待确认`）。

## 8. 验收标准

- AC-001（关联 FR-001）：
  - Given 游客态存在任意记录
  - When 用户登录成功
  - Then 必须弹出处理弹窗

- AC-002（关联 FR-003）：
  - Given 游客态与登录态存在同 `id` 记录
  - When 用户选择合并
  - Then 保留 `updatedAt` 最新记录，若相同时间戳保留登录态

- AC-003（关联 FR-003）：
  - Given 游客态存在新增 `id`
  - When 用户选择合并
  - Then 登录态新增该记录

- AC-004（关联 FR-004）：
  - Given 登录态已有数据
  - When 用户选择覆盖并确认
  - Then 登录态被游客态数据全量替换（含关联数据）

- AC-005（关联 FR-005）：
  - Given 用户选择共存
  - When 退出登录
  - Then 恢复游客态数据库并展示游客数据

- AC-006（关联 FR-006）：
  - Given 合并或覆盖执行失败
  - When 出现异常
  - Then 提示错误并保持原数据不变

## 9. 风险与未决问题

### 9.1 风险

- RISK-001：覆盖为不可逆操作，若文案不清易造成误操作。
- RISK-002：合并依赖 `updatedAt`，若时间戳不可信可能导致错误覆盖。
- RISK-003：批量处理数据量大时存在性能风险。

### 9.2 待确认问题

- Q-001：关联数据对象清单（文件夹/标签/附件/草稿）是否存在且必须纳入合并与覆盖。
- Q-002：处理流程是否需要事务性保障或显式回滚策略。
