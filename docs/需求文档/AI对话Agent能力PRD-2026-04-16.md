# AI 对话 Agent 能力 PRD

## 1. 文档信息

- 文档标题：AI 对话 Agent 能力 PRD
- 版本：v0.1
- 日期：2026-04-16
- 作者：Codex
- 输入原文：
  - “在ai对话中，优化ai对话的流程和工具方法连续调用，这能否成为agent”
  - “我们先编写agent的需求文档吧，边做边改不行，容易出很多问题”
- 需求粒度：`版本级`

## 2. 原始需求摘要

- 原始事实：
  - 当前 AI 对话已经具备一定的本地工具调用能力，但流程仍不稳定。
  - 用户希望先形成正式需求文档，再继续实现，避免边改边出问题。
  - 用户关注“AI 对话流程优化”和“工具方法连续调用”，并希望判断这是否能成为 Agent。
- 目标问题：
  - 当前 AI 对话能力更像“带工具的聊天助手”，缺少对任务状态、连续执行、失败恢复、结果收口的完整定义。
  - 没有正式需求文档，导致实现过程中容易混淆“当前已做能力”“下一步目标能力”“暂不做能力”。
- 期望结果：
  - 形成一份可评审的 Agent 能力 PRD，定义 fastnote 中 AI 对话升级为 Agent 的范围、流程、规则、验收标准与风险边界。

## 3. 背景与问题陈述

- 当前背景：
  - fastnote 已经存在首页 AI 对话入口、上下文注入、本地工具调用、确认执行和结构化结果卡片。
  - 当前实现已经不是纯聊天，但仍主要依赖模型单轮返回工具请求，再由前端执行并拼接结果继续对话。
  - 最近的问题已经暴露出当前模式的脆弱性，例如读取备忘录详情后对话停止、链接解析与续跑链路需要额外补丁才能工作。
- 要解决的问题：
  - 缺少统一的 Agent 定义，导致“是不是 Agent”没有边界。
  - 缺少任务生命周期定义，导致多步执行链路不可预测。
  - 缺少稳定的连续工具调用协议，导致工具执行后容易停在中间态。
  - 缺少对本地数据读取、保存、刷新恢复、失败重试的统一规则。
  - 缺少对高风险写操作、批量操作、自动继续执行的边界约束。
- 本次目标：
  - 定义 fastnote 中 AI 对话升级为 Agent 的产品边界。
  - 定义 Agent 的任务状态、连续工具调用、确认机制、结果回执和恢复能力。
  - 明确 Agent 的数据源优先级、执行规则和 UI 呈现方式。
  - 为后续技术方案、交互设计、任务拆解提供稳定输入。
- 非目标：
  - 不在本次文档中定义通用网页浏览 Agent、系统级自动化 Agent、跨应用操作 Agent。
  - 不在本次文档中直接确定最终模型供应商、服务端架构或 Prompt 文案细节。
  - 不默认开放无人确认的高风险批量删除、批量覆盖、自动外发内容。

## 4. 用户与场景

- 用户角色：
  - 普通记录用户：希望用一句自然语言完成查找、读取、改写、整理笔记。
  - 重度知识管理用户：希望 AI 能连续完成“定位对象 -> 读取内容 -> 分析处理 -> 执行修改 -> 返回结果”。
  - [推断] 研发/测试用户：需要验证 Agent 的状态流转、续跑逻辑和异常恢复是否稳定。
- 核心场景：
  - 用户给出一条备忘录链接，让 AI 读取内容后继续重写、总结或提炼。
  - 用户要求 AI 在多步任务中连续调用多个本地工具，而不是执行一步就停下。
  - 用户要求 AI 对当前笔记或当前目录下的内容做定位、分析、改写、移动、重命名等连续操作。
  - 用户在执行过程中刷新页面，仍希望恢复任务状态和当前路由上下文。
- 使用频率 / 时机：
  - 高频发生在日常记录、整理、复盘、内容改写场景。
  - [待确认] 批量整理、批量归类、批量修订是否属于首期高频场景。

## 5. 范围定义

### 5.1 In Scope

- I1：定义 fastnote Agent 的产品边界与目标能力。
- I2：定义 Agent 任务生命周期与前端可见状态。
- I3：定义连续工具调用机制，包括多步读取、分析、执行、收口。
- I4：定义对象定位机制，包括当前上下文、URL、当前页面对象、本地搜索结果。
- I5：定义本地数据读取优先级、写入确认、执行回执与刷新恢复。
- I6：定义 Agent 在 UI 中的任务呈现、进度反馈、确认交互与失败恢复。
- I7：定义首期可支持的任务类型与暂不支持的任务类型。

### 5.2 Out of Scope

- O1：通用互联网搜索、第三方 SaaS 操作、浏览器自动操作等跨应用 Agent 能力。
- O2：后台长期运行任务、定时任务、无人值守自动处理。
- O3：服务端统一编排平台、多模型路由平台、企业级审计后台。
- O4：首期开放复杂批量规则执行、无人确认的大范围写操作。

### 5.3 Agent 定义

- 原始事实：
  - 用户希望“优化 AI 对话流程和工具方法连续调用”，并希望它“成为 agent”。
- 推断：
  - fastnote 中的 Agent 不应定义为“无限自主”，而应定义为“以用户目标为中心、能在本地数据域内完成多步任务闭环的任务代理”。
- 本文定义：
  - Agent = `目标理解 + 对象定位 + 分步执行 + 状态可见 + 可确认写入 + 失败可恢复 + 结果可收口`
  - 若仅具备“用户说一句、模型调用一个工具、返回一次文本”的能力，则仍属于“带工具的助手”，不属于完整 Agent。

### 5.4 首期版本范围

- M1：单任务单线程 Agent。
- M2：围绕本地笔记与文件夹的多步连续执行。
- M3：支持读取型任务、改写型任务、单对象写入型任务。
- M4：支持链接驱动任务，例如用户直接粘贴 `/n/<noteId>`。
- M5：支持刷新恢复未完成会话与待确认执行态。
- M6：所有读取默认本地优先，不阻塞等待远端认证刷新或同步完成。
- M7：首期任务边界限定为“单对象、多步、可确认”的本地任务，不纳入多对象批量任务。

## 6. 功能需求

### FR-001 Agent 任务会话模型

- 来源：`推断`
- 用户价值：
  - 用户可以明确看到当前 AI 正在执行的是“一个任务”，而不是随机输出若干段回复。
- 前置条件：
  - 用户已进入 AI 对话并提交一条任务型请求。
- 触发方式：
  - 用户输入“读取这个备忘录并帮我重写”“把当前笔记移动到工作文件夹并总结重点”等目标型指令。
- 主流程：
  - 系统创建一个 Agent 任务会话。
  - 任务会话记录任务目标、当前阶段、执行步骤、涉及对象、风险级别和最终结果。
  - 同一条用户消息对应一个主任务；后续工具调用和续跑都归属于该任务。
- 异常流程：
  - 若任务无法识别为可执行目标，则降级为普通问答，不创建 Agent 执行态。
- 业务规则：
  - Agent 任务状态至少包括 `理解中`、`定位中`、`执行中`、`待确认`、`已完成`、`失败`、`已取消`。
  - 一个任务在完成前，不应被误判为新的无关消息片段。
- 数据变化：
  - 新增任务级会话状态，并与现有聊天消息建立关联。
- 依赖与约束：
  - 依赖现有 AI 对话消息存储，但需增加任务维度。
- 待确认点：
  - 一个会话内是否允许多个任务并行存在。

### FR-002 Agent 计划与步骤可视化

- 来源：`推断`
- 用户价值：
  - 用户能知道 AI 下一步要做什么，避免“读取完就停了”但不知道是失败还是结束。
- 前置条件：
  - Agent 已识别为可执行任务。
- 触发方式：
  - 任务进入执行链路。
- 主流程：
  - Agent 先生成最小可执行计划。
  - 界面展示当前步骤，如“已识别对象”“正在读取笔记”“正在生成改写结果”“等待你确认写回”。
  - 每个步骤执行后更新状态。
- 异常流程：
  - 如果某一步失败，界面明确显示失败步骤和原因，而不是只剩静默。
- 业务规则：
  - 首期计划不要求复杂推理树，只要求用户能看到当前阶段和下一动作。
  - 计划可以动态收缩，不要求一次性展示全部内部推理。
- 数据变化：
  - 记录任务步骤列表、步骤状态和最后更新时间。
- 依赖与约束：
  - 不暴露完整 chain-of-thought，只暴露可执行阶段描述。
- 待确认点：
  - 是否需要展示“预计还需 1-2 步”之类的弱进度提示。

### FR-003 对象定位与上下文解析

- 来源：`原始事实 + 推断`
- 用户价值：
  - 用户可以直接通过当前页面、当前选中对象、fastnote 链接或自然语言描述驱动 Agent 找到目标对象。
- 前置条件：
  - 用户请求涉及具体笔记、文件夹或当前页面对象。
- 触发方式：
  - 用户输入包含当前对象、标题关键词、目录名、fastnote 链接或路径的请求。
- 主流程：
  - Agent 优先解析当前页面上下文和消息中的显式对象。
  - 当消息中包含 `/n/<noteId>` 或完整 fastnote 笔记链接时，直接提取 `noteId`。
  - 当对象不唯一时，先返回候选对象，不直接写入。
  - 当对象唯一时，进入读取或执行阶段。
- 异常流程：
  - 对象不存在、已删除或无权限读取时，返回结构化失败结果。
- 业务规则：
  - 显式链接优先级高于模糊搜索结果。
  - 当前活动笔记/活动目录可作为默认上下文，但不能替代明确对象定位。
  - 写操作必须建立在对象唯一定位成功之后。
- 数据变化：
  - 记录任务命中的对象列表、对象来源和置信级别。
- 依赖与约束：
  - 依赖现有上下文注入、本地搜索能力和 `get_note_detail` 等工具。
- 待确认点：
  - 首期是否支持一次请求同时定位多个对象并进入批量任务。

### FR-004 连续工具调用与自动续跑

- 来源：`原始事实`
- 用户价值：
  - 用户发出目标型请求后，Agent 能连续完成“先读取，再分析，再回答或执行”，而不是在读取后停住。
- 前置条件：
  - Agent 已进入执行任务态，且当前任务需要多个工具或多轮执行。
- 触发方式：
  - 用户请求包含多步目标，例如“读取这条笔记并帮我重写”“找到工作文件夹中的会议纪要并总结”。
- 主流程：
  - 模型返回结构化工具调用。
  - 前端执行本地工具。
  - 系统把工具结果以结构化方式回传给模型，自动续跑下一步。
  - 直到得到最终自然语言答复、进入待确认写入，或达到安全终止条件。
- 异常流程：
  - 工具返回空数据、对象不存在、结果格式不合法或超过最大续跑深度时，任务进入失败或降级答复态。
- 业务规则：
  - 续跑输入必须包含工具执行结果的有效载荷，不能只传摘要。
  - `get_note_detail` 等读取型工具返回的正文内容，必须能被下一轮继续消费。
  - 首期连续调用深度应有限制，避免死循环。
  - 当命中待确认写操作时，自动续跑应暂停，等待用户确认。
- 数据变化：
  - 记录每一步工具请求、工具结果、续跑次数和终止原因。
- 依赖与约束：
  - 依赖结构化工具协议、执行器和任务会话状态机。
- 待确认点：
  - 最大连续调用深度、超时阈值和降级策略。

### FR-005 本地优先数据读取策略

- 来源：`原始事实 + 推断`
- 用户价值：
  - 页面刷新或新开页面时，Agent 能快速读取本地数据并开始工作，不被远端鉴权或同步阻塞。
- 前置条件：
  - 目标对象已定位成功。
- 触发方式：
  - Agent 需要读取笔记详情、目录信息、最近上下文或会话状态。
- 主流程：
  - Agent 读取数据时优先使用当前全局状态和本地可用缓存。
  - 若全局状态已有目标对象，直接用于展示和执行。
  - 若全局状态暂未就绪但本地缓存可读，可从本地缓存恢复。
  - 远端认证刷新和同步在后台进行，不阻塞首屏读取。
- 异常流程：
  - 本地没有命中且对象确实不存在时，返回对象不存在，而不是长时间黑屏等待。
- 业务规则：
  - 本地可展示数据优先于远端刷新完成。
  - Agent 不应为了读取一条本地笔记而强依赖 `/api/collections/users/auth-refresh` 成功返回。
  - 远端同步结果只能做后续纠偏，不能覆盖当前任务的即时本地可见结果。
- 数据变化：
  - 可能触发本地恢复、后台同步、数据状态纠偏。
- 依赖与约束：
  - 依赖 note store、本地缓存和现有同步机制。
- 待确认点：
  - 本地状态与持久化缓存的冲突优先级是否需要更细规则。

### FR-006 写操作确认与提交规则

- 来源：`推断`
- 用户价值：
  - 用户既能享受 Agent 自动化，又不会被误写、误删、误移动。
- 前置条件：
  - Agent 已获得目标对象并准备执行写操作。
- 触发方式：
  - 用户请求改写原文、更新标题、移动目录、删除笔记、修改锁定状态等。
- 主流程：
  - Agent 识别风险等级。
  - 低风险操作可在明确指令后直接执行。
  - 中高风险操作先展示预览与影响范围。
  - 用户确认后才执行正式写入，并回写结果。
- 异常流程：
  - 写入失败、版本冲突或对象状态变化时，返回失败原因和建议动作。
- 业务规则：
  - 所有写操作统一走现有本地写接口，不允许模型直接改前端状态。
  - 改写类任务至少支持“仅返回建议文本”和“直接写回原文”两种结果。
  - 当用户指令明确要求“直接改写”“直接覆盖”“直接写回”时，允许 Agent 在完成改写后进入写回流程。
  - 批量操作默认进入预览，不允许直接执行。
- 数据变化：
  - 更新笔记内容、标题、目录归属、锁状态或删除状态。
- 依赖与约束：
  - 依赖现有 `update_note`、`move_note`、`delete_note`、`set_note_lock` 等执行器。
- 待确认点：
  - 单条正文重写是否在用户指令足够明确时允许默认直接写回。

### FR-007 结果收口与后续动作建议

- 来源：`推断`
- 用户价值：
  - 用户在任务结束时能看到清晰结果，而不是一堆中间执行日志。
- 前置条件：
  - Agent 任务已完成或已中止。
- 触发方式：
  - 最后一步执行完成，或无法继续执行。
- 主流程：
  - Agent 返回一条最终结果消息。
  - 结果包含：完成了什么、影响了哪些对象、失败了什么、是否还需要用户下一步动作。
  - 对于读取或总结类任务，返回基于读取内容生成的最终答案。
  - 对于写入类任务，返回变更摘要和可继续动作。
- 异常流程：
  - 如果任务在中途停止，也要返回“停止在第几步、为什么停止、可以怎么继续”。
- 业务规则：
  - 任务结束消息必须区别于中间状态消息。
  - 结果卡片和结果文本都应支持刷新后恢复。
- 数据变化：
  - 新增结果消息、结果卡片和结束态。
- 依赖与约束：
  - 依赖现有卡片体系与本地会话持久化。
- 待确认点：
  - 是否需要支持“一键继续这个失败任务”。

### FR-008 刷新恢复与路由保持

- 来源：`原始事实 + 推断`
- 用户价值：
  - 用户从 `/n/<noteId>` 进入、刷新或新页面打开后，Agent 和页面上下文都能保持一致，不会被重置回 `/home`。
- 前置条件：
  - 用户从具体笔记或文件夹路由进入，并打开 AI 对话。
- 触发方式：
  - 页面刷新、直接访问深链接、从新页面打开链接。
- 主流程：
  - 页面优先按当前深链接恢复对象。
  - AI 会话恢复时保留当前任务关联的对象 ID、路由路径和面板模式。
  - 恢复后继续以当前路由对象作为上下文，而不是强制跳回首页。
  - 若刷新前任务处于 `待确认`，恢复后保留预览结果与确认动作，允许用户继续确认或取消。
  - 若刷新前任务处于 `执行中`，恢复后默认转为 `已中断/待继续`，展示最后完成步骤与“继续任务”入口，但不自动继续执行。
- 异常流程：
  - 如果目标对象已不存在，则保留原路由并展示对象不可用状态，而不是无提示跳转。
- 业务规则：
  - PC 端深链接路由保持必须与移动端保持一致，但不能触发移动端页面跳转逻辑。
  - 原生 URL 更新不得触发额外路由动画。
  - Agent 恢复不得破坏当前页面对象。
  - 首期不建议在刷新后自动恢复 `执行中` 任务，避免重复工具调用、重复写入和状态不一致。
- 数据变化：
  - 恢复路由状态、AI 面板状态和任务对象关联。
- 依赖与约束：
  - 依赖现有 PC 路由保持逻辑和 AI 模式恢复逻辑。
- 待确认点：
  - 首期是否要求“待确认中的任务”在刷新后可直接继续执行。

### FR-009 失败处理与显式降级

- 来源：`推断`
- 用户价值：
  - 当 Agent 无法继续执行时，用户能明确知道卡在哪，而不是看到黑屏、空白或停顿。
- 前置条件：
  - 任务执行过程中出现异常。
- 触发方式：
  - 工具失败、续跑失败、对象缺失、版本冲突、响应超时、协议不合法等。
- 主流程：
  - 系统捕获失败并归类。
  - 返回可读失败信息。
  - 根据场景提供重试、选择候选对象、改为只返回建议文本、取消任务等动作。
- 异常流程：
  - 如果失败原因未知，至少记录 trace 信息并展示统一失败态。
- 业务规则：
  - 不允许在失败后静默停留在“读取完成但无后续”的半状态。
  - UI 层失败和 Agent 层失败要区分展示。
- 数据变化：
  - 记录失败码、失败步骤、重试次数和最后错误。
- 依赖与约束：
  - 依赖消息卡片、任务状态机和日志埋点。
- 待确认点：
  - 是否需要对部分失败类型自动重试一次。

### FR-010 Agent 能力分层与扩展边界

- 来源：`推断`
- 用户价值：
  - 团队可以按清晰边界迭代，不会把“首期 Agent”无限扩张。
- 前置条件：
  - 需要规划版本路线。
- 触发方式：
  - 进入研发排期或新增能力评审。
- 主流程：
  - 首期只支持本地笔记域内的单任务 Agent。
  - 第二阶段再考虑批量任务、摘要长期记忆、更多结构化引用能力。
  - 更高阶能力需要独立评审，不默认归入首期。
- 异常流程：
  - 当新增需求超出首期边界时，应回到 PRD 或技术方案层补充，而不是直接进入开发。
- 业务规则：
  - “能连续调用多个工具”是 Agent 的必要条件之一，但不是充分条件。
  - 只有同时具备任务状态、连续执行、确认约束、失败恢复和结果收口，才定义为本产品中的 Agent。
- 数据变化：
  - 无直接数据变化。
- 依赖与约束：
  - 依赖后续技术方案、开发计划与测试计划拆解。
- 待确认点：
  - 第二阶段是否纳入批量操作与摘要层。

## 7. 非功能需求

- 性能：
  - 首屏进入 `/n/<noteId>` 或 `/f/<folderId>` 时，不应因为 Agent 或远端认证刷新造成明显黑屏等待。
  - 本地对象已存在时，详情展示和 Agent 上下文恢复应在本地数据可读后立即完成。
  - 首期 Agent 单任务连续调用建议控制在有限步数内，避免长时间阻塞。
- 可用性：
  - Agent 任务状态必须用户可见。
  - 任何中断都必须有显式状态，不允许静默卡死。
  - 刷新后至少恢复最近一次任务结果、待确认态和对象上下文。
- 安全与权限：
  - 写操作必须经过前端受控执行器，不能让模型直接写状态。
  - 高风险操作必须确认。
  - API Key 和 Provider 配置仍沿用受信任环境前提；若后续改服务端代理，需要另立方案。
- 兼容性：
  - PC 与移动端应共享深链接语义，但保留各自页面实现。
  - PC 端 URL 更新不得引发 Ionic 路由动画副作用。
- 可观测性：
  - 需要记录任务 ID、步骤、工具调用、失败原因、终止原因。
  - [待确认] 是否首期就需要可视化调试面板。
- 数据一致性：
  - 读取遵循本地优先，写入遵循现有本地真相源与同步链路。
  - 后台同步不能覆盖当前用户刚确认的本地写入结果。

## 8. 验收标准

- AC-001（关联 FR-001）：
  - Given 用户发起“读取这个备忘录并帮我重写”的请求
  - When 系统识别为 Agent 任务
  - Then 界面应创建一个可见任务状态，而不是只显示普通聊天气泡

- AC-002（关联 FR-002）：
  - Given Agent 正在执行多步任务
  - When 已完成读取并进入改写阶段
  - Then 用户应能看到“读取完成”“正在生成改写结果”等阶段提示

- AC-003（关联 FR-003）：
  - Given 用户消息中包含 `http://localhost:8888/n/vdpDshkn5m3k`
  - When Agent 解析对象
  - Then 应直接识别出 noteId 并读取对应笔记，而不是要求用户再次手动指定

- AC-004（关联 FR-004）：
  - Given 用户要求“读取这条笔记并帮我重写”
  - When `get_note_detail` 工具返回正文内容
  - Then Agent 必须继续生成最终重写结果或进入下一步，而不是停在读取结果

- AC-005（关联 FR-005）：
  - Given 用户在 PC 端直接访问 `/n/<noteId>`
  - When 页面初始化
  - Then 详情页和 Agent 上下文应基于本地可用数据尽快展示，不应阻塞等待远端认证刷新

- AC-006（关联 FR-006）：
  - Given 用户要求“把这条笔记改写后直接覆盖原文”
  - When Agent 准备写回
  - Then 至少要给出写回预览或明确确认动作，未经确认不得执行高风险写入

- AC-007（关联 FR-007）：
  - Given Agent 任务已完成
  - When 用户查看最终结果
  - Then 应看到“完成了什么、影响了哪些对象、是否还需下一步动作”的结果收口

- AC-008（关联 FR-008）：
  - Given 用户从 `/n/<noteId>` 打开 AI 对话并刷新页面
  - When 会话恢复完成
  - Then 路由仍保持 `/n/<noteId>`，且 AI 不应把页面重置为 `/home`

- AC-009（关联 FR-008）：
  - Given 用户刷新前任务处于 `待确认`
  - When 页面恢复完成
  - Then 原预览内容和确认按钮仍可见，用户可继续确认或取消

- AC-010（关联 FR-008）：
  - Given 用户刷新前任务处于 `执行中`
  - When 页面恢复完成
  - Then 任务应恢复为 `已中断/待继续`，而不是自动重新执行未完成步骤

- AC-011（关联 FR-009）：
  - Given Agent 在工具续跑阶段失败
  - When 失败发生
  - Then 界面必须显示失败步骤、失败原因和可执行恢复动作

- AC-012（关联 FR-010）：
  - Given 新需求要求 Agent 自动处理多个对象并批量写入
  - When 该需求进入评审
  - Then 团队应能依据本 PRD 判断其超出首期范围并单独拆分

## 9. 风险与未决问题

### 9.1 风险

- RISK-001：
  - 如果继续沿用“模型自由输出 JSON”而没有更稳定的协议约束，连续工具调用仍可能频繁出错。
- RISK-002：
  - 如果任务状态机定义不清，UI 会继续出现“看起来像完成，实际只是中途停住”的问题。
- RISK-003：
  - 如果本地优先读取和后台同步的边界不清，可能再次出现黑屏等待、数据闪烁或路由回跳。
- RISK-004：
  - 如果首期同时纳入批量写入、撤销、长期记忆，需求复杂度会明显失控。

### 9.2 待确认问题

- Q-001：
  - 首期 Agent 已确认只支持单任务单线程，不支持并发任务；是否需要在文案中进一步限制为“单对象任务”。
- Q-002：
  - 单条正文改写已确认允许在用户明确指令下直接写回；是否仍需在 UI 中保留“先预览后写回”的可选入口。
- Q-003：
  - 刷新后待确认任务是否允许直接恢复到“可确认执行”状态。
- Q-004：
  - 是否需要在首期提供 Agent 调试视图，例如工具调用轨迹与失败 trace。
- Q-005：
  - 第二阶段是否纳入批量任务、摘要长期记忆与撤销能力。

## 10. 需求澄清建议

- 建议优先确认的 3 个问题：
  - 首期 Agent 的任务边界是否只做“单对象、多步、可确认”的本地任务。
  - 改写类任务默认产物是“建议文本”还是“直接写回”。
  - 刷新恢复时，待确认任务和执行中任务分别恢复到什么状态。
- 若确认后可能变化的模块：
  - `features/ai-chat`
  - `processes/ai-chat-session`
  - `features/ai-note-command`
  - `entities/note/model/state`
  - 路由与会话持久化相关模块
- 下一步建议动作：
  - 基于本 PRD 再输出一份“AI 对话 Agent 技术方案”。
  - 将首期 Agent 拆成“任务状态机”“连续工具调用协议”“本地读取与恢复”“确认与结果收口”四个研发模块。
  - 在技术方案确定前，暂停继续零散补丁式扩展 Agent 行为。
