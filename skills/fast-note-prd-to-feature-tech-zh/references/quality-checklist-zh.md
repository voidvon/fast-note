# 交付质量检查清单（功能拆分 + 技术方案）

## 1. 范围与结构

- 是否明确写出“范围内/范围外”。
- 是否按能力域 -> 功能点 -> 原子任务分层。
- 是否每个功能点都有唯一编号。
- 多文件交付时，是否包含“总览 + 分模块”且命名规范一致。

## 2. 证据链

- 是否每个功能点至少引用 1 条代码证据（文件 + 行号）。
- 是否明确区分“代码事实 / 推断 / 待确认”。
- 是否避免无证据的确定性结论。

## 3. 架构与落点

- 是否按层说明改动落点（UI/hooks/stores/database/pocketbase/router）。
- 是否遵守 Fast-Note 边界（UI 不直接调 PocketBase、核心数据经 store）。
- 是否说明 offline-first 语义是否受影响。

## 4. 数据与同步

- 是否说明 `Note` 契约或字段变化影响。
- 是否说明 Dexie schema/index 是否需要调整。
- 是否说明同步策略与冲突处理策略。
- 是否说明失败补偿与回滚策略。

## 5. 可实施性与可验收性

- 是否拆到可执行任务（任务粒度可分配给开发）。
- 是否提供明确验收标准（可测试）。
- 是否包含最小验证路径（手动冒烟或测试项）。

## 6. 风险与未决问题

- 是否列出关键风险和影响范围。
- 是否列出必须业务确认的未决问题。
- 是否给出建议决策顺序（先决问题优先）。

## 7. 跨文件一致性（多文件）

- 总览文档是否包含所有模块文档的相对路径链接。
- 模块文档中的编号与总览中的功能点映射是否一致。
- 跨模块依赖是否仅在总览维护，并在模块文档引用。
