# 代码信号到需求映射

把实现细节转换为需求语言时，优先用下表。

| 代码信号              | 常见位置                      | 可反推的需求要点                         | 输出注意事项                  |
| --------------------- | ----------------------------- | ---------------------------------------- | ----------------------------- |
| 路由定义、守卫        | router、middleware            | 页面入口、访问路径、鉴权前置条件         | 区分匿名可访问与登录后可访问  |
| 页面事件处理          | views、components             | 用户触发动作、交互顺序、反馈提示         | 不要把 UI 细节误写成业务规则  |
| Store/Service 方法    | stores、services、hooks       | 核心业务能力（创建、编辑、发布、同步等） | 同时提取成功与失败分支        |
| API 调用与 DTO        | api、client、types            | 请求参数约束、返回字段、错误语义         | 缺少文档时保持字段命名一致    |
| 数据模型与 schema     | database、model、types        | 实体定义、状态机、字段约束、索引要求     | 明确必填/可选/默认值          |
| 校验逻辑              | validator、schema、form rules | 输入合法性规则、边界条件                 | 把校验规则写成可测试条目      |
| 权限判断              | auth、permission、role        | 角色能力边界、敏感操作限制               | 标注角色来源是否可证实        |
| Feature Flag / 配置项 | config、env、constants        | 版本差异、开关策略、运行条件             | 不确定默认值时标注 `[待确认]` |
| 异常处理与重试        | catch、retry、fallback        | 可用性策略、失败体验、恢复机制           | 优先描述用户可见结果          |
| 日志与埋点            | logger、tracking              | 关键业务事件、监控观测点                 | 若没有埋点不要虚构指标口径    |

## 证据分级

- A 级：代码直接声明（函数、常量、类型、校验、路由）。
- B 级：多处实现共同指向的稳定推断（命名 + 调用链 + UI 文案）。
- C 级：仅能提出假设，需业务方确认。

输出 PRD 时，为每条关键需求标注证据分级：

- `FR-001 ... [证据: E3(A), E7(B)]`

## 反推顺序

1. 先抽取角色与场景，再抽取功能。
2. 先写主流程，再补失败分支与边界条件。
3. 先写代码可证实内容，再写假设与未决问题。
4. 先保证可验收，再补充背景叙事。

## 常见误区

- 把技术实现细节直接照抄到需求，不做业务语义转换。
- 忽略异常分支，导致验收标准只覆盖 happy path。
- 证据不足时给出确定性结论。
- 需求术语与代码命名不一致，造成评审歧义。
