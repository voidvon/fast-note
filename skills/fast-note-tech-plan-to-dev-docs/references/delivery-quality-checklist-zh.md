# 交付质量检查清单（方案转开发文档）

## 1. 文件与结构

- 是否输出到 `docs/开发计划/`。
- 是否包含 3 份目标文档：
  - `迭代计划与任务看板.md`
  - `组件与数据契约清单.md`
  - `测试与发布计划.md`
- 是否每份文档都有“文档信息”与输入来源。

## 2. 任务可执行性

- 是否每个任务都有唯一 `任务ID`。
- 是否每个任务都包含依赖、风险、验收标准。
- 是否按 P0/P1/P2 排序并映射到里程碑。

## 3. 契约完整性

- 是否覆盖 `fastnote/pages/widgets`、`fastnote/features/processes`、`fastnote/entities`、`fastnote/shared`、`backend`、`router` 等关键层。
- 是否说明 `Note` 契约变化与兼容策略。
- 是否明确同步冲突策略与失败补偿。

## 4. 测试与发布可落地性

- 是否每个 P0 任务都有测试类型映射。
- 是否每个 P0 任务都有自动化测试（unit/integration/e2e 至少一种）。
- 是否存在“任务ID -> 用例ID -> 测试文件路径”三向映射。
- 是否定义 CI 门禁级别（阻断/非阻断）并与优先级匹配。
- 是否包含监控指标、告警阈值、回滚触发条件。
- 是否给出最小可执行发布路径（灰度 -> 全量）。
- 是否给出可执行命令与通过标准（例如 `cd fastnote && npm run test:unit -- --run` 全通过）。

## 5. 跨文档一致性

- 任务编号是否在 3 份文档中可追踪。
- 验收标准是否与测试用例一致。
- 阻塞项是否在迭代计划和发布计划中同步体现。

## 6. 风险与未决问题

- 是否列出关键风险及影响范围。
- 是否显式标注 `[待确认]` 而非伪造结论。
- 是否给出建议决策顺序（先决问题优先）。

## 7. 自动化测试实施准备

- 是否给出自动化优先顺序（P0 -> P1 -> P2）。
- 是否说明 Mock 策略与依赖隔离方式。
- 是否定义测试数据准备与清理方案。
