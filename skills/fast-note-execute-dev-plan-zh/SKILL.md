# Fast-Note 开发执行器（中文）

## 快速开始

1. 读取开发计划输入：
   - `doc/开发计划/迭代计划与任务看板.md`
   - `doc/开发计划/组件与数据契约清单.md`
   - `doc/开发计划/测试与发布计划.md`
2. 确定任务：
   - 优先使用用户指定 `任务ID`。
   - 未指定时默认选择首个 `P0` 且状态非“已完成”的任务。
3. 按契约清单改代码，并补齐测试计划中该任务映射用例。
4. 执行校验命令并修复失败项。
5. 回写任务状态与测试覆盖状态。

## 工作流

### 1) 锁定任务范围

- 在看板中定位任务行：`任务ID/功能点/优先级/依赖/验收标准`。
- 依赖未完成时先标记阻塞，不盲目开发。
- 默认单任务交付；用户要求可并行多任务。

### 2) 建立实现映射

- 在契约清单中定位该功能点对应的变更层：
  - `UI` -> `fastnote/src/pages`、`fastnote/src/widgets`
  - `编排` -> `fastnote/src/processes`、`fastnote/src/features`
  - `实体` -> `fastnote/src/entities`
  - `数据` -> `fastnote/src/shared/lib/storage`、`fastnote/src/shared/api/pocketbase`
  - `路由` -> `fastnote/src/app/router`
  - `后端宿主` -> `backend`
- 禁止绕过 `entities/features/processes` 直接在 UI 中改核心数据。

### 3) 实施开发变更

- 最小闭环：`代码实现 + 自动化测试 + 验证通过`。
- 按任务验收标准逐条对齐实现。
- 涉及数据契约变更时，必须同步更新契约文档对应条目。

### 4) 补齐自动化测试

- 从 `测试与发布计划.md` 中提取该任务映射：
  - `任务ID -> 用例ID -> 测试文件路径`
- 优先级规则：
  - `P0`：必须新增或更新自动化测试，且门禁为阻断。
  - `P1`：默认自动化；若暂无法自动化需注明原因。
  - `P2`：可手工优先，但需列出自动化候选。
- 若映射文件不存在，按计划路径创建测试文件骨架并实现最小用例。

### 5) 执行验证

- 默认执行：
  - `cd fastnote && npm run lint`
  - `cd fastnote && npm run test:unit -- --run`
  - `cd fastnote && npm run build`
- 若任务映射含 E2E，再执行：
  - `cd fastnote && npm run test:e2e`
- 若 `测试与发布计划` 命令与仓库脚本不一致，优先执行仓库可用命令并回写 `[待确认]`。

### 6) 回写状态与交付说明

- 回写 `迭代计划与任务看板.md` 的任务状态（建议：未开始/进行中/已完成/阻塞）。
- 回写 `测试与发布计划.md` 三向映射覆盖状态。
- 在交付说明中包含：
  - 变更文件
  - 测试执行命令与结果
  - 未完成项与阻塞项

## 状态约定

- 任务状态：`未开始`、`进行中`、`已完成`、`阻塞`。
- 覆盖状态：`未开始`、`进行中`、`已完成`。
- 仅当代码与自动化测试通过后，任务状态才可改为 `已完成`。

## Fast-Note 约束

- 保持 offline-first：本地可编辑和可读能力不被破坏。
- `Note` 契约若改动，必须同步到 `组件与数据契约清单.md`。
- 同步逻辑改动必须说明冲突策略与失败补偿。
- 公开访问链路改动需覆盖桌面与移动两条路径。

## 输出规则

- 默认输出中文。
- 优先直接落地代码与测试，不只停留在建议。
- 若信息不足，先标注 `[待确认]` 再给最小安全实现。
